# AI 검색 데모 질의셋 (OI5)

이 문서는 라우터(`router_node`)와 세 경로(A/B/C)의 기대 동작을 명세하는 **단일출처**다.
라우터 프롬프트 예시·단위 테스트 케이스·4.8 검증 매트릭스가 이 표를 참조한다.

- **경로 A (구조형, FR13/FR14)**: 가격·차종·연식·색상·지역·주행거리·연료 등 **명시적 조건** → Text-to-SQL.
- **경로 B (질적·의미형, FR15)**: 정확한 조건 대신 **용도·느낌·추천** → 문서 RAG(임베딩 유사도).
- **경로 C (매물 무관, FR16)**: 중고차 검색과 무관한 잡담·상식 → **정중한 거절 + 검색 유도**(listings 빈 목록).

회색지대(차 얘기인데 조건이 흐릿)는 **B로 보내 추천**한다(빈손보다 추천이 낫다). 정교한 되묻기(clarify) 멀티턴은 4.6 이후.

응답 계약은 모든 경로 공통으로 `{answer, listings[]}`다. 0건이어도 빈손이 아니라 조건 완화/재질문 안내를 준다(FR17).

## ① 구조형 → 경로 A

| 질의 | 기대 분류 | 기대 결과 요지 |
|---|---|---|
| "3천만원 이하 흰색 SUV" | A | `price <= 30000000 AND color='흰색' AND body_type='SUV' AND status='on_sale'` 매물 카드. |
| "2020년 이후 제네시스" | A | `manufacturer='제네시스' AND year >= 2020` 매물. |
| "10만km 미만 디젤" | A | `mileage < 100000 AND fuel='디젤'` 매물. |
| "서울 경차 보여줘" | A | `region='서울' AND body_type='경차'` 매물. |

## ② 질적·의미형 → 경로 B

| 질의 | 기대 분류 | 기대 결과 요지 |
|---|---|---|
| "패밀리카로 무난한 거" | B | 임베딩 유사도로 가족용에 어울리는 on_sale 매물 + 근거 가이드 1건. |
| "초보운전자에게 좋은 차" | B | 의미 유사 매물 추천(정확 조건 없음). |
| "연비 좋은 전기차 추천" | B | 의미형 추천(전기차 키워드 있어도 "추천" 의도라 B). |
| "가성비 좋은 차 없을까?" | B | 느낌·용도 기반 추천. |

## ③ 회색지대 (애매 — 기대 처리 명시: B 우선)

| 질의 | 기대 분류 | 기대 결과 요지 |
|---|---|---|
| "출퇴근용 적당한 차" | B | 용도만 있고 조건 흐릿 → B로 추천(되묻기 X). |
| "괜찮은 SUV 있어?" | B | 차종 1개뿐 + "괜찮은"이라는 주관 → 조건이 흐릿하므로 B. (LLM이 A로 봐도 카드를 주면 데모상 허용.) |
| "너무 비싸지 않은 중형차" | B 또는 A | "비싸지 않은"이 모호 → B 권장. LLM이 A로 판정해 가격 임계로 풀어도 무방. |

> 회색지대는 LLM 판정이 A/B로 갈릴 수 있다. 둘 중 어디로 가도 **매물 카드 또는 추천을 돌려주면** 데모 목적상 합격(빈손·거절만 아니면 됨).

## ④ 매물 무관 → 경로 C (가드 거절)

| 질의 | 기대 분류 | 기대 결과 요지 |
|---|---|---|
| "오늘 날씨 어때?" | C | "저는 중고차 매물 검색을 도와드리는 어시스턴트예요…" + `listings: []`. |
| "파이썬 코드 짜줘" | C | 정중한 거절 + 검색 유도. 상식/코드 Q&A 미제공(FR16). |
| "안녕" | C | 인사도 거절 문구로 검색 유도. |
| "1+1은 뭐야?" | C | 상식 질의 거절. |

## ⑤ 멀티턴 맥락 → 질의 재작성 후 경로 분류 (FR18)

직전 대화를 클라이언트가 `context`(턴 배열 `{role, content}`)로 동봉하면, 서버는 **그래프 앞단의 맥락화 노드**(`contextualize_node`)가 후속 질의를 **혼자서도 뜻이 통하는 독립 질의**로 재작성한 뒤, 그 재작성 질의를 평소처럼 라우터(A/B/C)에 흘린다. 서버는 대화를 저장하지 않는다(무상태 — 새로고침/세션 종료 시 초기화).

| 직전 대화(요지) | 후속 질의 | 기대 재작성(독립 질의) | 기대 경로 |
|---|---|---|---|
| user "패밀리카로 무난한 거" / assistant "○○ 등 추천" | "그 중 더 싼 거" | "패밀리카로 무난한 차 중 더 저렴한 매물" | B |
| user "3천만원 이하 SUV" / assistant "△△ 매물 N건" | "흰색만 보여줘" | "3천만원 이하 흰색 SUV" | A |
| user "초보운전자에게 좋은 차" / assistant "□□ 추천" | "그럼 전기차로" | "초보운전자에게 좋은 전기차 추천" | B |

- **단일턴(맥락 없음)**: `context`가 없거나 비면 맥락화 노드는 **LLM을 부르지 않고 원 질의를 그대로** 흘린다(4.5까지와 동일, 비용·지연 0).
- **안전장치**: 재작성 결과가 비거나 LLM 호출이 실패하면 **원 질의로 폴백**(빈손 금지). 직전 대화에 없는 새 조건·매물은 지어내지 않는다(지시어만 치환).
- **무상태 보증**: 맥락은 매 요청의 `context`에서만 온다. 같은 서버라도 `context` 없는 요청은 직전을 기억하지 못한다(FR18).
- **`context` 입력 형태**(웹/Flutter가 보낼 단일출처): `[{"role":"user","content":"…"}, {"role":"assistant","content":"…"}]`. 최대 12턴, 각 content 최대 2000자(초과 시 422).

## 차형(body_type) 용어 매핑 (경로 A SQL 생성 참고)

자연어 차형 표현을 DB `body_type` 허용값으로 매핑한다(단일출처: `sql_rag_node` 프롬프트).

| 자연어 | DB 매핑 |
|---|---|
| 세단 | `body_type IN ('준중형차','중형차','대형차')` |
| SUV | `body_type = 'SUV'` |
| 경차 | `body_type = '경차'` |
| 데모에 없는 차형(해치백·쿠페 등) | 무리한 매핑 금지 — 가격·기타 조건만 적용 |

> `body_type` 허용값: 경차·소형차·준중형차·중형차·대형차·스포츠카·SUV·RV·경승합차·승합차·화물차·기타.
> 정규화 단위: 주행거리 "만km" → ×10000, 가격 "천만원"=10,000,000 / "만원"=10,000.

## 4.8 검증 매트릭스 (SM3·CM1·CM2 합격 판정)

이 질의셋이 SM3·CM1·CM2 합격임을 **재현 가능한 pytest**로 못박는다. 질의 문자열·기대 경로는
`api/tests/demo_queries.py`가 위 표 ①②③④를 그대로 미러링한다(단일출처, drift 금지).

쿼터 보호: 판정은 라이브 LLM에 매번 의존하지 않는다.
- **SM3·CM1** — 라우터·경로 노드를 모킹해 분기·계약을 LLM/DB 없이 결정론적으로 검증.
- **CM2** — `sql_guard`는 순수 함수라 키 없이 항상 결정론적으로 완전 검증.
- 라이브 동작은 별도 스모크에서 소량(3건 이하)만 확인하며 기본 실행에서는 스킵된다.

| 판정 | 대상 질의 | 기대 | 검증 테스트(`api/tests/`) |
|---|---|---|---|
| SM3 | ① 구조형(A) | 경로 A가 매물 카드 반환(빈손 아님) | `test_demo_acceptance.py::test_sm3_pathA_returns_listings` |
| SM3 | ② 의미형(B) | 경로 B가 추천 매물 반환 | `test_demo_acceptance.py::test_sm3_pathB_returns_listings` |
| SM3 | ③ 회색지대 | A/B 어느 쪽이든 빈손·거절 아님 | `test_demo_acceptance.py::test_sm3_gray_zone_returns_listings_either_route` |
| SM3 | 경로 A 가드 통과 | 세단 IN-매핑 SQL이 sql_guard를 실제 통과 | `test_demo_acceptance.py::test_sm3_pathA_real_guard_passes_generated_sql` |
| CM1 | ④ 무관(C) | 전부 빈 목록 + 정중한 거절 문구 | `test_demo_acceptance.py::test_cm1_unrelated_rejected_via_graph`, `test_cm1_count_all_unrelated_rejected` |
| CM2 | 위반 SQL 코퍼스 | 범위밖 SQL 0건 통과(전부 실행 전 차단) | `test_demo_acceptance.py::test_cm2_violating_sql_is_blocked`, `test_cm2_zero_violations_pass_through` |
| CM2 | 정상 SQL 대조군 | 정상 SELECT는 통과(과차단 아님) | `test_demo_acceptance.py::test_cm2_valid_sql_still_passes` |
| 라이브 | A·B·C 각 1건 | 실물 동작 눈 확인(쿼터-세이프, 기본 skip) | `test_live_smoke.py`(`RUN_LIVE_SMOKE=1`로 활성) |

> 실행: `cd api && pytest`(전체) 또는 `pytest tests/test_demo_acceptance.py`(판정만). 라이브 스모크는
> `RUN_LIVE_SMOKE=1`(+ `GEMINI_API_KEY`·`DATABASE_URL`)일 때만 돈다. 429/키부재는 실패가 아닌 skip이다.
