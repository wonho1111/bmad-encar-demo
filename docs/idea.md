# IDEA.md

한국의 **"중고차 직거래(엔카·헤이딜러)"** 와 유사한 서비스의 핵심 기능만 최소로 구현.

- 기능 범위
  - 구매자: 가입 > 로그인 > 매물 검색(키워드/필터 + AI 자연어 검색) > 상세 확인 > 판매자에게 문의 채팅
  - 판매자: 가입 > 로그인 > 차량 등록 > 매물 관리 > 문의 채팅 응대
    - 매물 관리: 본인 매물만 조회/수정/삭제, "구매 완료" 시 판매 완료 처리되어 구매자에게 미노출.
  - 관리자: 가입 > 로그인 > 회원 관리 / 전체 매물 관리 / 거래 내역 관리 / 채팅 관리
  - 채팅: 구매자가 보내면 양쪽에 동일 채팅방 생성, 폴링 방식으로 대화.

- AI 매물 검색 (자연어 기반)
  - 노출 형태: 구매자용 **AI 채팅 어시스턴트** — "3천만원 이하 흰색 SUV 추천해줘" 같은 자연어 한 문장으로 매물 검색.
  - *RAG* (Retrieval-Augmented Generation, 검색 증강 생성): LLM이 외부 데이터(DB·문서)를 먼저 검색해 그 결과를 근거로 답을 생성하는 방식.
  - *LangGraph*(LLM 흐름을 노드·엣지 그래프로 구성하는 프레임워크)로 아래 흐름 오케스트레이션:
    - 1. 라우터 노드: 질문이 *매물 조건 검색*인지 *차량 상식/안내*인지 판단.
    - 2. SQL기반 RAG 경로: 조건형 질문 → *Text-to-SQL*(자연어를 SQL 쿼리로 변환)로 매물 DB 정밀 필터링(가격·연식·차종·색상 등).
    - 3. 문서기반 RAG 경로: 의미형/상식형 질문 → *임베딩*(텍스트를 의미 기반 벡터로 변환) 유사도 검색으로 근거 문서 검색.
    - 4. 답변 생성 노드: 검색 결과를 근거로 자연어 답변 + 매물 카드 반환.
  - RAG 대상 데이터: ① 매물 데이터(차량 설명·옵션·상태), ② 차량 상식/구매 가이드/FAQ 문서.
  - 안전장치: Text-to-SQL은 SELECT 전용 + 판매완료 매물 제외 등 쿼리 범위 제한.
  - 대화 방식: **클라이언트 보관 멀티턴(multi-turn)** — AI 대화 맥락은 브라우저/앱 메모리에 두고 요청 시 직전 턴들을 함께 전송. 서버·DB에는 **AI 대화 이력을 저장하지 않음**(전용 테이블 불필요). 후속 질문("그 중 더 싼 거")은 지원하되, 새로고침/세션 종료 시 AI 대화는 초기화됨(검색 어시스턴트의 자연스러운 동작).
    - 주: 이는 **구매자↔판매자 문의 채팅**(DB 저장 필수)과 별개의 기능임.

- 기술 스택
  - 1. 사용자(구매자/판매자)용 반응형 웹 (Next.js)
  - 2. 관리자용 반응형 웹 (Next.js)
  - 3. 사용자용 모바일 앱 (Flutter)
  - 4. 통합 API (FastAPI)
  - 5. DB (Supabase, PostgreSQL)
  - 6. 배포: **Vercel로 통일** (프론트 + 백엔드)
    - 프론트엔드(Next.js) + 백엔드(FastAPI + LangGraph) 모두 Vercel
    - 근거: 본 과제는 시연용·경부하. AI 검색은 요청당 짧은(수초) Gemini 호출 1~2회로 끝나 Vercel Hobby 함수 시간 제한(최대 300초) 내 충분. Python(ASGI) 런타임으로 FastAPI 구동, 의존성도 번들 500MB 한도 내.
    - 유의: ① 무상태(stateless)이므로 채팅 이력은 Supabase에 저장(폴링과 잘 맞음) ② Supabase 연결 풀링 설정 ③ 무거운 라이브러리 미적재.
    - 차선(부하 증가 시): 백엔드만 Google Cloud Run / Railway로 분리.
    - 모바일 테스트: Android 에뮬레이터(+ mobile-mcp/Maestro) — Flutter는 디바이스/에뮬레이터로 확인. Expo Go는 React Native 전용이라 Flutter엔 미사용.
    - 비고: 기술 연구 + 경부하 시연 스코프 재검토 반영 (planning-artifacts/research/technical-ai-search-architecture-research-2026-06-16.md)
  - 7. AI 오케스트레이션: LangGraph (Python) — FastAPI 백엔드 내 서비스로 탑재
  - 8. LLM: Google Gemini API 무료 등급 (Gemini Flash 계열) — 라우팅/Text-to-SQL/답변 생성
  - 9. 임베딩: Gemini 임베딩 — 생성 LLM과 동일 SDK·API 키로 통일 (관리 단순화)
  - 10. 벡터 검색: Supabase pgvector (PostgreSQL 임베딩 유사도 검색) — 기존 DB 재사용
  - 비고: 과제용 저비용 구성. LangGraph는 모델 비종속(provider-agnostic)이라 추후 다른 LLM으로 교체 가능.
