# 03. LangGraph — AI 검색 두뇌

> 이 프로젝트의 하이라이트입니다. "3천만원 이하 흰색 SUV 추천해줘" 같은 자연어가 **어떻게 SQL 검색 또는 문서 검색으로 바뀌어 답이 되는지**를 다룹니다.
> 대상 코드: `api/app/graph/` 폴더 8개 파일.

---

## 3-1. LangGraph가 뭔가요

- *LangGraph*: LLM(거대 언어 모델) 작업 흐름을 **노드(node)와 엣지(edge)로 이뤄진 그래프**로 짜는 라이브러리.
- *노드*: 하나의 처리 단계(함수). 예) "질문 분류하기", "SQL 만들기".
- *엣지*: 노드에서 노드로 가는 화살표(흐름). 조건에 따라 갈라질 수 있음.
- *상태(State)*: 노드들 사이를 흐르며 정보를 담아 나르는 "공유 메모". 각 노드가 읽고 채움.

왜 그래프로 짤까요? AI 작업은 "분류 → 분기 → 처리 → 답변 조립"처럼 **여러 단계가 조건에 따라 갈라지는** 경우가 많습니다. 이걸 그림(그래프)으로 명확히 표현하면 흐름을 보기 쉽고 고치기 쉽습니다.

---

## 3-2. 이 AI의 전체 그림

```
            사용자 질문 + 이전 대화(context)
                       │
                       ▼
     [맥락화] contextualize  ("그 중 더 싼 거" → "2천만원 이하 흰색 중형차")
                       │  (그래프 진입 전, graph.py에서 먼저 실행)
                       ▼
              ┌──── router (라우터: 질문 분류) ────┐
              │              │                    │
         A(조건검색)     B(의미·추천)         C(매물무관)
              │              │                    │
              ▼              ▼                    ▼
    sql_rag (Text-to-SQL)  doc_rag(임베딩검색)   guard(정중한 거절)
              │              │                    │
              └──────────────┼────────────────────┘
                             ▼
                    answer (답변 정규화) → 끝
                             │
                             ▼
              { answer: "...", listings: [매물카드...] }
```

상태 객체 `SearchState`에는 4개 칸이 있습니다: `query`(질문), `route`(A/B/C 판정), `answer`(답변 문장), `listings`(매물 카드 목록).

---

## 3-3. 파일별 설명

### `graph.py` — 그래프 조립 + 실행 (지휘자)

- `_build_graph()`가 노드 5개(router, sql, doc, guard, answer)를 만들고 엣지로 잇습니다.
- **진입점은 router.** router 결과(A/B/C)에 따라 `add_conditional_edges`로 갈라지고, 세 경로 모두 answer로 모인 뒤 끝(END).
- 그래프는 서버 시작 시 **한 번만 컴파일**해 재사용(요청마다 다시 만들면 느림).
- 외부에서 부르는 진입 함수가 **`run_search(query, context)`**:
  1. 먼저 `contextualize_query(query, context)`로 멀티턴 맥락을 합쳐 "독립적인 질문"을 만듦.
  2. 그 질문으로 `COMPILED_GRAPH.invoke(...)` 실행 → `{answer, listings}` 반환.

### `contextualize_node.py` — 멀티턴 맥락 합치기

- "그 중 더 싼 거", "아까 그거" 같은 **이전 대화에 의존하는 질문**을, 그것만 읽어도 뜻이 통하는 한 문장으로 다시 씁니다.
- 똑똑한 절약: 이전 대화(context)가 없으면 LLM을 **안 부르고** 그냥 통과(비용 0).
- **주제 전환 감지**(`_is_topic_shift`): "중형차 보다가 → 갑자기 SUV"처럼 새 검색으로 바뀌면, 이전 조건을 **버립니다**(맥락 오염 방지). 반대로 "그 중에서"처럼 좁히는 표현이면 맥락을 유지.
- 서버는 무상태: 맥락을 저장하지 않고 매번 인자로만 받습니다.

### `router_node.py` — 질문 분류 (A/B/C)

질문을 셋 중 하나로 분류:
- **A (구조형)**: 가격·차종·연식·색상 등 **명확한 조건**이 있음 → SQL 검색.
- **B (의미형)**: "패밀리카로 무난한 거"처럼 **느낌·용도만** 있음 → 문서/의미 검색.
- **C (매물 무관)**: 세금·보험·잡담 등 → 거절.

작동:
- Gemini에게 분류 프롬프트를 주고 `with_structured_output(RouterDecision)`으로 **A/B/C만 나오게 강제**(구조화 출력).
  - *구조화 출력*: LLM이 자유 문장 대신 정해진 형식(여기선 route 값)으로만 답하게 함.
- **이중 안전**: LLM이 형식을 벗어나면, 코드가 `_fallback_route`로 보정(매물 신호 단어가 있으면 B, 없으면 C). LLM은 틀릴 수 있으니 코드가 한 번 더 점검.
- *temperature=0*: LLM의 무작위성을 0으로 → 같은 질문엔 같은 분류(재현성).

### `guard_node.py` — 경로 C (정중한 거절)

- 고정된 거절 문구를 반환합니다. **LLM을 안 부름**(비용 0, 빠름).
- 막다른 길이 아니라 갈림길로: "세금 계산은 제 전문이 아니에요. 대신 예산·차종을 알려주시면 매물을 찾아드릴게요"처럼 유도.

### `sql_rag_node.py` — 경로 A (Text-to-SQL, 핵심)

자연어를 SQL로 바꿔 매물 DB를 정밀 검색합니다.

```
질문 → [Gemini] SELECT문 생성 → [sql_guard] 검사 → [읽기전용] 실행 → 매물카드
                                      │
                            막히면 에러를 LLM에 알려 1회 재생성
```

- 시스템 프롬프트에 **스키마·규칙**을 자세히 박음: "`status='on_sale'` 항상 포함, OR 금지, AND만, LIMIT 붙이기, 단위 변환('3천만원'→30000000), 차형 매핑('세단'→준중형/중형/대형)" 등.
- 생성된 SQL은 반드시 `validate_select_sql()`(02번의 sql_guard) 검사를 통과해야 실행.
- **1회 재시도**: 안전장치에 걸리면 에러 메시지를 LLM에 다시 줘서 고칠 기회를 줌. 2번째도 실패하면 에러를 위로 던짐(→ 400).
- DB 오류는 재시도 안 함(재생성으로 못 고치는 문제라).

### `doc_rag_node.py` — 경로 B (임베딩 유사도 검색)

느낌·용도 질문을 **의미 벡터**로 바꿔 비슷한 매물·문서를 찾습니다.

```
질문 → [embed_query] 768차원 벡터 → pgvector 코사인 검색
        → 매물 5건 (ORDER BY embedding <=> 질문벡터)
        → 가이드 문서 1건(근거) → "...에 어울리는 매물 N건 (참고: 가이드제목)"
```

- 여기는 LLM이 SQL을 만들지 **않습니다.** 코드가 **고정 SQL + 파라미터 바인딩**을 쓰므로 인젝션 위험이 없습니다.
  - *`<=>` 연산자*: pgvector의 코사인 거리. 작을수록 의미가 가까움 → `ORDER BY ... LIMIT 5`로 가장 가까운 5건.
- 결과가 0건이면 안내 문구를 반환.

### `answer_node.py` — 답변 정규화 (마무리)

- 경로 A/B/C가 만든 결과를 받아 **형식만 다듬습니다.** 새 답변을 LLM으로 또 만들지 않음(과잉 구현 금지).
- answer가 비었으면 보정: 매물이 있으면 "N건 찾았어요", 0건이면 "조건을 넓혀보세요" 같은 fallback.
- 항상 `{answer: 문자열, listings: 리스트}` 모양을 보장.

### `listing_cards.py` — 매물 카드 변환 (공유 헬퍼)

- 경로 A·B가 **둘 다** 쓰는 헬퍼. DB 결과 행(튜플)을 `ListingCard` 객체로 바꿉니다.
- `SELECT_COLUMNS = "id, manufacturer, model, year, price, mileage, region"` 를 한곳에 두어, 두 노드가 같은 컬럼·순서를 쓰도록 통일(불일치 방지).
  - *단일 출처(Single Source of Truth)*: 같은 정보를 한 곳에만 정의 → 고칠 때 한 곳만.

## 3-4. 경로별 전체 흐름 예시

**경로 A — "3천만원 이하 흰색 SUV"**
```
contextualize(맥락 없음) → query 그대로
router → "A" (가격·색상·차종 조건 있음)
sql_rag → Gemini가 SQL 생성:
   SELECT id,manufacturer,model,year,price,mileage,region FROM listings
   WHERE status='on_sale' AND price<=30000000 AND color='흰색' AND body_type='SUV'
   LIMIT 5
   → sql_guard 통과 → 읽기전용 실행 → 3건
answer → "조건에 맞는 매물 3건을 찾았어요." + 카드 3장
```

**경로 B — "패밀리카로 무난한 거"**
```
router → "B" (명확한 조건 없이 용도만)
doc_rag → 질문을 벡터로 임베딩 → pgvector 코사인 검색
   → 매물 5건 + 가이드문서 1건("패밀리카 선택 가이드")
answer → "'패밀리카로 무난한 거'에 어울리는 매물 5건을 찾았어요. (참고: 패밀리카 선택 가이드)" + 카드 5장
```

**경로 C — "파이썬 코드 짜줘"**
```
router → "C" (매물 무관)
guard → 고정 거절 문구 (LLM 호출 없음)
answer → 그대로
```

**멀티턴 — Q1"2천만원 이하 중형세단" → Q2"그 중 흰색은?"**
```
contextualize(Q2, context=[Q1...]) → "그 중" 감지(좁히기) → LLM 재작성
   → "2천만원 이하 중형세단 흰색"
이후 경로 A로 진행
```

## 3-5. 핵심 개념 정리 (이 파트에서 꼭 배워야 할 것)

1. **상태 그래프(StateGraph)** — 노드·엣지·상태로 AI 흐름을 조립. 진입점/조건부 분기/종료점.
2. **조건부 분기(conditional edges)** — router 결과로 경로가 갈라지는 구조.
3. **RAG(검색 증강 생성)** — 답하기 전에 DB/문서를 먼저 검색해 근거로 삼기(환각 감소).
4. **Text-to-SQL** — 자연어→SQL 변환. 그리고 **절대 그대로 믿지 않고 검증**하는 습관.
5. **임베딩 유사도 검색** — 텍스트를 벡터로 바꿔 의미가 가까운 것 찾기(pgvector `<=>`).
6. **구조화 출력(structured output)** — LLM이 정해진 형식(A/B/C)으로만 답하게 강제 + 코드 보정.
7. **temperature=0** — 재현성(같은 입력→같은 출력)이 필요한 분류/생성에 사용.
8. **무상태 멀티턴** — 대화 맥락을 서버에 저장 않고 매 요청에 함께 보냄. 주제 전환 감지로 오염 방지.
9. **비용 절약 설계** — 단일턴·거절 경로는 LLM을 아예 안 부름.
10. **단일 출처 + 다층 방어** — 공유 헬퍼로 일관성, 검증+권한으로 안전.
