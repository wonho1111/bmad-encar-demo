# 01. 데이터베이스(DB) — Supabase / PostgreSQL

> 모든 것의 토대입니다. "어떤 데이터가 어떤 모양으로 저장되고, 누가 그 데이터를 볼 수 있는지"가 여기서 정해집니다.
> 대상 코드: `supabase/migrations/` 의 SQL 파일 9개 (`0001`~`0009`).

---

## 1-1. 먼저 알아둘 것: 마이그레이션이 뭔가요

- *마이그레이션(migration)*: DB 구조 변경을 **번호 붙인 SQL 파일로 차곡차곡 기록**하는 방식.
- 파일을 순서대로(`0001` → `0009`) 실행하면 빈 DB가 완성된 DB가 됩니다.
- 장점: "언제 어떤 테이블/규칙이 왜 생겼는지"가 git 이력처럼 남습니다. 되돌리기·재현이 쉽습니다.

이 프로젝트는 작은 데모라 일부러 **단순하게**, 핵심 테이블 위주로 시작해 필요할 때 컬럼을 덧붙였습니다(아래 0007~0009가 그 예).

---

## 1-2. 테이블 한눈에 (ER 관계)

> *ER(Entity-Relationship): 테이블(개체)들이 서로 어떻게 연결되는지의 그림.*
> *FK(Foreign Key, 외래키): 한 테이블의 값이 다른 테이블의 행을 가리키는 연결고리.*

```
auth.users  (Supabase가 자동 관리하는 "로그인 계정" 테이블)
   │  id
   ▼ (1:1)
profiles    (우리 앱의 "회원": role, status, name)
   │  id
   ├─────────────► listings   (매물: 차 정보 15필드 + 임베딩 + seller_name)
   │   seller_id        │ id
   │                    ▼
   └──► chat_rooms ◄────┘  (채팅방: listing_id, buyer_id, seller_id, 양쪽 이름)
            │ id
            ▼
        chat_messages   (메시지: room_id, sender_id, body)

guide_documents   (AI용 가이드 문서 + 임베딩 벡터. FK 없는 독립 테이블)
```

핵심 관계 5개:
- `profiles.id` = `auth.users.id` (로그인 계정 1개 = 회원 1명)
- `listings.seller_id` → `profiles.id` (매물의 주인 = 판매자)
- `chat_rooms` 는 `listing_id`·`buyer_id`·`seller_id` 3개를 가리킴
- `chat_messages.room_id` → `chat_rooms.id` (메시지는 어느 방 소속인지)
- `guide_documents` 는 아무와도 연결 안 됨 (AI 검색 전용 지식 창고)

**ON DELETE CASCADE**: 부모가 지워지면 자식도 함께 지워지는 설정입니다.
예) 매물을 삭제하면 그 매물의 채팅방·메시지가 자동으로 같이 삭제됩니다(고아 데이터 방지).

## 1-3. 파일별 설명

### `0001_profiles.sql` — 회원 테이블 + 가입 자동화

**profiles 테이블**

| 컬럼 | 타입 | 제약 | 의미 |
|------|------|------|------|
| id | uuid | PK, FK→auth.users (CASCADE) | 로그인 계정과 동일 ID |
| role | text | NOT NULL, CHECK(buyer/seller/admin) | 역할 |
| status | text | NOT NULL, default 'active', CHECK(active/suspended) | 활성/정지 |
| name | text | (0009에서 추가) | 표시 이름(이메일 @앞부분) |
| created_at | timestamptz | NOT NULL, default now() | 가입 시각 |

- *PK(Primary Key, 기본키)*: 행을 유일하게 구분하는 값.
- *CHECK 제약*: 컬럼에 들어올 수 있는 값을 DB가 직접 제한(예: role은 셋 중 하나만).

핵심 장치 2개:
1. **`handle_new_user()` 트리거** — 누가 회원가입(`auth.users`에 행 추가)하면, **자동으로 `profiles` 행을 만들어 줍니다.** 이때 role은 가입 시 보낸 메타데이터에서 읽되 `admin`은 차단(buyer/seller만 허용 → 아무나 관리자가 못 됨).
   - *트리거(trigger)*: 특정 DB 이벤트(여기선 "새 회원 생성")가 일어나면 자동 실행되는 함수.
2. **`is_admin()` 함수** — "이 사람 관리자야?"를 판별하는 헬퍼. RLS 안에서 `profiles`를 직접 조회하면 무한 반복(재귀) 에러가 나서, 이를 우회하려고 별도 함수로 뺐습니다.
   - *SECURITY DEFINER*: 함수를 "호출한 사람" 권한이 아니라 "함수를 만든 주인" 권한으로 실행 → RLS를 우회해 조회 가능.

**RLS 정책**: 본인 프로필만 조회(`profiles_select_self`), 관리자는 전체 조회(`profiles_select_admin`).

### `0002_listings.sql` — 매물 테이블 (이 앱의 핵심 데이터)

매물 1건은 차량 정보 **15개 필드**로 이뤄집니다.

| 컬럼 | 타입 | 제약/단위 |
|------|------|-----------|
| id | uuid | PK |
| seller_id | uuid | FK→profiles (CASCADE) |
| status | text | 'on_sale'/'sold' (판매중/판매완료) |
| manufacturer | text | CHECK(고정 15개 제조사) |
| model | text | 자유 입력 |
| body_type | text | CHECK(경차~기타 12종) |
| year | int | 1990 ~ 올해+1 |
| price | bigint | ≥0, 단위 **원(KRW)** |
| mileage | int | ≥0, 단위 **km** |
| color | text | CHECK(9색) |
| fuel | text | 가솔린/디젤/하이브리드/전기/LPG |
| transmission | text | 자동/수동 |
| displacement | int | ≥0, 단위 **cc** |
| seats | int | 2~11 |
| region | text | CHECK(서울~제주 17개) |
| accident_free | boolean | 무사고 여부 |
| options | text[] | 옵션 목록(임베딩 대상) |
| description | text | 설명(임베딩 대상) |
| embedding | vector(768) | AI 의미검색용 벡터 (처음엔 NULL, Epic 4에서 채움) |
| created_at / updated_at | timestamptz | 등록/수정 시각 |

핵심 장치:
- **`set_updated_at()` 트리거** — 매물을 수정(UPDATE)할 때마다 `updated_at`을 현재 시각으로 자동 갱신. 동시에 `created_at`은 원래 값으로 고정(등록일 위조 방지).
- **`vector(768)`** — 차량 설명/옵션을 768개 숫자로 요약한 "의미 벡터". AI 검색 경로 B에서 유사도 검색에 씁니다.

**RLS 정책 (가장 중요)**:
| 정책 | 동작 | 누가 |
|------|------|------|
| listings_select_on_sale | SELECT | **누구나** 판매중 매물 조회 |
| listings_select_own | SELECT | 판매자는 본인 매물(sold 포함) 조회 |
| listings_select_admin | SELECT | 관리자는 전부 조회 |
| listings_insert_own | INSERT | **본인(seller_id=나)** 만 등록 |
| listings_update_own | UPDATE | 본인만 수정 |
| listings_delete_own | DELETE | 본인만 삭제 |

> 이 6개 정책이 "남의 매물을 함부로 못 고친다"를 **DB 차원에서** 보장합니다. 프론트 코드 실수가 있어도 DB가 막아줍니다.

### `0003_chat.sql` — 채팅방 + 메시지

- **chat_rooms**: 한 매물에 대해 (구매자, 판매자) 1쌍당 방 1개. `UNIQUE(listing_id, buyer_id, seller_id)`로 중복 방 생성 방지, `CHECK(buyer_id ≠ seller_id)`로 자기 자신과의 방 금지.
- **chat_messages**: 방에 속한 메시지. `body`는 공백만 금지(`length(btrim(body)) > 0`).
- **인덱스**: `chat_messages(room_id, created_at)` — 폴링(3초마다 새 메시지 조회)을 빠르게.
  - *인덱스(index)*: 특정 컬럼으로 빨리 찾도록 만든 "책 색인". 조회 속도↑.

**RLS**: 방의 **당사자(구매자 or 판매자)** 만 그 방·메시지를 조회/작성 가능. 메시지엔 UPDATE/DELETE 정책이 없어 **한 번 보낸 메시지는 수정/삭제 불가**(관리자 삭제만 0005에서 별도 허용).

---

### `0003c_chat_room_integrity.sql` — 채팅방 위조 차단 (보안 패치)

문제: 0003의 INSERT 정책은 "만드는 사람이 buyer나 seller 중 하나면 통과"라, 구매자가 **임의의 seller_id를 적어** 엉뚱한 방을 만들 여지가 있었습니다(코드리뷰에서 발견된 High 등급 이슈).

해결: **`enforce_chat_room_seller()` 트리거** — 방을 만들 때 클라이언트가 보낸 `seller_id`를 무시하고, **그 매물의 진짜 주인(listings.seller_id)으로 강제로 덮어씁니다.** 매물이 없으면 에러. 구매자가 자기 매물에 문의하면 buyer=seller가 되어 CHECK 제약에 걸려 자동 거부됩니다.

> 교훈: **클라이언트가 보낸 값을 믿지 말고, 서버(여기선 DB 트리거)에서 진짜 값으로 다시 확정**한다.

---

### `0004_guide_documents.sql` — AI용 가이드 문서 + 벡터 인덱스

- **guide_documents**: 차량 구매 가이드/상식 문서. `title`, `content`, `embedding vector(768)`.
- **HNSW 인덱스** — 임베딩 벡터를 빠르게(거의 O(log n)) 유사도 검색하게 하는 특수 인덱스. `listings`와 `guide_documents` 둘 다 추가.
  - *HNSW*: 벡터 검색용 그래프형 인덱스. "의미가 가까운 것 빨리 찾기"에 최적.
  - *코사인 유사도*: 두 벡터가 가리키는 방향이 얼마나 비슷한지로 의미 유사도를 잼.
- **ai_readonly 롤에 SELECT 권한 + RLS 정책(using true)** — AI 검색이 이 문서를 읽을 수 있게.

> 함정: 단순히 `GRANT SELECT`만 주면 RLS가 켜진 테이블은 행이 0건만 보입니다. **RLS 정책(`using true`)까지 있어야** 실제 행이 보입니다.

---

### `0005_admin_policies.sql` — 관리자 권한(여러 테이블 가로지르기)

관리자는 여러 테이블에 걸쳐 권한이 필요하므로, 관리자용 RLS를 여기 한곳에 모았습니다. `is_admin()`이 true일 때만 열리는 정책 7개:
- 회원 수정/삭제(정지·강퇴), 매물 삭제, 모든 채팅방·메시지 조회/삭제.

기존 정책과 **OR로 결합**되므로 일반 사용자에겐 추가로 열리는 게 전혀 없습니다(회귀 0). 즉 "관리자에게만 더 열어주는" 안전한 방식.

---

### `0006_readonly_role.sql` — AI 전용 "읽기 전용" 계정

AI 검색 백엔드는 절대 데이터를 바꾸면 안 됩니다(자연어→SQL이라 위험). 그래서 **`ai_readonly`라는 읽기 전용 롤**을 만들어, AI 백엔드는 이 롤로만 DB에 접속합니다.
- *롤(role)*: DB에서 "이 계정이 뭘 할 수 있는지" 묶어둔 권한 집합.
- SELECT(읽기) 권한만 주고 INSERT/UPDATE/DELETE는 안 줌 → DB가 쓰기를 원천 차단.

> 판매완료(sold) 매물 숨김은 RLS가 아니라 **AI 검색 쿼리가 `WHERE status='on_sale'`로 직접 책임**집니다(이 롤은 모든 행을 보도록 열려 있기 때문). → 백엔드 `sql_guard`가 이를 강제 (02번 문서 참고).

---

### `0007`·`0008`·`0009` — 이름 컬럼 추가 (비정규화)

UI에서 사용자를 UUID 대신 "이름"으로 보여주려고, 이메일의 `@` 앞부분을 이름으로 잘라 각 테이블에 **복사 저장**했습니다.
- `0007`: `listings.seller_name` (매물 목록에 판매자 이름)
- `0008`: `chat_rooms.buyer_name`, `seller_name` (채팅에서 상대 이름)
- `0009`: `profiles.name` (관리자 회원관리에서 이름)

전부 BEFORE INSERT 트리거(SECURITY DEFINER)로 자동 채웁니다. 기존 데이터는 백필(backfill) UPDATE로 채웠습니다.
- *비정규화(denormalization)*: 원래 한 곳에만 둘 데이터를 성능·편의를 위해 여러 곳에 복사. 대신 트리거로 항상 동기화.
- *백필(backfill)*: 새 컬럼이 생겼을 때 기존 행들에 값을 소급해서 채우는 작업.

> 왜 복사했나? 구매자는 RLS상 남의 `auth.users`(이메일)를 못 읽기 때문에, 보여줄 이름을 매물/방에 미리 박아둔 것입니다. (데모라 이메일 앞부분 노출을 감수 — 실서비스면 개인정보 고려 필요)

## 1-4. 핵심 개념 정리 (이 파트에서 꼭 배워야 할 것)

1. **RLS (Row Level Security)** — 행 단위 접근 제어. 이 앱 보안의 심장. "누가 어떤 행을 읽고/쓰나"를 DB에 박아둠.
2. **FK + ON DELETE CASCADE** — 테이블 간 연결과 연쇄 삭제(고아 데이터 방지).
3. **트리거 (BEFORE/AFTER)** — 특정 이벤트에 자동 실행. 가입 시 프로필 생성, 수정 시각 갱신, seller_id 강제 등.
4. **SECURITY DEFINER 함수** — RLS를 우회해야 하는 특수 작업(이메일 읽기, 관리자 판별)에 사용.
5. **CHECK 제약 / 단위 통일** — 잘못된 값(이상한 role·연식·가격)을 DB가 직접 거부. 단위(원/km/cc)는 코드·문서·DB가 같은 약속.
6. **pgvector + HNSW** — 임베딩 벡터 유사도 검색(AI 의미 검색의 토대).
7. **비정규화와 트리거 동기화** — 성능/편의를 위한 데이터 복사와 그 관리법.
8. **마이그레이션 의존 순서** — FK 대상 테이블이 먼저 와야 함(profiles→listings→chat).
9. **읽기 전용 롤(ai_readonly)** — AI에게 최소 권한만 주는 보안 설계.
10. **"클라이언트 값을 믿지 않는다"** — 0003c처럼 서버/DB에서 진짜 값으로 다시 확정하는 사고방식.
