## 0-1. 이 프로젝트가 뭔가요?

**엔카·헤이딜러 같은 "중고차 직거래 서비스"의 핵심 기능만 최소로 구현한 데모**입니다.
(상세 기획은 `docs/idea.md` 참고)

세 종류의 사용자가 있습니다.

| 역할 | 할 수 있는 일 |
|------|--------------|
| **구매자(buyer)** | 가입·로그인 → 매물 검색(필터 + AI 자연어 검색) → 상세 확인 → 판매자에게 문의 채팅 |
| **판매자(seller)** | 가입·로그인 → 차량 등록 → 본인 매물 관리(수정/삭제/구매완료 처리) → 문의 응대 |
| **관리자(admin)** | 회원 관리 / 전체 매물 관리 / 거래 내역 / 채팅 관리 |

여기에 더해, 이 프로젝트의 **간판 기능**은 **AI 매물 검색 어시스턴트**입니다.
"3천만원 이하 흰색 SUV 추천해줘" 같은 자연어 한 문장으로 매물을 찾아줍니다.

## 0-2. 큰 그림: 어떤 조각들로 이루어져 있나

이 프로젝트는 한 덩어리가 아니라 **여러 개의 독립된 프로그램이 하나의 데이터베이스를 공유**하는 구조입니다.
이렇게 여러 프로젝트를 한 저장소(repository)에 모아둔 것을 **모노레포(monorepo)** 라고 합니다.

```
                 ┌─────────────────────────────────────────────┐
                 │                                             │
   [사용자]      │   web/  (Next.js 웹)        app/ (Flutter 앱) │   ← 화면(프론트엔드)
   브라우저/폰  ─┤      │                          │            │
                 │      │ ① 로그인·매물·채팅       │ ① 동일      │
                 │      │   (Supabase 직접 호출)   │            │
                 │      ▼                          ▼            │
                 │   ┌──────────────────────────────────┐      │
                 │   │  supabase/  (PostgreSQL + 인증)   │ ◀────┤   ← 데이터베이스 + 인증
                 │   │  - 회원, 매물, 채팅 데이터        │      │
                 │   │  - pgvector(임베딩 벡터)          │      │
                 │   └──────────────────────────────────┘      │
                 │      ▲                          ▲            │
                 │      │ ② AI 검색만              │ ② 동일      │
                 │      │   (FastAPI 호출)         │            │
                 │   ┌──────────────────────────────────┐      │
                 │   │  api/  (FastAPI + LangGraph)      │      │   ← AI 백엔드
                 │   │  - 자연어 → SQL / 문서검색         │      │
                 │   │  - Gemini(LLM·임베딩) 호출         │      │
                 │   └──────────────────────────────────┘      │
                 └─────────────────────────────────────────────┘
```

**핵심 포인트 2가지** (처음엔 헷갈리는 부분):

1. **일반 기능(로그인·매물·채팅)은 프론트(web/app)가 Supabase에 직접 말을 겁니다.** 별도 백엔드 서버가 없습니다.
   - Supabase가 DB + 인증 + 권한관리(RLS)를 다 해주기 때문입니다. (아래 0-4 설명)
2. **AI 검색만 별도의 FastAPI 백엔드(`api/`)를 거칩니다.** 자연어를 SQL로 바꾸거나 문서를 검색하는 일은 LLM(거대 언어 모델)이 필요해서, 그 일을 전담하는 Python 서버를 따로 둔 것입니다.

> 즉 **백엔드 서버(api/)는 "AI 검색 전용"** 입니다. 회원가입·매물등록 같은 일반 CRUD는 백엔드를 안 거치고 프론트→Supabase로 바로 갑니다.
> *CRUD(Create/Read/Update/Delete): 데이터를 만들고·읽고·고치고·지우는 기본 4동작.*

## 0-3. 기술 스택 한눈에

| 파트 | 기술 | 폴더 | 학습 문서 |
|------|------|------|-----------|
| 데이터베이스 | Supabase (PostgreSQL + pgvector) | `supabase/migrations/` | [01-db.md](01-db.md) |
| AI 백엔드 | FastAPI (Python) | `api/app/` (graph 제외) | [02-backend-api.md](02-backend-api.md) |
| AI 챗봇 흐름 | LangGraph + Gemini | `api/app/graph/` | [03-langgraph-ai.md](03-langgraph-ai.md) |
| 웹 프론트 | Next.js (React, TypeScript) | `web/src/` | [04-web-frontend.md](04-web-frontend.md) |
| 모바일 앱 | Flutter (Dart) | `app/lib/` | [05-flutter-mobile.md](05-flutter-mobile.md) |
| **전체 파일 색인** | 모든 파일 한 줄 설명 | (전 폴더) | [06-file-reference.md](06-file-reference.md) |

용어 풀이:
- *Supabase*: PostgreSQL 데이터베이스에 인증·권한·API를 얹어주는 백엔드 서비스(BaaS, Backend-as-a-Service).
- *pgvector*: PostgreSQL에서 "임베딩 벡터" 유사도 검색을 가능하게 하는 확장 기능.
- *FastAPI*: Python으로 API 서버를 만드는 웹 프레임워크.
- *LangGraph*: LLM 작업 흐름을 노드·엣지 "그래프"로 짜는 라이브러리.
- *Gemini*: 구글의 LLM(거대 언어 모델). 답변 생성과 임베딩 둘 다 담당.
- *Next.js*: React 기반 웹 프레임워크. 서버에서 미리 그려주는 화면(SSR)에 강함.
- *Flutter*: 하나의 코드로 안드로이드/iOS 앱을 만드는 구글의 프레임워크. 언어는 Dart.

## 0-4. 처음 보면 가장 낯선 개념: RLS (꼭 이해하고 넘어가세요)

보통 웹앱은 "백엔드 서버가 문지기 역할을 하며, 누가 무슨 데이터를 봐도 되는지 검사"합니다.
그런데 이 프로젝트는 프론트가 DB(Supabase)에 직접 접근하죠. 그러면 **아무나 남의 데이터를 빼가지 않을까?**

그걸 막는 게 **RLS(Row Level Security, 행 수준 보안)** 입니다.
- DB 테이블의 **각 행(row)마다 "누가 이 행을 읽고/쓸 수 있는지" 규칙을 DB 자체에 박아두는 것**입니다.
- 예: "매물(listings) 행은 누구나 읽을 수 있지만, **수정/삭제는 그 매물의 주인(seller)만** 가능."
- 이 규칙은 DB 안에 있어서, 프론트가 직접 호출하든 백엔드가 호출하든 **무조건 적용**됩니다. 그래서 문지기 서버가 없어도 안전합니다.

> 이 프로젝트에서 "보안의 핵심"은 거의 다 `supabase/migrations/`의 RLS 정책에 있습니다. → 자세한 건 [01-db.md](01-db.md)

## 0-5. AI 검색은 어떻게 동작하나 (맛보기)

사용자가 "3천만원 이하 흰색 SUV 추천해줘"라고 입력하면:

1. **라우터(router)** 가 질문 종류를 판단합니다 → "이건 매물 조건 검색이다(경로 A)" vs "이건 차량 상식 질문이다(경로 B)".
2. **경로 A (Text-to-SQL)**: 자연어를 안전한 `SELECT` SQL로 바꿔 매물 DB를 정밀 필터링 → 매물 카드로 답변.
3. **경로 B (문서 RAG)**: 질문을 임베딩 벡터로 바꿔, 미리 저장해둔 가이드 문서들 중 의미가 가까운 걸 찾아 그걸 근거로 답변.
4. **답변 노드** 가 결과를 자연어 문장 + 매물 카드로 조립해 돌려줍니다.

이 "라우터 → 경로 분기 → 답변 조립"의 흐름을 **LangGraph 그래프**로 짠 것입니다. → 자세한 건 [03-langgraph-ai.md](03-langgraph-ai.md)

용어:
- *RAG(Retrieval-Augmented Generation)*: LLM이 답하기 전에 외부 데이터(DB·문서)를 먼저 "검색"해서 그 결과를 근거로 답을 만드는 방식. 환각(없는 말을 지어냄)을 줄여줍니다.
- *Text-to-SQL*: 자연어 질문을 SQL 쿼리로 변환하는 것.
- *임베딩(embedding)*: 텍스트의 "의미"를 숫자 벡터로 바꾼 것. 의미가 비슷하면 벡터도 가깝습니다.
- *멀티턴(multi-turn)*: "그 중에 더 싼 거"처럼 이전 대화를 이어받는 연속 대화. 단, 이 AI 대화는 **DB에 저장하지 않고** 브라우저/앱 메모리에만 둡니다(문의 채팅과 다른 점).

## 0-6. 배포는 어디에? (실제 운영 환경)

기획서(idea.md)엔 "Vercel로 통일"이라 적혀 있지만, **실제로는 도중에 바뀌었습니다**(중요):

| 파트 | 실제 배포처 | 이유 |
|------|-------------|------|
| 웹(web) | **Vercel** | Next.js에 최적화. `main` 브랜치 = 운영, 그 외 = 미리보기(preview) |
| AI 백엔드(api) | **Google Cloud Run** (서울 리전) | FastAPI+AI 라이브러리 묶음이 480MB로 Vercel 한도(250MB) 초과 → 컨테이너 방식인 Cloud Run으로 이전 |
| DB | **Supabase** (클라우드) | — |
| 모바일(app) | 스토어 미배포(데모) | 실제 폰에 직접 설치해 테스트 |

- 운영 웹 주소: https://bmad-encar-demo.vercel.app
- *Cloud Run*: 컨테이너(프로그램+환경을 통째로 담은 상자)를 올리면 자동으로 굴려주는 구글 클라우드 서비스.
- *컨테이너(container)*: 코드와 실행 환경을 한 상자에 담아 "어디서든 똑같이 돈다"를 보장하는 기술(예: Docker).

배포 흐름: `develop` 브랜치에 push → 웹은 Vercel preview, API는 Cloud Run 개발 서비스가 **자동 배포**. 운영(`main`) 반영은 사용자 승인 후 병합으로만.

## 0-7. 폴더 구조 지도

```
bmad-encar-demo/
├─ supabase/migrations/   ← ① DB: 테이블·RLS·트리거 정의 (SQL 9개 파일)
├─ api/                    ← ② AI 백엔드 (FastAPI + LangGraph)
│   ├─ app/
│   │   ├─ main.py         · 서버 시작점
│   │   ├─ config.py       · 환경설정 로드
│   │   ├─ auth.py         · JWT 인증
│   │   ├─ embeddings.py   · Gemini 임베딩 호출
│   │   ├─ db/             · 읽기전용 DB 접속 + SQL 안전장치
│   │   ├─ routers/ai.py   · /ai 엔드포인트
│   │   ├─ schemas/ai.py   · 요청·응답 데이터 모양(Pydantic)
│   │   └─ graph/          · ③ LangGraph 노드들 (라우터/SQL/문서/답변)
│   ├─ corpus/             · 가이드 문서 원본(임베딩 대상)
│   └─ tests/              · 자동 테스트
├─ web/src/                ← ④ 웹 프론트 (Next.js App Router)
│   ├─ app/                · 페이지(라우트). (auth)/(user)/(admin) 그룹
│   ├─ components/         · 재사용 UI 조각
│   └─ lib/                · Supabase 클라이언트·헬퍼 함수
├─ app/lib/                ← ⑤ Flutter 앱 (Dart)
│   ├─ core/              · 공통(테마·Supabase 초기화·포맷)
│   └─ features/          · 기능별(auth/listings/ai_search/chat)
├─ docs/                   ← 기획·테스트·이 학습 문서
└─ _bmad-output/           ← BMad 산출물(스토리·아키텍처 문서 등)
```

## 0-8. 학습 순서 추천

웹앱이 처음이라면 **"데이터가 어디 있고 → 어떻게 화면에 나오나"** 순서로 따라가는 게 가장 이해가 쉽습니다.

2. **[04-web-frontend.md] 웹 프론트** — 그 데이터가 실제 화면에 어떻게 나오고 사용자가 어떻게 조작하는지. (가장 친숙할 부분)
3. **[02-backend-api.md] 백엔드 API** — AI 검색 요청이 서버에서 어떻게 처리되는지의 뼈대.
4. **[03-langgraph-ai.md] LangGraph AI** — 이 프로젝트의 하이라이트. 자연어가 어떻게 SQL/문서검색으로 바뀌는지.
5. **[05-flutter-mobile.md] Flutter** — 웹과 "같은 백엔드, 다른 화면 기술"이 어떻게 구현되는지 비교하며.
