# 데이터베이스 구조 설명서 — bmad-encar-demo

> 중고차 직거래 서비스(엔카·헤이딜러 류)의 데이터베이스 구조를 **시연·발표용**으로 정리한 문서입니다.
> 테이블 간 관계(1:1·1:N·N:N), 각 컬럼의 자료형과 그렇게 정한 이유, 보안 설계(RLS)까지 비전공자도 이해할 수 있게 풀어 썼습니다.
>
> - **DB 종류**: PostgreSQL 17 (Supabase 호스팅, 서울 리전)
> - **확장 기능**: `pgvector` — AI 의미검색(임베딩)용 벡터 자료형 지원
> - **스키마 단일 출처**: `supabase/migrations/` 폴더의 SQL 파일들(0001~0009)
>
> *용어 한 줄 설명*
> - **테이블(table)**: 엑셀 시트처럼 행(레코드)·열(컬럼)로 데이터를 담는 저장 단위.
> - **자료형(data type)**: 그 칸에 들어갈 값의 종류(숫자·글자·날짜 등).
> - **PK(Primary Key, 기본키)**: 각 행을 유일하게 구분하는 대표값(주민번호 같은 것).
> - **FK(Foreign Key, 외래키)**: 다른 테이블의 행을 가리키는 연결고리.
> - **RLS(Row Level Security, 행 수준 보안)**: "이 사람은 이 행만 볼 수 있다"를 DB가 직접 강제하는 잠금장치.

## 1. 전체 그림 — 테이블 5개와 관계

이 서비스의 데이터는 **5개의 핵심 테이블**로 이뤄져 있습니다.
(`auth.users`는 Supabase가 제공하는 로그인 계정 테이블로, 우리가 만든 게 아니라 **빌려 쓰는** 테이블입니다.)

```
        ┌────────────────────────┐
        │  auth.users (Supabase) │   로그인 계정(이메일·비밀번호) — Supabase 기본 제공
        │  PK: id                │
        └───────────┬────────────┘
                    │ 1 : 1   (가입하면 트리거가 자동으로 프로필 1개 생성)
                    ▼
        ┌────────────────────────┐
        │  profiles              │   우리 서비스의 사용자(역할·상태·이름)
        │  PK: id                │
        └───┬──────────┬─────────┘
            │          │
   seller_id│ 1:N      │ buyer_id / seller_id  1:N  (한 사람이 여러 방의 당사자)
            ▼          ▼
   ┌─────────────┐   ┌────────────────────┐
   │  listings   │   │   chat_rooms       │   문의 채팅방(매물 1건당 구매자↔판매자 1:1)
   │  PK: id     │◄──┤   PK: id           │
   └──────┬──────┘ listing_id 1:N         │
          │        └─────────┬──────────┘
          │                  │ 1:N  (방 하나에 메시지 여러 개)
          │                  ▼
          │        ┌────────────────────┐
          │        │   chat_messages    │   채팅 메시지(영속·불변)
          │        │   PK: id           │
          │        └────────────────────┘
          │
   (임베딩으로 AI 검색)
          │
   ┌──────▼───────┐
   │ guide_documents │   차량 상식·구매 가이드 문서(AI RAG용, 다른 테이블과 FK 연결 없음)
   │ PK: id          │
   └─────────────────┘
```

### 관계 요약표

| 관계 | 종류 | 연결 컬럼 | 의미 |
|---|---|---|---|
| `auth.users` ↔ `profiles` | **1 : 1** | `profiles.id` = `auth.users.id` | 로그인 계정 1개당 서비스 프로필 1개 |
| `profiles` → `listings` | **1 : N** | `listings.seller_id` | 한 판매자가 매물 여러 개 등록 |
| `profiles` → `chat_rooms` | **1 : N** (×2) | `chat_rooms.buyer_id`, `chat_rooms.seller_id` | 한 사람이 여러 방에 구매자/판매자로 참여 |
| `listings` → `chat_rooms` | **1 : N** | `chat_rooms.listing_id` | 한 매물에 여러 문의 방 |
| `profiles` → `chat_messages` | **1 : N** | `chat_messages.sender_id` | 한 사람이 메시지 여러 개 발송 |
| `chat_rooms` → `chat_messages` | **1 : N** | `chat_messages.room_id` | 한 방에 메시지 여러 개 |
| **구매자 ↔ 판매자** | **N : N** | `chat_rooms`가 다리(junction) | 구매자·판매자는 여러 매물에 걸쳐 서로 여러 번 대화 가능 |

> **N:N(다대다)이 직접 없는 이유**: 관계형 DB는 다대다를 직접 표현하지 못하고, **중간 테이블(junction table)**로 풀어냅니다.
> 여기서는 `chat_rooms`가 "(어느 매물에 대해) 어느 구매자와 어느 판매자가" 만났는지를 한 행에 묶어, 다대다 관계를 1:N 두 개로 분해합니다.

## 2. 테이블별 상세

각 컬럼의 **자료형**과 **그렇게 정한 이유**를 함께 적었습니다.
자료형 약어: `uuid`=무작위 고유 ID, `text`=글자, `int`=정수, `bigint`=큰 정수, `bool`=참/거짓, `timestamptz`=시간대 포함 시각, `vector(768)`=768개 숫자로 된 AI 임베딩, `text[]`=글자 목록(배열).

### 2-1. `profiles` — 사용자 프로필

> 로그인 계정(`auth.users`)에 우리 서비스만의 정보(역할·상태·이름)를 덧붙이는 테이블.
> **비밀번호·이메일은 여기 두지 않습니다** — 그건 Supabase의 `auth.users`가 보안 관리합니다.

| 컬럼 | 자료형 | 제약 | 자료형·설계를 그렇게 정한 이유 |
|---|---|---|---|
| `id` | `uuid` | PK, FK→`auth.users.id` | 로그인 계정 ID를 그대로 프로필 ID로 씀(1:1 보장). 계정이 삭제되면 프로필도 같이 삭제(`on delete cascade`). |
| `role` | `text` | `buyer`/`seller`/`admin`만 허용 | 역할은 정해진 3가지뿐이라 `CHECK` 제약으로 오타·잘못된 값을 DB가 막음. 숫자코드 대신 글자라 읽기 쉬움. |
| `status` | `text` | `active`/`suspended`, 기본 `active` | 관리자가 회원을 정지(suspended)할 수 있어 상태값 필요. 역시 두 값만 `CHECK`로 강제. |
| `created_at` | `timestamptz` | 기본 `now()` | 가입 시각. 시간대 포함형(`timestamptz`)이라 서버·사용자 위치가 달라도 정확. |
| `name` | `text` | nullable | 화면 표시용 이름 = 이메일의 `@` 앞부분. 관리자가 회원을 UUID 대신 이름으로 식별하려고 추가(0009). |

**핵심 설계**
- **가입 자동 연동**: 누군가 회원가입하면(`auth.users`에 행 추가) **트리거**가 자동으로 `profiles` 행을 만듭니다. 사람이 따로 안 만들어도 됩니다.
- **권한 상승 차단**: 가입 시 역할을 `admin`으로 위조해 보내도, 트리거가 `buyer`로 강제 교정합니다(관리자는 가입으로 못 됨).
- **RLS**: 본인 행만 조회 가능 + 관리자는 전체 조회 가능.

### 2-2. `listings` — 차량 매물 (★ 가장 중요한 테이블)

> 판매 중인 중고차 한 대 = 한 행. 차량 정보 15개 항목 + 시스템 정보(소유자·상태·AI 임베딩).

| 컬럼 | 자료형 | 제약 | 자료형·설계를 그렇게 정한 이유 |
|---|---|---|---|
| `id` | `uuid` | PK, 기본 무작위 생성 | 순번(1,2,3…) 대신 추측 불가한 무작위 ID라 URL로 매물 수를 엿보거나 순회하기 어려움(보안). |
| `seller_id` | `uuid` | FK→`profiles.id` | 소유자. 본인만 수정/삭제 가능(RLS). 판매자 삭제 시 매물도 함께 삭제. |
| `status` | `text` | `on_sale`/`sold`, 기본 `on_sale` | 판매중/판매완료 두 상태만. **구매자에겐 `sold`를 숨김**(RLS). |
| `embedding` | `vector(768)` | nullable | 설명·옵션 텍스트를 AI가 768개 숫자로 변환한 값. **의미 기반 검색**("가성비 패밀리 SUV")에 사용. Gemini 임베딩 모델이 768차원이라 일치. |
| `created_at` / `updated_at` | `timestamptz` | 기본 `now()` | 등록·수정 시각. 수정 시 트리거가 `updated_at` 자동 갱신, `created_at`은 위조 못 하게 보존. |
| `manufacturer` | `text` | 15개 제조사 `CHECK` | 현대·기아·BMW… 고정 목록. UI 드롭다운·AI 검색·시드데이터가 **모두 같은 목록**을 써야 해서 DB가 기준값으로 못박음. |
| `model` | `text` | 필수 | 모델명은 종류가 무한해 자유 입력. |
| `body_type` | `text` | 12개 차종 `CHECK` | 경차·SUV·승합 등 분류 고정. |
| `year` | `int` | `1990 ~ 올해+1` | 연식. 상한을 "올해+1"로 둬 신차 연식 허용하되 하드코딩 회피(자동 계산). |
| `price` | `bigint` | `>= 0` | 가격(원). 일반 `int`는 약 21.4억까지라, 초고가 매물 대비 더 큰 `bigint` 사용. |
| `mileage` | `int` | `>= 0` | 주행거리(km). 음수 불가. |
| `color` | `text` | 9개 색상 `CHECK` | 색상 고정 목록. |
| `fuel` | `text` | 가솔린/디젤/하이브리드/전기/LPG | 연료 고정 목록. |
| `transmission` | `text` | 자동/수동 | 변속기 두 종류. |
| `displacement` | `int` | `>= 0` | 배기량(cc). 전기차는 0 허용. |
| `seats` | `int` | `2 ~ 11` | 승차인원. 현실 범위로 제한. |
| `region` | `text` | 17개 시·도 `CHECK` | 지역 고정 목록(서울~제주). |
| `accident_free` | `bool` | 기본 `true` | 무사고 여부. 참/거짓이라 `boolean`. |
| `options` | `text[]` | 기본 빈 배열 | 옵션 여러 개를 한 칸에 목록으로. AI 임베딩 대상. |
| `description` | `text` | nullable | 상세 설명. AI 임베딩 대상. |
| `seller_name` | `text` | nullable | 판매자 표시 이름(이메일 @앞부분). 목록에서 식별용(0007, 비정규화). |

**핵심 설계**
- **사진 컬럼이 없는 이유**: 데모는 AI 검색(RAG)에 집중하므로 이미지 저장(Supabase Storage)을 생략했습니다.
- **고정 목록을 `CHECK`로 강제하는 이유**: UI 드롭다운·AI Text-to-SQL·시드데이터가 같은 단어를 써야 검색이 어긋나지 않습니다. DB가 "허용 단어 사전" 역할을 합니다.
- **`seller_name`은 일부러 중복 저장(비정규화)**: 구매자는 RLS상 남의 프로필을 못 읽어, 조인으로 판매자 이름을 가져올 수 없습니다. 그래서 매물 행에 이름을 복사해 둡니다.

### 2-3. `chat_rooms` — 문의 채팅방

> "어느 매물에 대해, 어느 구매자가, 어느 판매자에게" 문의했는지를 담는 방. **구매자↔판매자 다대다 관계를 푸는 중간 테이블**.

| 컬럼 | 자료형 | 제약 | 이유 |
|---|---|---|---|
| `id` | `uuid` | PK | 방 고유 ID. |
| `listing_id` | `uuid` | FK→`listings.id` | 어느 매물에 대한 문의인지. 매물 삭제 시 방도 정리. |
| `buyer_id` | `uuid` | FK→`profiles.id` | 문의를 연 구매자. |
| `seller_id` | `uuid` | FK→`profiles.id` | 매물 소유자(판매자). |
| `created_at` | `timestamptz` | 기본 `now()` | 방 생성 시각. |
| `buyer_name` / `seller_name` | `text` | nullable | 양쪽 표시 이름(이메일 @앞부분). 관리자·당사자 화면에서 UUID 대신 이름 표시(0008). |

**핵심 설계**
- **`UNIQUE(listing_id, buyer_id, seller_id)`**: 같은 매물·구매자·판매자 조합은 **방 1개만**. 같은 매물에 다시 문의해도 기존 방을 재사용.
- **`CHECK(buyer_id <> seller_id)`**: 자기 자신과의 방 금지(내 매물엔 문의 불가).
- **위조 방지 트리거(0016)**: 구매자가 보낸 `seller_id`를 무시하고, DB가 **그 매물의 실제 소유자로 강제 교정**합니다. "엉뚱한 사람에게 방 강제 생성" 같은 공격을 원천 차단.
- **RLS**: 방의 당사자(buyer/seller)만 조회·생성. 제3자는 0건. 관리자는 전체 열람.

### 2-4. `chat_messages` — 채팅 메시지

> 방 안에서 오간 메시지 한 줄 = 한 행. 한 번 쓰면 **수정·삭제 불가(영속)** — 관리자만 삭제 가능.

| 컬럼 | 자료형 | 제약 | 이유 |
|---|---|---|---|
| `id` | `uuid` | PK | 메시지 고유 ID. |
| `room_id` | `uuid` | FK→`chat_rooms.id` | 어느 방의 메시지인지. 방 삭제 시 메시지도 정리. |
| `sender_id` | `uuid` | FK→`profiles.id` | 보낸 사람. |
| `body` | `text` | 공백만 있는 글 금지 | 메시지 본문. `CHECK(length(btrim(body)) > 0)`로 빈 메시지 차단. |
| `created_at` | `timestamptz` | 기본 `now()` | 보낸 시각. 시간순 정렬·폴링에 사용. |

**핵심 설계**
- **실시간 대신 폴링(polling)**: 3~5초마다 새 메시지를 조회하는 방식. `(room_id, created_at)` 인덱스로 빠르게.
- **RLS**: 메시지가 속한 방의 당사자만 읽기/쓰기. 남의 명의로 메시지 위조 불가(`sender_id = 본인` 강제).

### 2-5. `guide_documents` — AI 가이드 문서 (RAG 코퍼스)

> 차량 상식·구매 가이드 문서 모음. **다른 테이블과 FK로 연결되지 않은 독립 테이블**로, AI가 질문에 답할 때 참고하는 지식 창고입니다.

| 컬럼 | 자료형 | 제약 | 이유 |
|---|---|---|---|
| `id` | `uuid` | PK | 문서 고유 ID. |
| `title` | `text` | 필수 | 문서 제목. |
| `content` | `text` | 필수 | 문서 본문. 이걸 임베딩해서 검색. |
| `embedding` | `vector(768)` | nullable | 본문의 768차원 임베딩(L2 정규화). 질문과 의미가 가까운 문서를 찾는 데 사용. |
| `created_at` | `timestamptz` | 기본 `now()` | 생성 시각. |

> *RAG(Retrieval-Augmented Generation)*: AI가 답하기 전에 관련 문서를 먼저 **검색**해서, 그 내용을 근거로 답을 생성하는 방식. 환각(없는 말 지어냄)을 줄입니다.

## 3. 발표 때 강조하면 좋은 설계 포인트

### ① 보안을 "앱"이 아니라 "DB"에서 강제 — RLS
누가 어떤 행을 볼 수 있는지를 앱 코드가 아니라 **데이터베이스가 직접** 막습니다. 앱에 버그가 있어도 DB 잠금은 뚫리지 않습니다.
- 구매자: 판매중(`on_sale`) 매물만 보임 — 판매완료(`sold`)는 숨김.
- 판매자: 본인 매물은 상태와 무관하게 전부 보임.
- 채팅: 방 당사자만. 제3자는 한 줄도 못 봄.
- 관리자: 전체 열람·관리 권한.

### ② AI 전용 "읽기 전용" 계정 — `ai_readonly`
AI가 DB를 조회할 때 쓰는 별도 권한(롤)을 만들어, **읽기만 가능하고 쓰기는 일절 불가**하게 했습니다(NFR2). AI가 실수나 공격으로 데이터를 바꿀 수 없습니다.

### ③ 위조 방지를 트리거로 못박기
클라이언트가 보낸 값을 그대로 믿지 않고, DB 트리거가 **권위 있는 값으로 강제 교정**합니다.
- 매물 등록 시 `seller_name`을 실제 이메일에서 도출.
- 채팅방 생성 시 `seller_id`를 실제 매물 주인으로 교정.
- `updated_at` 자동 갱신, `created_at` 위조 차단.

### ④ AI 의미검색 — pgvector + HNSW 인덱스
텍스트를 768개 숫자(임베딩)로 바꿔, "비슷한 의미"를 수학적 거리로 찾습니다. `HNSW`라는 그래프형 인덱스(`m=16, ef_construction=200`)로 빠르게 검색합니다.

### ⑤ 마이그레이션으로 이력 관리
모든 스키마 변경을 `supabase/migrations/`의 번호 매긴 SQL 파일(0001~0009)로 기록해, **언제·무엇을·왜 바꿨는지** 추적 가능합니다.
