# 역할별 내비게이션 · 정보구조(IA) 규칙

> 작성 2026-06-24. 출처: 메인화면 개편 타이밍 party-mode 결정(deferred-work.md "🟢 결정 완료") + 🏗️Winston 산출물.
>
> **이 문서의 목적**: 이것은 화면 디자인이 아니라 **계약(contract)**이다. web과 Flutter(Epic 7)가 **공유**해야 할 "어떤 역할이 어디서 무엇을 하고, 어떻게 이동·복귀하는가"의 규칙만 담는다. 코드(React/CSS·위젯)는 이월되지 않지만, 이 규칙은 이월된다. Epic 7의 화면을 짤 때 이 문서를 입력으로 쓴다.
>
> **핵심 범위 사실(AR9)**: 모바일(Epic 7) 대상 = **구매자 + 판매자**. **관리자는 web 전용**(모바일 제외). 따라서 §1~§3에서 *모바일로 이월되는 규칙은 구매자·판매자 행(行)뿐*이고, 관리자 규칙은 web 전용 표시(🖥️)다.

관련 라우트(현재 web 기준): 홈 `/` · 탐색 `/search` · 상세 `/listings/[id]` · AI `/ai` · 채팅 `/chat`·`/chat/[roomId]` · 판매 `/sell`·`/sell/[id]/edit` · 관리자 `/admin/*`(web 전용).

## §1. 역할 × 진입점 매트릭스 (로그인 후 첫 화면이 무엇을 보여주는가)

| 역할 | 첫 화면(홈) | 홈이 보여주는 것 | 이월 |
|---|---|---|---|
| **구매자(buyer)** | `/` | ① 본인정보(이름·역할) · ② 매물탐색 **미리보기**(상위 N건 읽기전용 카드 + '더보기'→`/search`) · ③ AI 채팅 전역 진입 · 문의 채팅 진입 | ✅ 모바일 7.2 |
| **판매자(seller)** | `/` | 구매자와 동일(①②③+문의채팅) **＋ 내 매물 등록·관리 진입**(`/sell`) | ✅ 모바일 7.3·7.4 |
| **관리자(admin)** 🖥️ | `/admin` | 운영 허브(회원·매물·거래·채팅 관리 4진입) — **로그인 시 `/`→`/admin` 자동 유도** | ❌ web 전용 |

**규칙 R1**: 역할은 *계정 속성*이 아니라 *행위*다 — **판매자도 잠재적 구매자**다. 그래서 판매자 홈은 구매자 홈의 **상위집합**(구매자 동선 전부 + 판매 동선 추가)으로 둔다. 별도의 분리된 판매자 전용 홈을 만들지 않는다.

**규칙 R2**: 홈은 "버튼 허브"가 아니라 **역할의 1순위 과업으로 바로 착지**한다. 구매자/판매자의 1순위 = 매물 탐색 → 그래서 ②미리보기를 홈에 직접 노출(버튼 한 번 더 누르게 하지 않음). 단 홈은 필터·URL 상태를 **소유하지 않는다**(미리보기는 읽기 전용, 본격 탐색은 `/search`가 소유) — 회귀·상태 이원화 방지(Amelia·Winston).

---

## §2. 역할별 1차 내비 항목과 순서

홈/주 내비에 노출되는 **1차 동선의 항목과 순서**(표현은 플랫폼별로 — web=링크/섹션, Flutter=하단탭/FAB):

- **구매자**: ① 본인정보 → ② 탐색(미리보기→전체) → ③ AI 검색(전역 진입) → 문의 채팅
- **판매자**: 위 구매자 4개 → ＋ 내 매물 등록·관리
- **관리자** 🖥️: 회원 관리 → 매물 관리 → 거래 내역 → 채팅 관리 (운영 허브, 모바일 없음)

**규칙 R3 (AI는 전역 진입점)**: AI 검색(③)은 특정 화면 안의 한 메뉴가 아니라 **어느 화면에서든 한 번에 닿는 보조 동작**이다. web에서는 가벼운 플로팅/상시 진입으로, **Flutter에서는 FAB(Floating Action Button, 머티리얼 표준 위젯) 또는 상시 탭**으로 재현한다. *web의 FAB CSS는 이월되지 않으나, "AI=전역 진입점"이라는 이 규칙은 이월된다.*

**규칙 R4**: 문의 채팅은 구매자·판매자 **공통**(둘 다 당사자가 될 수 있음). 역할로 숨기지 않는다.

---

## §3. 복귀(뒤로가기) 규칙

**규칙 R5 (출처 복귀가 기본)**: 여러 진입 경로에서 같은 상세로 들어오는 화면의 '돌아가기'는 **들어온 그 화면으로 복귀**한다(브라우저/모바일 뒤로가기 시맨틱). 진입 경로별 고정 목적지로 하드코딩하지 않는다.
- 구현 패턴: 이전 히스토리가 있으면 뒤로, 없으면(새 탭·직접 URL·딥링크) **그 화면이 속한 기본 목록으로 폴백**.
- web 선례: 관리자 매물 상세 `BackButton`(`router.back()` + 폴백 `/admin/listings`). 🖥️ web 전용이지만 *규칙 자체는 이월* — Flutter의 시스템 뒤로가기 + 딥링크 폴백 설계에 그대로 적용.
- 모바일 적용 지점(이월): 구매자 매물 상세(탐색·AI결과·딥링크에서 진입) → 출처 복귀 + 폴백 `/search` 상당.

**규칙 R6**: 파괴적 액션 후(매물 삭제·구매완료 전환 등)의 이동은 출처 복귀가 아니라 **안전한 목록으로 명시 이동**한다(삭제된 항목으로 되돌아가지 않게). 출처 복귀(R5)와 구분.

## §4. 플랫폼 무관 불변식 (web ↔ Flutter 공유 vs 셸 표현 경계)

**이월된다 (공유 계약 — 이 문서가 박는 것):**
- §1 역할×진입점 매트릭스(누가 어디로 착지하는가)
- §2 역할별 1차 내비 항목·순서
- §3 복귀 규칙(출처 복귀 + 폴백, 파괴적 액션 후 명시 이동)
- 이미 확정된 하위 계약: Supabase 스키마·RLS·`status` enum·`/ai/search` 응답 포맷(AR5·AR9) · buyer≠seller 무결성(3중 차단)

**이월되지 않는다 (플랫폼 셸 표현 — 각자 새로 구현):**
- 레이아웃 정렬(상단/중앙)·`max-w-md`·CSS·React 컴포넌트 트리·위젯 트리
- ④ 데스크톱 반응형(web 전용, **Cut** — Flutter는 단일 폼팩터라 무의미)
- ⑤ 관리자 대시보드 UI(관리자=web 전용이라 모바일 자체가 없음. 집계쿼리 계약은 모바일 관리자가 로드맵에 확정될 때만 별도)

**규칙 R7 (경계 원칙)**: 무엇을 손볼 때 "이게 §1~§3 규칙(이월)인가, 셸 표현(web 전용)인가"를 먼저 가른다. 규칙이면 이 문서를 고치고 양 플랫폼에 반영, 표현이면 해당 플랫폼에서만 처리한다.
