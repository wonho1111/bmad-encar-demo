- **성격**: 이번 증분(이미지·UI개편·반응형·랜딩) PRD/UX/스키마에서 **반드시 결정해야 할 열린 항목**. `reference-synthesis.md`와 함께 `bmad-prd` 입력.

## 1. 상단 내비게이션 · IA 재검토 (UX)
- **배경**: 목업 상단 메뉴 = `매물 탐색 · 내 차 팔기 · 문의` + (우측) `로그인 · 내 차 등록`.
- **결정 필요**:
  - `문의` 버튼이 최상위 내비에 필요한가? (문의는 매물 상세의 판매자 채팅이 주 경로 — 전역 내비 자리값이 맞는지)
  - `매물 탐색` → **`내 차 사기`**로 명칭 변경 검토(판매="내 차 팔기"와 대구, 사용자 의도 언어).
- **참고**: 기존 역할별 내비 규칙은 `nav-ia-rules.md`. 이 항목은 그 연장선 재검토.

## 2. 계정 역할 모델 재검토 (구매자/판매자)
- **배경**: 목업의 `내 차 등록` 버튼 노출은 현재 **계정이 판매자/구매자 역할로 구분**돼 있음을 전제(회원가입 시 역할 선택 — Story 1.2).
- **결정 필요**: 역할 구분을 **유지 / 계정 내 역할 전환 허용 / 구매자·판매자 통합**(누구나 사고팔기) 중 어디로 갈지.
  - 통합 시: `내 차 등록`을 모두에게 노출, RLS·프로필 스키마·가입 흐름 영향(Epic 1 전반). 영향 범위 큼 → PRD에서 신중히.
- **주의**: 역할 통합은 **인증/권한 구조(profiles.role, RLS)** 를 건드리므로 이미지/랜딩 작업과 별개의 큰 결정. 분리 판단 권장.

## 3. ★ 카드 뱃지·옵션 데이터 구조 (가장 큰 항목 — 후속 리서치 필요)
- **배경**: 목업 카드가 옵션(후방카메라·스마트키 등)을 하단 태그로, 무사고를 사진 위 오버레이로 표시. 사용자 지적:
  - **(a) 옵션 희소도**: 스마트키·블루투스·후방카메라는 이제 **보편화**돼 변별력이 없음 → 태그로 나열하기보다 **오토파일럿·파노라마·V2L 등 희소성 있는 옵션을 우선 노출**.
  - **(b) 옵션 아닌 신뢰속성 분리**: `무사고·1인소유·비흡연`은 "장비 옵션"이 아니라 **차량 신뢰속성**이라 별도 데이터로 구분·관리해야 함.
  - **(c) 검증 가능성 차등**(중요):
    - `무사고` — 보험이력·성능기록부에서 도출(객관 근거). 현재 seed는 `accident_free`(boolean) 보유.
    - `1인소유` — 보험이력 데이터가 있어야 도출 가능 → **추가 데이터 필요**.
    - `비흡연` — **객관 검증 수단 없음**(판매자 자기신고). 데모엔 실제 보험이력 연동이 없으므로 이 셋의 근거 수준을 어떻게 다룰지 정의 필요.
  - **(d) 배치 방향(사용자 선호)**: 무사고·1인소유·비흡연 등 **신뢰속성 = 카드 상단(무사고 옆)**, **옵션 = 하단 + 희소 옵션 우선**.
- **결정 필요(스키마·표시)**:
  1. 신뢰속성을 `options(text[])`와 분리된 **별도 구조**(예: 신뢰속성 boolean 컬럼들 또는 태그 테이블)로 둘지.
  2. 옵션 **희소도** 개념을 어떻게 표현할지(보편 옵션 목록 정의 후 숨김 / 희소도 순 정렬 / 큐레이션).
  3. 데모 제약(보험이력 API 없음) 하에서 각 신뢰속성의 **근거 라벨**(검증됨 vs 판매자 신고)을 UI에 어떻게 표기할지.
- **🔎 후속 리서치 필요**(agent-reach, sonnet — PRD 前 권장):
  - 중고차 플랫폼이 **무사고/1인소유/비흡연/옵션**을 각각 어떻게 **저장·검증·표시**하는가.
  - **보험이력·성능기록부**에서 나오는 데이터 항목(무사고·소유자변경·용도이력 등)과, 자기신고 항목의 구분 관행.
  - 옵션 **표준화/희소도** 처리(보편 옵션 vs 셀링포인트 옵션) 사례.
  - (참고) 조사 종합 `reference-synthesis.md`에서 기아 "보험이력없음", K Car 특징 태그, 엔카 성능기록부 노출 등 단서 이미 있음 → 그 위에 심화.
