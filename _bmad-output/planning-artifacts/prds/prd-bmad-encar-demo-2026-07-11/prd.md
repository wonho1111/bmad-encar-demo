# 차장님 — UI·이미지 고도화 증분 PRD

> **브라운필드 증분 PRD.** Epic 1~7·v1.0.0까지 완료된 `bmad-encar-demo`(중고차 직거래 데모) 위에 얹는 실서비스급 고도화 증분. 원본 PRD(`prds/prd-bmad-encar-demo-2026-06-17/prd.md`, FR1~25, final)를 **기준선**으로 삼고, 이 문서는 그 **델타(신규·변경)만** 정의한다. 원본에서 바뀌지 않는 FR은 그대로 유효하다.

## 1. 개요 — 이 증분이 바꾸는 것

원본 데모는 "AI 검색 RAG 경험"을 최종 목표로 삼아 **사진을 의도적으로 제외**하고(FR5), 채팅을 **폴링**으로(FR20/21) 구현했다. 이 증분은 데모를 **실서비스에 가까운 제품 경험**으로 끌어올린다. 세 축:

1. **이미지** — 매물 사진을 도입한다(원본 FR5의 "사진 미포함" 결정을 **뒤집는다**). 기존 매물 100건은 사진이 없으므로 **nullable(선택) 도입 + "사진 준비중" 플레이스홀더**로 안 깨지게 한다.
2. **UI·랜딩·반응형 개편** — 매물 카드를 이미지 중심으로 재설계하고, **AI 검색을 주인공으로 한 랜딩페이지**를 신설하며, 단일 코드 반응형을 고도화한다. **Flutter 앱도 같은 디자인 언어로 개편**한다.
3. **실시간화** — 문의 채팅을 **폴링 → 웹소켓 실시간**으로 전환하고(FR20/21 개정), **RAG 검색 품질**을 개선한다(FR15 연장).

**확정된 제품 방향**(랜딩 목업 v1 승인): 서비스명 **차장님** · 디자인 톤 **페트롤+앰버**(CSS 토큰 기반 = 후일 변경 저비용) · 이미지 비율 **5:3** · **AI 자연어 검색 히어로** · **마켓플레이스형 그리드(인기/최신 2단)** · 카드 = 대표사진 + 무사고 오버레이 + "N장" 배지 + 가격 대형 + 옵션/신뢰속성 태그.

## 2. 사용자 여정 델타 (원본 UJ-1~3 위의 변화)

원본 여정(UJ-1 구매자 AI검색, UJ-2 판매자 등록, UJ-3 관리자)은 유효하다. 이 증분이 바꾸는 **체감 지점**만 서술한다.

- **UJ-1 구매자 (지훈)** — 서비스 진입 시 필터가 아닌 **랜딩의 AI 검색 히어로**를 마주하고, 그 아래 **사진이 실린 매물 카드**(인기/최신 2단)를 스캔한다. 카드에서 **대표사진·무사고 뱃지·가격**을 한눈에 보고, 상세에서 **여러 장의 사진 갤러리**를 넘겨본다. 판매자에게 문의하면 **실시간으로** 답이 온다(폴링 지연 없음).
- **UJ-2 판매자 (민서)** — 매물 등록 시 **사진을 여러 장 올린다**(선택 — 안 올려도 등록 가능). 무사고·1인소유·비흡연 같은 **신뢰속성을 "판매자 신고"로 체크**한다(옵션과 별개 영역). 구매자 문의에 **실시간**으로 응대한다.
- **[ASSUMPTION] 신규 매물은 사진 등록을 권장하되 강제하지 않는다**(기존 100건과의 일관성 + 데모 단순성). 사진 없는 매물은 목록에서 "사진 준비중"으로 정직하게 표시된다.
- **Flutter 앱 사용자** — 웹과 동일한 디자인 언어(색·카드 구성·신뢰속성 표시)의 개편된 화면에서 위 경험을 모바일 네이티브로 누린다.

## 3. 신규 · 변경 기능 요구사항 (FR)

> 원본 FR1~25에 이어 **FR26부터 신규**. 기존 FR을 고치는 경우 "(개정)"으로 표시한다. 모든 신규 FR은 **데모/과제 규모(A2 단순함 우선, 과설계 금지)**를 전제한다.

### F7. 매물 이미지 (신규)

- **FR26** 판매자는 매물에 **사진을 여러 장 등록**할 수 있다(선택 사항 — 사진 없이도 등록 가능). 이미지는 **고정 비율 5:3**으로 크롭/표시한다. 기존 매물(사진 없음)은 그대로 유효하다(**nullable 도입, non-null 금지**).
  - *[ASSUMPTION] 매물당 사진 최대 10장, 첫 장을 대표사진으로 지정(순서 변경 가능).*
- **FR27** **사진이 없는 매물**은 목록·상세에서 빈 영역 대신 **"사진 준비중" 플레이스홀더**(아이콘 + 문구)로 표시한다. (근거: 현대 "판매예정"·KB `noimage` 선례. 기존 100건이 전부 이 상태)
- **FR28** 매물 상세는 **대표사진 + 갤러리**(스와이프/넘김)로 여러 장을 보여준다(FR10 개정). 목록 카드는 **대표사진 1장**만 보여준다. **AI 검색 응답의 매물 카드(FR17)에도 대표사진**을 반영한다(사진 없으면 "사진 준비중" 플레이스홀더).
- **FR29** 매물 카드에 **사진 개수 "N장" 배지**를 표시한다(사진 있는 매물에 한함). (근거: 기아 CPO 패턴 — 사진 충실도를 목록에서 구분)

### F8. 매물 카드 · 신뢰속성 · 옵션 (신규 — 데이터 리서치 결론 반영)

- **FR30** **신뢰속성(무사고·1인소유·비흡연)을 옵션(장비)과 분리**해 관리·표시한다.
  - **데이터**: 옵션(장비, `options`) ↔ 신뢰속성(별도 필드: 무사고 상태 / 1인소유 / 비흡연)로 스키마·UI 양쪽에서 구분한다.
  - **검증 티어**: 실제 업계는 ①법정·보험개발원 ②플랫폼 진단 ③판매자 자기신고(면책)의 3단이나, **본 데모는 보험이력·성능기록부 연동 인프라가 없으므로 세 속성 모두 "판매자 신고" 티어로 통일**한다(엔카·KB "핫마크" 선례와 동일).
  - **정직성(필수)**: 신뢰속성 뱃지 옆/상세에 **"판매자 제공 정보" 면책 라벨**을 붙인다(엔카 "사실여부 직접 확인" 선례). 과장 없는 문구를 쓰고, **가짜 검증 UI를 만들지 않는다**.
  - **배치**: 신뢰속성은 **카드 상단(사진 위 오버레이 또는 상단 뱃지)**, 옵션은 **카드 하단**.
- **FR31** 매물 카드는 **희소성 있는 옵션을 우선 노출**한다.
  - 스마트키·블루투스·후방카메라·열선시트 등 **보편 옵션은 저순위**(코드의 "보편 옵션 목록"으로 관리), 오토파일럿·파노라마선루프·V2L 등 **희소 옵션은 상위**(옵션명→우선순위 매핑).
  - **카드에는 상위 3~4개만** 노출(희소 옵션이 없으면 보편 옵션으로 fallback), **상세에서는 카테고리별 전량** 노출.
  - *옵션 금전가치 환산 표시는 데모 범위 밖(seed에 옵션가 없음).*
  - *기술부채 #11(옵션 쉼표 문자열 라운드트립 손실, `SellForm.tsx` 동일 표면) — F8이 옵션 표시/우선순위를 손대므로 **이번에 함께 정비 권장**(옵션 저장구조를 배열/정규화로). 부담되면 명시적 이월.*
- **FR32** 매물 카드 정보 위계를 **차량명 → 연식 → 주행거리 → 연료 → 지역** 순 한 줄로 정리하고 **가격을 가장 크게(대형 볼드)** 표시한다. **찜 버튼은 사진 밖(정보 영역)**에 둔다(엔카 2025 개편). **비교(compare) 기능은 이번 범위에서 제외**한다(다중선택+비교표=데모 과설계, A2).
- **FR55 찜(좋아요) 기능** (신규, party-mode) — 로그인 사용자는 매물을 **찜**할 수 있다(listing_id 저장, 카드에서 토글). **"찜한 매물" 목록**에 도달할 수 있다. 재방문의 핵심 고리. (평판·비교표는 범위밖)
- **FR56 판매자 정보 노출** (신규, party-mode) — 매물 상세에 **경량 판매자 정보 섹션**을 둔다: 닉네임 + 가입 시점 + "이 판매자의 다른 매물 N건". P2P의 "이 사람 믿을 만한가" 불안 완화(역할 통합으로 누구나 판매자가 되므로 필요). *평판 점수·응답률·인증 배지는 데이터가 없어 [범위밖](가짜 표시는 FR30 정직성과 충돌).*

### F9. 랜딩페이지 (신규)

- **FR33** 랜딩(홈) 첫 화면은 **AI 자연어 검색을 주인공**으로 한다 — 히어로에 검색 입력 + 예시 질의 칩. (우리 차별점; 원본 FR12 AI 검색으로 연결)
- **FR34** 히어로 아래에 **매물 그리드**를 바로 노출한다 — **인기 매물 / 최근 등록 매물 2단**(서비스 소개 없이 매물부터, K Car 안심직거래 참조).
  - **"인기" = 실제 조회수(view_count) 정렬**(OI-5 확정). 상세 진입 시 SECURITY DEFINER RPC로 조회수 +1(소유자 전용 UPDATE RLS·service_role 금지 우회, Supabase 표준). 최근 등록은 `created_at` 정렬.
  - *과거 tech-debt ⚪에서 "홈이 매물탐색을 직접 품는 전체판은 보류, 미리보기판 유지"로 결정했으나, 이 증분은 **AI 히어로를 주인공으로 한 신규 랜딩**을 세우는 별건이다. 랜딩의 그리드는 탐색 화면 전체 대체가 아니라 **인기/최신 발췌 + '전체 보기' 진입**으로 가볍게 둔다(전체 필터·목록은 매물탐색 화면 유지, A2). 정확한 발췌 개수·배치는 bmad-ux.*
- **FR35** 랜딩에 **차종 빠른 진입**(경차·SUV·전기 등 카테고리 칩/그리드)을 제공해 검색어가 없는 사용자의 진입 장벽을 낮춘다.

### F10. UI 개편 · 반응형 (신규 / 일부 개정)

- **FR36** 사용자 웹은 **단일 코드 반응형**으로 데스크톱↔모바일을 대응한다(별도 m-dot 도메인 아님). 화면폭에 따라 **카드 그리드 → (모바일) 캐러셀/1열**, **필터 사이드바 → (모바일) 바텀시트/드릴다운**, **상단 내비 → (모바일) 햄버거**로 재배치한다. (원본 NFR3의 "반응형 웹" 구체화)
- **FR37** 전 화면에 **일관된 디자인 시스템**을 적용한다 — 페트롤+앰버 팔레트를 **CSS 토큰(Tailwind v4 `@theme`)**으로 중앙 관리(색 변경이 저비용이 되도록), 카드/뱃지/버튼/타이포 위계 통일.
- **FR38 (내비·IA — 도달성 요구, 배치는 UX)** 사용자는 다음 기능에 **명확한 경로로 도달**할 수 있어야 한다: 매물 탐색(사기) · 매물 등록(팔기) · AI 검색 · 내 채팅(메시지함) · 내 매물 관리 · 찜 목록.
  - **확정 규칙(기능 수준)**: ① **AI 검색을 핵심 진입점으로 노출**(차별점). ② **문의는 매물 상세의 "문의하기"로 개시** — 별도 전역 "문의" 메뉴를 두지 않는다(우리 데모엔 고객센터가 없고 "문의"="구매자↔판매자 채팅"이라 채팅과 중복. 조사: 6/6 서비스가 문의를 상단 GNB에 안 둠).
  - **UX 가이드(bmad-ux 입력 — PRD가 고정하지 않음):** 상단 메뉴는 **최소화**(경쟁 중고차 사이트 대비 메뉴 과다 지양) · **별도 "마이페이지" 전용 화면 신설은 지양**(내 매물·찜을 무거운 마이페이지로 만들기보다 기존 화면/드롭다운에 접기).
  - *내비 구조·메뉴 개수·화면 구성·모바일 탭바·반응형 배치의 **정확한 결정은 `bmad-ux`**(승인된 랜딩 목업 v1 + `reference-synthesis.md` + `increment-open-decisions.md`를 입력으로).*

### F11. Flutter 앱 UI 개편 (신규)

- **FR39** Flutter 앱 화면을 웹과 **동일한 디자인 언어**(색 토큰·카드 구성·신뢰속성/옵션 표시·플레이스홀더 규칙)로 개편한다. 이미지(F7)·카드 재설계(F8)를 앱에도 반영한다.

### F12. 실시간 문의 채팅 (변경 — FR20/21 개정)

### F13. AI 검색 RAG 실사용 수준 고도화 (변경 — FR13~FR18 대폭 확장)

> **배경·현황(코드 조사 확증):** 현재 라우팅은 A(SQL)/B(벡터)/C(거절) 3분기이고, **SQL+벡터 하이브리드 분기가 없으며**, 가이드 문서는 조회되지만 **title만 장식적으로 붙고 content(본문)가 답변 근거에 미반영**(진짜 문서 RAG 아님), 되묻기·청킹 없음, LangSmith 미연동. 이를 **상용 서비스 수준**으로 끌어올린다.
> **설계 확정(리서치 3종 기반 — 상세 how는 addendum):** ① 4분기는 기존 `add_conditional_edges` 키 확장으로 **작은 구조 변경** ② 되묻기는 `interrupt()`/체크포인터 없이 answer 반환 + 기존 멀티턴 이어붙이기(REST 단발요청에 적합) ③ 하이브리드는 SQL WHERE + `ORDER BY embedding <=>`를 **단일 쿼리**로(RRF 과설계) ④ 가이드 문서 **content 반영 + 거리 임계값 컷오프** ⑤ 청킹은 현 규모(12문서≈page-level) **불필요**, 문서 ≥20개/≥800토큰 시 헤딩 청킹 ⑥ **LangSmith 무료 트레이싱만 도입**(env 2개, 코드변경 0), 평가자동화·배포는 미도입.

- **FR43 라우팅 다분기 고도화** — 질문 의도를 **4갈래**로 라우팅한다: ① **무관/법적 질문**(날씨·주가·보험료 등) → **부드러운 거절** · ② **너무 광범위/애매한 질문** → **조건을 좁혀 되묻기**(예: "사회초년생 1인 무사고 작은 차" → *"2천만원 이하 준중형·소형, 유지비 낮은 가솔린, 1인소유·무사고로 추천드릴까요?"*) · ③ **구조형** → Text-to-SQL(FR14) · ④ **조합형** → SQL + 벡터 하이브리드(FR45).
- **FR44 가이드 문서 RAG 실제 활용** — 별도 **가이드 문서 코퍼스**를 필요 시 선택적으로 참조한다(현재는 매물 설명 임베딩만 사용 → 첫 의도 복원). "언제 가이드 문서를 참조할지"의 판단 분기를 둔다. (매물 설명 유사도 검색은 유지)
- **FR45 하이브리드 검색(SQL+벡터 조합)** — *"패밀리카로 쓸만한 4천 이하 SUV 중 통풍시트 있는 차"* 같은 **정형 조건 + 의미 조건** 혼합 질의에서, Text-to-SQL 필터와 임베딩 유사도를 **한 질의에서 조합**해 매물을 추천한다(현재 미구현).
  - **★ 필수 AC(보안, 블로커 — Amelia):** HYBRID가 생성하는 SQL도 **예외 없이 `sql_guard.validate_select_sql()`를 통과**하고 **`status='on_sale'`를 강제**해야 한다. ai_readonly 롤은 sold 포함 전 행을 보므로(`0006`), 이 가드를 우회하면 FR11(판매완료 비노출)·읽기전용 방어가 동시에 뚫린다(CM-B 위반). 벡터 연산자/`::vector` 캐스트가 가드 화이트리스트를 통과하도록 함께 정비.
- **FR46 조건 좁혀 되묻기(clarification)** — FR43-②의 대화 패턴. 광범위 질의(예: *"사회초년생, 유지비 적고 무사고 1인소유 작은 차 추천"*)에 곧바로 0건/엉뚱한 결과 대신 **제안형 되묻기**(예: *"2천만원 이하 준중형·소형, 유지비 낮은 가솔린, 1인소유·무사고로 추천드릴까요?"*)로 범위를 좁힌다.
  - *되묻기 제안은 **tappable 칩으로도 응답 가능**(자유 타이핑은 그대로 유지 — 칩은 선택지 추가일 뿐 입력을 막지 않음, **추가 LLM 비용 없음**: AI가 이미 만든 제안을 버튼으로 렌더). 정확한 칩 배치는 bmad-ux. (Sally 제안)*

- **FR48 가이드 문서 청킹·임베딩 (조건부)** — 현 규모(12문서 ≈ 문서당 1임베딩 = page-level)에서는 **청킹을 도입하지 않는다**(과설계 회피, 설계 확정 ⑤). 문서가 **≥20개 또는 문서당 ≥800토큰**으로 커지면 `##` 헤딩 섹션 단위 청킹 후 임베딩으로 확장한다. (즉 "지금 당장 청킹"이 아니라 "커지면 청킹"이 요구사항)
- **FR49 (기술부채 #16)** 유사도 **거리 컷오프** — 의미상 동떨어진 문서/매물이 근거로 첨부되지 않게 한다.
- **FR50 (기술부채 #5, 선행)** AI 검색 DB 접근에 **커넥션 풀 + 타임아웃 + async 논블로킹**을 도입한다(부하 대비, RAG 착수 전 선행).

- *위험·검증*: CLARIFY 오분류로 되묻기가 과다하면 이탈 위험 → CLARIFY를 "조건 전무한 극단 질의"에만 좁게 적용하고, 기존 A/B 질의셋(Phase B 하니스)으로 발동 빈도를 실측 후 배포.

**카운터 지표(망가지면 안 되는 것):**

- **CM-A** 기존 매물 100건·기존 기능이 이미지 도입으로 깨지지 않는다(nullable 마이그레이션, 임베딩 보존).

**측정 게이트 (품질 — party-mode John, 새 도구 없이 기존 하니스 재사용):**

- **G1** 되묻기(CLARIFY) 발동률 — 기존 A/B 질의셋(Phase B 하니스, ~44개)에서 CLARIFY가 **과다 발동하지 않는다**. 판정 기준: 이 질의셋은 대부분 조건이 명시된 질의라 CLARIFY 발동은 **소수(예: 10% 이하, 구현 시 베이스라인 실측으로 확정)**여야 하며, 명시조건 질의(A/B로 라우팅돼야 할 것)가 CLARIFY로 새면 실패. LangSmith 트레이싱으로 분기 분포 확인. (과다 되묻기=이탈 방지, FR46 위험 대응)
- **G2** RAG 회귀 게이트 — 4분기·하이브리드 도입 후 기존 Phase B 회귀 하니스 점수가 베이스라인 이하로 떨어지지 않는다(SM-F 보강).
- **G3** 이미지 하위호환 — 사진 없는 기존 100건이 placeholder로 렌더되고 목록/상세/AI카드에서 0건 에러(CM-A 실측).
