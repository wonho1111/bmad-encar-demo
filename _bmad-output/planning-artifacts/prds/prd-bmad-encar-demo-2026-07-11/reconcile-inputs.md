# PRD 입력 정합성 리뷰 — 2026-07-11 증분 PRD vs 소스 입력

- **검토 대상**: `prd.md` + `addendum.md` (증분 PRD, 2026-07-11)
- **대조 입력**: `reference-synthesis.md` · `increment-open-decisions.md` · `research-data-trust-attributes.md` · `research-data-options.md` · `research-account-nav-model.md` · `research-current-rag-implementation.md` · `research-langx-rag-patterns.md` · `research-supabase-viewcount-rpc.md` · `project-context.md` · `docs/conventions.md` · `docs/tech-debt.md` · 베이스라인 PRD(`prd-bmad-encar-demo-2026-06-17/prd.md`, FR1~25)
- **목적**: FR 구조가 번호로 깔끔하게 담기 어려운 정성적 의도(톤·우선순위·경고)나 리서치 결론이 조용히 드러났는지 점검. 새 기능을 제안하지 않는다(A2 단순함 우선 — 과설계 지적도 유효한 발견).

## 1. `reference-synthesis.md` 대비

- **[MEDIUM] 가격 필터 촘촘함/자유입력 인사이트 완전 누락** — §5 "KB차차차 반면교사: 가격 구간이 1500만 이후 500만 단위로만 나뉘어 실사용자 불만 → 우리 가격 필터는 촘촘히/자유입력"이라는 구체적 결론이 있으나, PRD FR26~58 어디에도 필터 그래뉼래러티에 대한 언급이 없다. FR36(반응형: 필터 사이드바→바텀시트)이 필터를 건드리는 유일한 FR인데 표시 방식만 다루고 "구간 촘촘함"은 다루지 않는다. 이번 증분이 필터 자체를 스코프로 안 잡았다면 명시적으로 "필터 개선은 범위 밖(기존 FR9 유지)"이라고 §6 범위에 밝히는 게 좋다 — 지금은 조용히 빠져 있어 리서치를 읽은 사람만 이 결론의 존재를 안다.
- **[LOW] "AI 검색을 전체 매물이 아닌 신뢰 데이터 하위셋부터 단계 적용"이라는 엔카 교훈 미반영** — §5 "엔카 교훈 채택: AI를 전체 매물에 무리하게 넓히기보다 신뢰 데이터 하위셋부터 단계 적용"이 있으나, F13(FR43~51)은 신뢰속성 유무와 무관하게 전체 `on_sale` 매물을 그대로 대상으로 한다. 데모 규모(100건)에서는 무의미할 수 있어 실제로는 드롭해도 무방해 보이지만, PRD/addendum에 "이 교훈은 데모 규모상 적용 불필요"라는 판단 한 줄이 없어 리서치를 반영했는지 그냥 놓쳤는지 구분이 안 된다.
- **[LOW] 매물 상세 페이지 정보 위계(신뢰요소=가격 바로 아래, 판매자 연락처=하단, 긴 표 대신 섹션 카드)** — §5에 상세페이지 레이아웃 결론이 있으나 PRD는 FR30(카드 배치)만 다루고 **상세 페이지**의 배치 규칙은 언급이 없다("카드"와 "상세"를 명확히 구분하지 않음). `bmad-ux` 위임이라면 명시가 필요.
- 확인됨(문제 없음): 이미지 5:3·N장 배지·fallback·카드 위계·랜딩 그리드·반응형 인터랙션 전환·AI 검색 랜딩 배치는 전부 FR에 정확히 반영됨.

## 8. `research-supabase-viewcount-rpc.md` 대비

- **[MEDIUM] 보안 체크리스트 4번째 항목("`REVOKE EXECUTE ... FROM PUBLIC` 후 필요한 역할에만 재부여, 신규 함수는 기본적으로 PUBLIC에 열림")이 PRD/addendum 요약에서 누락** — OI-5·addendum 둘 다 "보안 필수 3가지"로 ①search_path='' ②범위 최소화 ③GRANT EXECUTE to anon/authenticated만 나열한다. 리서치 원문은 이 세 가지 외에 "PostgreSQL 신규 함수는 기본적으로 PUBLIC에 EXECUTE가 열려 있으므로 REVOKE부터 해야 안전"이라는 4번째 항목을 명시했는데 PRD가 3가지로 축약하면서 이 항목이 빠졌다. 구현 시 이 REVOKE 단계를 놓치면 anon/authenticated 외 역할(있다면)도 실행 가능해질 여지가 있다.
- 확인됨(문제 없음): rate-limit/어뷰징 방지는 "데모는 무시, 실서비스는 추가 검토"로 정확히 반영.

## 9. `project-context.md` / `docs/conventions.md` 대비

- **[HIGH] 베이스라인 FR10·FR17(ListingCard 계약)이 이미지 도입으로 실질적으로 바뀌는데 "(개정)" 표시가 없음** — `conventions.md §4`는 "ListingCard 필드(사진 없음 — 썸네일 필드 없음)"이라고 명시하고, addendum은 "계약 변경: `conventions.md §4 ListingCard`에 사진 필드 추가"라고 적어 두었다. 그런데 PRD 본문의 "변경되는 기존 FR" 표(§1)에는 FR5/FR15/FR20/FR21만 나열되고, **FR10(매물 상세 조회 — 옵션·상태 포함, 이제 사진 갤러리·신뢰속성 뱃지 포함)과 FR17(AI 응답 ListingCard 최소 필드 — 제조사·모델·연식·가격·주행거리·지역)이 이미지 필드를 포함할지 여부가 어느 FR에도 명문화되지 않았다.** 특히 FR17은 AI 검색 결과 카드 스키마라서, F7(이미지)이 매물 목록/상세 카드에는 적용되지만 **AI 답변 카드에도 사진이 실리는지**가 FR26~29 어디에도 확정돼 있지 않다 — 아키텍처 단계에서 조용히 빠질 위험이 큰 대표적 "갭".
- **[MEDIUM] 이미지 스토리지 도입이 project-context 규칙6("service_role 키 금지, anon key만 사용")과 정합한지 명시 확인이 없음** — NFR7은 "[ASSUMPTION] 스토리지 = Supabase Storage"라고만 적고, 업로드 경로가 anon key + Storage RLS로 가능한지(즉 규칙6을 지키면서 판매자가 사진을 올릴 수 있는지)는 OI-3(아키텍처 소관)로 미뤄졌다. viewcount RPC 항목은 규칙6 준수를 addendum에서 재확인(`service_role` 금지 명시)했지만, 이미지 업로드 쪽은 그 재확인이 없어 대칭이 깨짐.
- **[LOW] project-context 규칙8("채팅 현재 방식=4초 증분 폴링")을 F12 적용 후 갱신해야 한다는 사실이 PRD 본문에 없음** — addendum이 "Last Updated" 갱신 원칙을 "사람용" 절에서 일반론으로만 언급("채팅 방식이 바뀌면 이 파일을 먼저 고친다"). 실행 체크리스트로 남기지 않으면 구현 후 문서 최신화가 누락되기 쉬움.

## 10. `docs/tech-debt.md` 대비

- **[MEDIUM] 🟡 #11(옵션 `text[]` 쉼표 라운드트립 손실)이 F8에서 옵션 스키마·표시를 대폭 건드리는데도 PRD/addendum에 언급이 없음** — F8(FR30~32)은 옵션을 희소도별로 재구성하고 신뢰속성과 분리하는 등 옵션 데이터 구조 자체를 크게 건드린다. 그런데 기존 부채 #11("SellForm이 options를 쉼표로 join/split, 값에 쉼표 있으면 손실")이 F8과 정확히 같은 코드 표면(`SellForm.tsx`, options 데이터)을 다루는데도 이번 증분이 이 부채를 흡수하는지 명시적 판단이 없다. FR41(#6 흡수)·FR42(#7 흡수)·FR49(#16 흡수)·FR50(#5 흡수)처럼 "(기술부채 #N 흡수)" 표기가 있는 다른 항목과 비교하면, #11만 조용히 스코프 밖에 남아 있어 의도적 배제인지 누락인지 구분이 안 된다.
- **[MEDIUM] ⚪ "홈이 탐색을 직접 품기 — 전체판(보류)" 결정과 신규 FR34(랜딩 그리드)의 관계가 불명확** — tech-debt.md는 "현재는 미리보기판(최근 N건 읽기전용 + 더보기→/search). 홈이 필터·URL을 직접 소유하는 전체판은 이후 판단"이라고 명시적으로 못박아 두었다. 그런데 F9(FR33~35)는 "히어로 아래에 매물 그리드를 바로 노출 — 인기/최신 2단"이라고 랜딩을 사실상 매물 브라우징의 주 화면으로 재설계하면서, 이게 기존에 Cut한 "전체판"을 다시 채택하는 것인지, 여전히 "미리보기판"(더보기 클릭 시 별도 /search로) 수준을 유지하는 것인지 PRD 어디에도 명시가 없다. 랜딩 개편이라는 이번 증분의 핵심 축이 과거 명시적 보류 결정을 뒤집는지 여부는 "(개정)" 표시급으로 다뤄져야 할 사안인데 조용히 넘어감.
- 확인됨(문제 없음): #5(DB풀)→FR50, #6(멱등키)→FR41, #7(폴링무알림)→FR42, #13/#14(AI 회귀테스트)→addendum "party-mode 리뷰" 절에서 명시적으로 연계, #16(거리컷오프)→FR49 전부 정확히 추적됨.
