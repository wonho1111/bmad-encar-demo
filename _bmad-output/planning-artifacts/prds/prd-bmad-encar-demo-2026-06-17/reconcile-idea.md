## 2. 발견된 갭·모순

### A. (중대) 문서기반 RAG 대상 데이터에서 "① 매물 데이터"가 누락됨
- **원본(idea.md L20):** RAG 대상 데이터 = **① 매물 데이터(차량 설명·옵션·상태)**, ② 차량 상식/구매 가이드/FAQ 문서. → 즉 매물의 **서술형 텍스트(설명·옵션·상태)** 자체도 임베딩 의미검색 대상으로 명시.
- **PRD(FR15·섹션4-B):** 문서기반 RAG의 코퍼스를 **"차량 상식·구매 가이드 문서"로만** 한정. 매물의 서술형 텍스트를 의미검색 대상으로 두는 흐름이 사라짐.
- **addendum(L32):** "① 매물 데이터 — ... (구조 필드는 Text-to-SQL 대상)"이라고만 적어, 매물 데이터를 사실상 구조검색 전용으로 재배치. 원본은 매물의 비정형 설명도 의미검색 근거로 보던 뉘앙스가 있었음.
- **영향:** "패밀리카로 무난한"류 질의에서 가이드 문서로 차종을 좁힌 뒤 **매물 설명 텍스트의 의미 유사도**까지 활용하려던 여지가 줄어듦. 데모 핵심(2경로 RAG)의 표현력에 영향 가능.
- **권고:** FR15 또는 addendum에 "매물 설명/옵션 텍스트도 의미검색 대상에 포함할 수 있음"을 명시하거나, 의도적으로 제외했다면 그 결정 사유를 1줄 기록.

### B. (모순) 모바일 테스트 도구 — Expo Go vs Flutter 디바이스/에뮬레이터
- **원본(idea.md L36):** "모바일 테스트: **Expo Go**".
- **addendum(L9):** "Flutter는 **Expo Go 대신** 디바이스/에뮬레이터로 확인".
- **분석:** Expo Go는 React Native 전용 도구로 Flutter에는 사용 불가 → 원본 기재가 기술적으로 부정확했고, addendum이 이를 올바르게 정정한 것. **실질은 개선이지 누락이 아님.** 다만 "원본과 다르게 결정"한 항목이므로 추적 차원에서 기록. 별도 조치 불요(addendum 판단이 타당).

### C. (경미) "정확한 모델명을 모르는 초보 구매자" 페르소나 의도 — 부분 반영
- **원본 의도:** 자연어 검색의 존재 이유는 "필터가 너무 많아 막막한 초보가 모델명 없이도 차를 찾게 함"(idea.md의 자연어 검색 동기).
- **PRD:** UJ-1 주인공 지훈 설정에 "정확한 모델명은 모른다 / 필터가 너무 많아 막막하다"로 **톤·의도가 잘 보존됨**. → 갭 아님(확인 완료, 양호 사례로 기록).

### D. (경미) 안전장치 표현 폭 — "화이트리스트 컬럼"
- **원본(idea.md L21):** "SELECT 전용 + 판매완료 제외 등 쿼리 범위 제한".
- **PRD(FR14·NFR2·CM2):** "SELECT 전용·쿼리 범위 제한"으로 반영. **addendum(L23)**에서 "화이트리스트 컬럼"으로 구체화 → 오히려 보강됨. 갭 아님.

### E. (경미) 라우터 분류 항목 C의 "차량 상식/안내" 처리 뉘앙스 변화
- **원본(idea.md L16):** 라우터는 질문을 "*매물 조건 검색* vs *차량 상식/안내*"로 2분. 차량 상식/안내는 문서기반 RAG로 **답변**하려던 뉘앙스(상식형도 안내).
- **PRD(FR13/FR16):** A(구조)/B(질적·의미형 **매물검색**)/C(무관→거절)로 3분. **순수 상식 Q&A는 명시적으로 제외**(의도적 스코프 결정과 일치). 즉 원본의 "상식/안내 응답" 일부가 "매물 추천에 종속된 의미검색"으로 좁혀짐.
- **상태:** 사용자가 명시한 의도적 스코프 결정("순수 차량상식 Q&A 제외, 문서기반 RAG는 정성적 매물검색용으로 유지")과 **정확히 부합**. → 갭 아님(스코프 결정으로 처리됨을 재확인).

### F. (정보) "결과 없을 때 조건 완화 제안"은 PRD가 원본보다 추가한 항목
