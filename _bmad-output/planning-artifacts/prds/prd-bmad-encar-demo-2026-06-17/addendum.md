# Addendum — bmad-encar-demo PRD

PRD 본문에 담기엔 "기술적 어떻게(technical-how)"에 가까운 내용을 모은 부록. 다운스트림(아키텍처·솔루션 설계) 입력으로 사용한다.

## 기술 스택 (idea.md 기준)

- **사용자용 반응형 웹:** Next.js
- **관리자용 반응형 웹:** Next.js
- **사용자용 모바일 앱:** Flutter (테스트: Expo Go 대신 Flutter는 디바이스/에뮬레이터로 확인)
- **통합 API:** FastAPI
- **AI 오케스트레이션:** LangGraph (Python) — FastAPI 백엔드 내 서비스로 탑재
- **LLM:** Google Gemini API 무료 등급(Gemini Flash 계열) — 라우팅/Text-to-SQL/답변 생성
- **임베딩:** Gemini 임베딩(생성 LLM과 동일 SDK·API 키로 통일)
- **벡터 검색:** Supabase pgvector
- **DB:** Supabase(PostgreSQL)
- **배포:** Vercel로 통일(Next.js 프론트 + FastAPI 백엔드). 차선: 부하 증가 시 백엔드만 Cloud Run / Railway로 분리.

## AI 검색 기술 메커니즘 (LangGraph 노드 흐름)

PRD 섹션 4의 동작 흐름을 LangGraph 그래프로 구현:

1. **라우터 노드** — 질의 의도를 A(구조형)/B(질적·의미형)/C(무관)로 분류. 가드(C)는 라우터 프롬프트에 분류 항목으로 포함(별도 노드 불필요).
2. **경로 A — SQL 기반 RAG 노드** — Text-to-SQL로 자연어를 SELECT 쿼리로 변환, 매물 테이블 필터링. 쿼리 범위 제한(SELECT 전용, 판매완료 제외, 화이트리스트 컬럼).
3. **경로 B — 문서 기반 RAG 노드** — ① 매물 설명·옵션 텍스트 + ② 차량 상식·구매 가이드 문서 임베딩을 pgvector 유사도 검색 → 적합 차종/특성 근거 추출 → 해당 조건으로 매물 조회.
4. **답변 생성 노드** — 검색 결과를 근거로 자연어 답변 + 매물 카드 구성.

- **멀티턴:** 서버 무상태(stateless). 대화 맥락은 클라이언트 보관, 요청 시 직전 턴 동봉. AI 대화 전용 테이블 없음.
- **provider-agnostic:** LangGraph는 모델 비종속 → 추후 다른 LLM으로 교체 가능.

## RAG 대상 데이터

- **① 매물 데이터** — 구조 필드(가격·연식·차종·색상·연료·변속기·지역 등)는 **Text-to-SQL**(경로 A) 대상. 설명·옵션 **텍스트**는 **임베딩**(경로 B) 대상으로도 활용.
- **② 차량 상식·구매 가이드 문서** — 문서 기반 RAG(경로 B)의 코퍼스. **필수 산출물**이며 실제 문서 내용은 구현 단계에서 작성한다. (예: 차종별 특성, 패밀리카/초보자 적합 차종, 유지비·신뢰성 가이드)
- **임베딩 차원 제약:** 임베딩 모델 출력 차원 = pgvector `vector(N)` 컬럼 차원이 **반드시 일치**해야 한다(불일치 시 미동작). 모델·차원은 OI1에서 확정.

## Vercel 배포 유의사항 (research 반영)

- 무상태이므로 채팅 이력은 Supabase에 저장(폴링과 잘 맞음).
- Supabase 연결 풀링 설정 필요.
- 무거운 라이브러리 미적재(번들 500MB 한도, 함수 시간 최대 300초 내).
- 근거: 경부하 시연용, AI 검색은 요청당 짧은 Gemini 호출 1~2회로 종료.

## 그라운딩 메모 — 매물 등록 필드 선정

엔카·KCar·헤이딜러 웹리서치 결과 확장 셋(≈16개) 채택. 데모 복잡도로 제외한 항목: 차대번호·69항목 검수기록부·압류/저당·명의이전·전주이력.
