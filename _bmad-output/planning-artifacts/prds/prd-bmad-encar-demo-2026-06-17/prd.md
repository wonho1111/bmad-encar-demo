# 중고차 직거래 서비스 (bmad-encar-demo) — PRD

## 1. 개요

엔카·헤이딜러류 **중고차 직거래 서비스**의 핵심 기능을 최소로 구현한 데모/과제용 제품. 구매자·판매자를 직접 연결하고, **AI 자연어 매물 검색**(SQL 기반 RAG + 문서 기반 RAG)을 핵심 차별점으로 시연한다. 결제·계약은 서비스 범위 밖(오프라인)이며, 직거래 연결(문의 채팅)까지가 범위다.

- **성격:** 컨설팅 과정 과제 제출 + 간단한 시연용 (경부하).
- **플랫폼:** 사용자용 반응형 웹(Next.js) + 관리자용 반응형 웹 + 사용자용 모바일 앱(Flutter).
- **핵심 차별점:** 자연어 한 문장으로 매물을 찾는 AI 채팅 어시스턴트.

**용어 정의 (문서 전체 일관 사용):**

- **매물** — 판매자가 등록한 한 건의 차량 판매글. (= "차량"과 혼용하지 않고 "매물"로 통일)
- **구매 완료** — 판매자가 누르는 **액션**. 이 액션의 결과로 매물은 **판매 완료** **상태**가 된다.
- **문의 채팅 / 채팅방** — 구매자↔판매자 간 DB 저장 대화. AI 채팅(어시스턴트)과는 별개.
- **AI 채팅(어시스턴트)** — 구매자의 자연어 매물 검색 대화. 서버에 저장하지 않음.

## 2. 사용자 여정 (User Journeys)

### UJ-1. 구매자 — AI로 차를 찾는다 (핵심 여정)

**주인공:** 지훈(32세 직장인). 첫 차를 사려는 중고차 초보. 예산 약 3천만 원, 가족용 넓은 SUV를 원하지만 정확한 모델명은 모른다. 기존 서비스는 필터가 너무 많아 막막하다.

1. 지훈이 서비스에 **가입·로그인**한다 (구매자 역할).
2. 검색 필터 대신 **AI 채팅 어시스턴트**에 *"3천만원 이하 흰색 SUV 추천해줘"* 라고 자연어로 입력한다.
3. AI가 **구조 조건(Text-to-SQL)**으로 매물 DB를 필터링해 **매물 카드 목록 + 근거 설명**을 답한다. 지훈이 *"그 중 더 싼 거"* 라고 **멀티턴 후속질문**을 한다.
4. 지훈이 *"패밀리카로 무난한 거 있어?"* 처럼 **질적/의미형 질문**을 한다 → AI가 **문서 기반 RAG**로 차량 상식·가이드 문서에서 적합 차종 근거를 찾아 매물을 추천한다.
5. (가드) 지훈이 매물과 무관한 질문을 하면 AI는 **정중히 거절하고 매물 검색으로 유도**한다.
6. 마음에 드는 매물의 **상세 페이지**(설명·옵션·상태)를 확인한다.
7. 판매자에게 **문의 채팅**을 보낸다 → 채팅방이 생성되고 **폴링 방식**으로 대화한다.
8. (여정의 끝) 거래 조건 합의 → 오프라인 거래. 결제·계약은 서비스 밖.

### UJ-2. 판매자 — 차를 직접 판다

**주인공:** 민서(38세). 타던 SUV를 딜러 헐값 대신 직접 팔고 싶다.

1. 민서가 **가입·로그인**한다 (판매자 역할 — 가입 시 역할 선택, 계정은 해당 역할 고정).
2. **차량을 등록**한다(제조사·모델·차종·연식·가격·주행거리·연료·변속기·지역·사고이력 등 + 설명·옵션 — 전체 필드는 FR5). **등록 즉시 노출**(관리자 사전 승인 없음).
3. **본인 매물만** 조회/수정/삭제하며 관리한다.
4. 구매자(지훈)의 **문의 채팅에 응대**한다 (폴링).
5. 거래 성사 시 **"구매 완료"를 누른다**(판매자만, 상대 지정 불필요) → 매물이 **판매 완료**로 바뀌어 구매자에게 더 이상 노출되지 않는다.

### UJ-3. 관리자 — 서비스 건전성을 관리한다

**주인공:** 운영자. 이상 회원·부적절 매물·문제 채팅을 점검한다.

1. **회원 관리** — 회원 목록 조회 + 정지/삭제.
2. **전체 매물 관리** — 모든 매물 조회 + 부적절 매물 삭제.
3. **거래 내역 관리** — 판매 완료된 매물 목록 조회.
4. **채팅 관리** — 채팅방 열람 + 삭제.

## 3. 기능 요구사항 (Functional Requirements)

> FR(Functional Requirement): 시스템이 "무엇을" 해야 하는지 정의한 항목. 전역 번호(FR1~)로 고정 관리한다.

### F1. 인증·계정

- **FR1** 사용자는 이메일/비밀번호로 가입하며, 가입 시 역할(구매자/판매자)을 선택한다. (Supabase Auth 사용, 소셜 로그인은 범위 밖)
- **FR2** 사용자는 로그인/로그아웃할 수 있다.
- **FR3** 역할별 접근 제어 — 역할(구매자/판매자/관리자)에 따라 사용할 수 있는 기능이 다르다.
- **FR4** 관리자 계정이 별도로 존재한다(시드 또는 별도 경로로 생성).

### F2. 매물 등록·관리 (판매자)

- **FR5** 판매자는 차량 매물을 등록한다. 등록 필드(15개):
  제조사 · 모델 · 차종 · 연식 · 가격 · 주행거리 · 색상 · 연료 · 변속기 · 배기량 · 인승 · 지역 · 사고이력(무사고 여부) · 옵션 · 설명.
  (차종은 Encar·K-Car 등 국내 중고차 앱 분류를 따른다: 경차·소형차·준중형차·중형차·대형차·스포츠카·SUV·RV·경승합차·승합차·화물차·기타)
  - **사진 미포함(범위 결정):** 본 과제의 최종 목표는 RAG 경험이며, 사진은 RAG 코퍼스(설명·옵션 텍스트 + 가이드 문서)에 포함되지 않는다. 구현·시연 단순화를 위해 사진 등록 개념을 서비스 전체(웹·앱·DB)에서 제외한다.
- **FR6** 판매자는 **본인 매물만** 조회/수정/삭제할 수 있다.
- **FR7** 등록한 매물은 즉시 노출된다(관리자 사전 승인 없음).
- **FR8** 판매자는 본인 매물을 **"구매 완료" 처리**하여 판매 완료 상태로 전환한다. (판매자만 가능, 상대 구매자 지정 불필요)

### F3. 매물 탐색 (구매자)

- **FR9** 구매자는 키워드/필터(가격·연식·차종·색상·연료·변속기·지역 등)로 매물을 검색한다.
- **FR10** 구매자는 매물 상세를 조회한다(설명·옵션·상태 포함).
- **FR11** **판매 완료 매물은 구매자의 모든 화면에서 노출되지 않는다** — 필터 검색·AI 검색·목록·상세 공통으로 적용되는 단일 규칙. (다른 FR은 이 규칙을 따른다)

### F4. AI 검색 어시스턴트 (핵심)

- **FR12** 구매자는 AI 채팅 어시스턴트에 자연어 한 문장으로 매물을 검색한다.
- **FR13** 라우터가 질문 의도를 분류한다: **(A) 구조형 매물 검색 / (B) 질적·의미형 매물 검색 / (C) 매물과 무관한 질문**.
- **FR14** (A) 구조형 → **Text-to-SQL**로 매물 DB를 정밀 필터링한다. SELECT 전용·쿼리 범위 제한(안전장치)을 적용하며, 판매완료 미노출은 FR11을 따른다.
- **FR15** (B) 질적·의미형 → **문서 기반 RAG**로 매물을 추천한다. 임베딩 유사도 검색(pgvector) 대상 코퍼스는 **① 매물 설명·옵션 텍스트 + ② 차량 상식·구매 가이드 문서** 둘 다이다(idea.md 원안). 적합 차종/특성 근거를 찾아 매물을 추천한다.
- **FR16** (C) 매물과 무관한 질문 → 정중히 거절하고 매물 검색으로 유도한다. (순수 상식 Q&A는 제공하지 않음) 경계가 모호한 질의(예: 보험료·할부 등 거래 주변 정보)는 매물 검색으로 연결 가능하면 검색하고, 아니면 거절한다.
- **FR17** 답변은 **자연어 설명 + 매물 카드** 형태로 반환한다. 매물 카드 최소 표시 필드: 제조사·모델·연식·가격·주행거리·지역. **검색 결과가 0건이면** 빈 목록 대신 **조건 완화를 제안하는 안내**를 반환한다.
- **FR18** **멀티턴** — 직전 대화 맥락은 클라이언트(브라우저/앱)가 보관해 후속 질문을 지원한다. 서버·DB에는 AI 대화 이력을 저장하지 않는다(새로고침/세션 종료 시 초기화).

### F5. 문의 채팅 (구매자 ↔ 판매자)

- **FR19** 구매자가 매물에 문의를 보내면 구매자·판매자 양쪽에 동일한 채팅방이 생성된다.
- **FR20** 양측이 **폴링 방식**으로 메시지를 주고받는다.
- **FR21** 채팅 메시지는 DB에 저장된다(AI 대화 이력과 별개).

### F6. 관리자

- **FR22** 회원 목록 조회 + 정지/삭제.
- **FR23** 전체 매물 조회 + (부적절 매물) 삭제.
- **FR24** 거래 내역(판매 완료된 매물 목록) 조회.
- **FR25** 채팅방 열람 + 삭제.

## 4. AI 검색 동작 흐름

핵심 차별점인 AI 검색이 한 번의 질의에서 어떻게 동작하는지(능력 수준)를 정의한다. 구체적 구현 메커니즘(프레임워크·모델·벡터DB)은 `addendum.md` 참조.

1. **입력** — 구매자가 자연어 질의를 보낸다. 멀티턴 후속 질문이면 클라이언트가 보관한 직전 대화 맥락을 함께 전송한다(FR18).
2. **라우터 분류**(FR13) — 질의 의도를 (A) 구조형 / (B) 질적·의미형 / (C) 매물 무관 으로 분류한다.
3. **검색 실행**
   - **(A) 구조형**(예: *"3천만원 이하 흰색 SUV"*) → Text-to-SQL로 매물 DB를 가격·연식·차종·색상·연료·변속기·지역 등 구조 조건으로 필터링한다(FR14).
   - **(B) 질적·의미형**(예: *"패밀리카로 무난한 거", "사회초년생이 사기 좋은"*) → 매물 설명·옵션 텍스트 + 차량 상식·가이드 문서를 임베딩 유사도 검색해 적합 차종/특성 근거를 얻고, 그 조건으로 매물을 조회한다(FR15).
   - **(C) 매물 무관**(예: *"오늘 날씨 어때?"*) → 정중히 거절하고 매물 검색으로 유도한다(FR16).
4. **답변 생성**(FR17) — 검색 결과를 근거로 **자연어 설명 + 매물 카드**를 반환한다. 결과가 없으면 조건 완화를 제안하는 안내를 반환한다.
5. **멀티턴 갱신**(FR18) — 응답 후 대화 맥락은 클라이언트에 누적되고, 다음 질의 시 직전 턴들과 함께 전송된다. 서버·DB에는 저장하지 않는다.

**안전장치(공통):** Text-to-SQL은 **SELECT 전용 + 쿼리 범위 제한**, 판매 완료 매물 제외(FR11). 범위 외 질의는 라우터 단계에서 차단(FR16)하여 오·남용을 방지한다.

## 5. 비기능 요구사항 (Non-Functional Requirements)

데모/경부하 시연 기준으로 핵심만 정의한다. (출시용 SLA·고가용성·대규모 운영 요건은 범위 밖)

- **NFR1 성능** — AI 검색 응답은 통상 수 초 내(요청당 Gemini 호출 1~2회). 문의 채팅은 폴링 방식으로 준실시간 갱신(폴링 주기는 구현 시 결정, 예: 3~5초).
- **NFR2 보안·권한** — Text-to-SQL은 SELECT 전용·쿼리 범위 제한. 프롬프트 지시에만 의존하지 않고 **결정론적 안전장치**를 둔다: ① AI 검색 경로는 **읽기 전용 DB 롤**로 실행, ② 생성 쿼리에 **기본 LIMIT 강제**, ③ 실행 전 **쿼리 검증**(SELECT만 허용·허용 컬럼/테이블 화이트리스트). 본인 매물만 수정/삭제(FR6), 역할별 접근 제어(FR3). DB 접근은 행 수준 보안(RLS, Row Level Security — 행 단위로 접근을 제한하는 PostgreSQL 기능) 적용 권장.
- **NFR3 플랫폼** — 반응형 웹(사용자용·관리자용 각각) + 모바일 앱(Flutter)에서 핵심 기능이 동작한다.
- **NFR4 배포·확장** — Vercel 단일 배포(프론트+백엔드). 무상태 전제이므로 상태는 Supabase에 저장. 부하 증가 시 백엔드 분리(Cloud Run/Railway)로 확장 가능.
- **NFR5 데이터·프라이버시** — 문의 채팅 메시지는 DB에 저장(영속). AI 대화 이력은 저장하지 않음(클라이언트 보관, 세션 종료 시 소멸).
- **NFR6 비용** — 과제용 저비용 구성(Gemini 무료 등급, Vercel Hobby, Supabase 무료/저가 티어 범위 내).

## 6. 성공 지표 (Success Metrics)

과제/시연 목적에 맞춰 "시연이 끝까지 매끄럽게 동작하는가"를 중심으로 정의한다.

- **SM1** 핵심 시연 시나리오 3종이 처음부터 끝까지 동작한다: ① 구매자 AI 검색→상세→문의 채팅, ② 판매자 등록→문의 응대→구매 완료, ③ 관리자 조회·삭제.
- **SM2** 기본 검색이 동작한다 — 구매자가 키워드/필터(가격·연식·차종·색상·연료·변속기·지역 등)로 원하는 매물을 찾을 수 있다(FR9).
- **SM3** AI 검색이 **두 경로(SQL 기반·문서 기반 RAG)를 모두 시연**할 수 있다 — 과제 핵심 목표.
- **SM4** 대표 시연 질의(구조형·질적형)에 대해 적절한 매물 카드가 반환된다.

**카운터 지표(Counter-metrics — 위 지표를 좇다가 망가지면 안 되는 것):**

- **CM1** 가드 동작 — 매물과 무관한 질의에 엉뚱한 매물/답변을 내지 않는다(FR16).
- **CM2** 안전 범위 — Text-to-SQL이 SELECT 전용·범위 제한을 벗어나지 않는다.
- **CM3** 데이터 정합 — 판매 완료 매물이 구매자에게 노출되지 않는다(FR11).

## 7. 범위 (Scope)

**범위 내 (In Scope):**

- 기능 요구사항 FR1~FR25 전체.
- 3개 플랫폼(사용자 웹·관리자 웹·Flutter 앱)에서의 핵심 기능.
- AI 자연어 매물 검색(SQL 기반 + 문서 기반 RAG 2경로) + 라우터 가드.

**범위 밖 (Out of Scope):**

- 결제·계약(오프라인에서 처리). 앱 내 결제 없음.
- 소셜 로그인, 사용자 신고 기능.
- 실시간 웹소켓 채팅(폴링으로 대체), AI 대화 이력 서버 저장.
- 관리자 사전 승인 워크플로, 대규모 운영·모니터링·고가용성.
- 순수 차량 상식 Q&A(매물 추천 없는 안내문).
