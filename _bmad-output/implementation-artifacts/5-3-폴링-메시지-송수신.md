# Story 5.3: 폴링 메시지 송수신 (FR20, FR21)

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 채팅 당사자(구매자 또는 판매자),
I want 채팅방에서 메시지를 입력해 보내고 상대가 보낸 메시지를 폴링으로 받아보고 싶다,
so that 매물 거래 조건을 실시간에 가깝게 협의한다.

## Acceptance Criteria

1. **Given** 채팅방(`/chat/[roomId]`)에 진입한 당사자가, **When** 입력창에 메시지를 적어 전송하면, **Then** 그 메시지가 `chat_messages`에 INSERT되어 **영속 저장**되고(새로고침해도 남는다 — FR21·NFR5), 내 화면 메시지 목록에 즉시 나타난다(낙관적 반영 또는 폴링 반영). `sender_id`는 RLS상 본인(`auth.uid()`)으로 강제된다. (epics.md L623~625, 0003 `chat_messages_insert_participant`)
2. **Given** 폴링(3~5초 주기)이 동작할 때, **When** 상대가 같은 방에 메시지를 보내면, **Then** 내 화면에 폴링 주기 안에 그 메시지가 갱신되어 나타난다(메시지 insert → 상대 폴링 1건 수신 — NFR1 준실시간). (epics.md L627~629)
3. **Given** 메시지 목록이, **When** 렌더되면, **Then** `created_at` **시간순(오름차순, 오래된→최신)**으로 정렬되고, **내 메시지와 상대 메시지가 시각적으로 구분**되어 표시된다(예: 내 메시지 오른쪽 정렬·강조, 상대 왼쪽 정렬). 첫 대화(메시지 0건)이면 빈 상태 안내가 보인다.
4. **Given** 빈 본문(공백만)으로, **When** 전송을 시도하면, **Then** 클라이언트가 1차로 막고(전송 안 됨), 만약 통과해도 DB `CHECK(length(btrim(body)) > 0)`가 거부한다. 전송 실패 시 한국어 안내(원본 에러는 콘솔만).
5. **Given** 컴포넌트가 unmount되면(다른 방·페이지로 이동/탭 닫힘), **When** 폴링 인터벌이 살아 있다면, **Then** 인터벌을 정리한다(메모리 누수·유령 요청 방지). 전송·폴링이 겹쳐도 중복 표시가 없어야 한다(증분 조회 또는 id 기준 dedupe).
6. **Given** 제3자(비당사자)가 그 방 메시지를 조회/전송하려 하면, **When** RLS(`chat_messages_select_participant`·`chat_messages_insert_participant`)가 평가하면, **Then** 조회는 0건, 전송은 거부된다(앱은 진입 자체가 5-2 RLS로 이미 차단됨 — 이 스토리는 그 보호를 깨지 않는다).
7. **Given** 5-2가 만든 채팅방 진입 화면의 기존 요소(매물 요약 헤더·상대 표기·"채팅방 목록으로" 링크·sold 매물 빈 상태 플레이스홀더[옵션D])가, **When** 메시지 송수신 UI가 추가되어도, **Then** 그대로 유지된다(회귀 금지). 5-2의 "메시지 빈 골격"(안내 박스)은 실제 메시지 목록+입력창으로 **대체**된다.

### 핵심: chat_messages 스키마·RLS·인덱스(신뢰하고 활용 — 5-1에서 적용 완료)

- 테이블: `chat_messages(id uuid pk, room_id uuid→chat_rooms cascade, sender_id uuid→profiles cascade, body text, created_at timestamptz default now())`.
- **CHECK** `chat_messages_body_not_blank`: `length(btrim(body)) > 0` — 공백만 메시지 거부(빈값 전송 시 `23514`). not null은 `''`를 못 막으므로 이 CHECK가 최종 방어. 클라가 1차로 `trim()` 차단.
- **인덱스** `chat_messages_room_created_idx (room_id, created_at)` — **이미 존재**. 폴링 증분 조회(`room_id=… AND created_at>커서 ORDER BY created_at`)를 이 인덱스가 커버한다. 새 인덱스/마이그레이션 **불필요**(이 스토리는 DB 변경 없음 — 클라이언트만).
- **RLS(참여자 한정, 이미 적용)**:
  - `chat_messages_select_participant` — 메시지가 속한 방(`chat_rooms`)의 당사자(buyer/seller=`auth.uid()`)만 SELECT. 제3자 0건(AC#6).
  - `chat_messages_insert_participant` — `auth.uid() = sender_id` **AND** 그 방의 당사자일 때만 INSERT. → 남의 명의 위조·끼어들기 차단. 따라서 `sendMessage`는 `sender_id`에 본인 id(`myUserId`)를 넣어야 통과한다(다른 값이면 RLS 거부).
  - UPDATE/DELETE 정책 없음 → 메시지 영속·불변(FR21·NFR5). 관리자 삭제는 Epic6(0005).
- **ai_readonly 영향 없음**: 0003 헤더 주석대로 chat 테이블엔 ai_readonly 가시성 정책이 없어 AI 경로는 0건(의도된 안전). 이 스토리와 무관.

### 폴링 구현 방식(확정 — 증분 조회 + id dedupe + 인터벌 정리)

```
mount:
  rows = fetchMessages(roomId)            # 커서 없음 = 전체(시간 오름차순)
  setMessages(rows); cursorRef = last(rows)?.created_at ?? null

interval(4000ms):                          # NFR1 3~5초 → 4초
  res = fetchMessages(roomId, cursorRef)   # gt(created_at, cursor) 증분
  if res.messages:
    setMessages(prev => dedupeById([...prev, ...res.messages]))  # id 중복 제거
    cursorRef = max(cursorRef, last(res.messages)?.created_at)

send(body):
  if !body.trim() or sending: return
  res = sendMessage(roomId, myUserId, body)
  if res.message:
     setMessages(prev => dedupeById([...prev, res.message]))     # 낙관적 즉시 표시
     cursorRef = max(cursorRef, res.message.created_at)
     clear input
  else: show korean error; restore input

unmount: clearInterval                     # AC#5 누수 방지
```

- **왜 증분+id dedupe인가**: created_at 커서만으로는 (a) 동시각 다중 메시지가 경계에 걸려 누락, (b) 내가 보낸 메시지를 폴링이 또 가져와 중복 — 두 위험이 있다. **id 기준 dedupe**가 둘 다 흡수한다(이미 있는 id면 무시). 데모 규모에선 전량 재조회도 가능하나 증분이 네트워크·렌더 모두 가볍고 의도가 명확.
- **stale closure 주의**: `setInterval` 콜백은 생성 시점 상태를 캡처한다. 커서는 `useRef`로, 목록 머지는 `setMessages((prev)=>…)` 함수형 업데이트로 다뤄 인터벌을 재생성하지 않고도 최신값을 참조한다(React 폴링 표준 패턴). roomId는 effect 의존성에 넣어 방이 바뀌면 인터벌을 정리·재시작.
- **주기**: 4000ms(3~5초 중앙). 상수로 두고 주석에 NFR1 근거 표기.

### 폴리글랏 일관성(AR5)

- 통신선/컬럼 snake_case 유지(`room_id`, `sender_id`, `created_at`, `body`). 이 스토리는 Supabase 직접 접근(클라이언트 RLS)이라 FastAPI 에러 포맷(`{error:{code,message}}`)은 직접 해당 없음 — 사용자 표시 에러는 한국어 일반 안내로 통일(원본 비노출, 5-2 규칙). 단위 변환 없음(메시지는 자유 텍스트).

### 범위 컷(과구현 금지)

- **읽음 표시·타이핑 인디케이터·푸시 알림·웹소켓/Realtime 구독은 범위 밖.** NFR1이 "폴링 준실시간 3~5초"로 못박았다 — Supabase Realtime을 끌어오지 않는다(데모 단순성·NFR 합치).
- **메시지 수정/삭제 없음**(DB도 UPDATE/DELETE 정책 없음 = 영속·불변). 관리자 채팅 열람·삭제는 Epic6(6-5).
- **무한 스크롤·페이지네이션 없음**(데모 대화량 소규모). 전체 로드 후 증분 폴링으로 충분. 필요 시 후행 개선.
- DB 스키마 변경 없음 — 0003에서 테이블·RLS·인덱스가 이미 완비됨. 마이그레이션 추가 금지.

### Completion Notes List

- **무엇을·왜**: 5-2가 빈 골격만 두었던 채팅방에 **메시지 송수신과 폴링 수신**을 채웠다. 이제 채팅방에서 메시지를 입력해 보내면 DB(`chat_messages`)에 저장되어 영속(새로고침해도 남음, FR21)되고, 상대가 보낸 메시지는 **4초마다 자동으로 새로고침 없이** 내 화면에 나타난다(폴링, FR20·NFR1). 메시지는 시간순으로 쌓이고 내 것은 오른쪽(검정)·상대 것은 왼쪽(테두리)으로 구분된다.
- **폴링 방식(증분 + gte + id dedupe)**: 전량 재조회 대신 "마지막으로 받은 시각 이후"만 가져오는 증분 조회로 가볍게 만들었다. 처음엔 `gt(>)`로 짰는데, 라이브 DB 검증에서 **같은 시각에 찍힌 메시지를 놓치는 버그**를 발견해(빠른 연속 전송 시 발생 가능) `gte(>=)`로 바꾸고, 경계에서 다시 딸려오는 중복은 **id 기준 dedupe**로 걸러 "누락 0·중복 0"을 달성했다. 인터벌은 컴포넌트를 떠날 때 `clearInterval`로 정리해 메모리 누수·유령 요청을 막았다(`useRef` 커서로 stale closure도 회피).
- **보안(DB가 집행)**: 전송 시 `sender_id`는 본인 id라야 RLS(`insert_participant`)를 통과한다(남 명의 위조·끼어들기 차단). 조회도 RLS(`select_participant`)가 방 당사자에게만 허용 — 제3자는 0건. 라이브 세션 검증으로 이 두 가드를 확인했다. 빈 메시지는 클라이언트 1차 + DB CHECK 이중 차단.
- **회귀 방지(AC#7)**: 진입 페이지는 빈 골격 `<section>`만 컴포넌트로 교체하고, 매물 헤더·sold 빈 상태 플레이스홀더(5-2 옵션D)·"채팅방 목록으로" 링크·error/없음 분기·`force-dynamic`은 그대로 뒀다. E2E로 `/chat` 목록·헤더가 깨지지 않음을 확인했다.
- **범위 준수**: 읽음표시·타이핑·푸시·Realtime 구독·메시지 수정/삭제·페이지네이션은 만들지 않았다(NFR1 폴링 명시·데모 단순성). DB/마이그레이션/proxy 변경 없음(5-1·5-2에서 완비).
