## Story

As a 비로그인 방문자,
I want 로그인 없이 매물을 둘러보길,
so that 가입 전에 "여기 살 만한 차가 있는지" 확인하고 결정할 수 있다.

> **이 스토리의 성격 (개발자 필독):** 증분(Epic 9~16)이 딛고 설 **접근제어 토대**다. 지금 서비스는 매물 목록·상세·AI 검색 **전부 로그인 필수**다(3중 게이트: web `proxy.ts` 라우트 보호 · DB `listings` RLS가 `to authenticated`뿐 · api `/ai/search` JWT 필수). FR58은 이 중 **열람(목록·상세)만 anon으로 열고, 게이트는 행동(AI 검색·문의·등록·찜)에만** 남긴다. **역할 통합 본체(signup 역할선택 제거·트리거 기본 role·role CHECK 완화 = FR52~54)는 이 스토리 범위 밖(Epic 14)** — 여기선 anon 읽기 경로만 연다. **A3 외과적 변경**: 손대는 곳은 아래 "손대는 파일 지도"에 열거된 것뿐 — 새 기능(찜·이미지·나머지 UI)은 후속 에픽 몫이다.

> **⚠️ AC 개정 (2026-07-14, 코드리뷰 + party-mode) — AI 검색은 로그인 필수로 남는다**
> 최초 AC-2는 `/ai/search`의 JWT 게이트 **완화**를 요구했고 1차 구현은 그대로 했으나, 코드리뷰에서 **anon `/ai/search`에 레이트리밋·쿼터·비용통제가 전무**함이 드러나 정책 자체를 재검토했다. 결론: **철회**.
> - **왜**: AI 검색 1회 = Gemini 호출 3회 내외(질문 다듬기·경로 판단·SQL/답변 생성) = **실제 과금**. JWT는 호출자를 식별하는 **유일한 수단**이었고, 그것이 곧 **유일한 과금 울타리**였다. `ai_readonly`·`sql_guard`는 **DB 권한**을 지키지 **API 키 지출**은 전혀 막지 않는다 — 아래 "핵심 설계 결정 ②"가 참이면서도 결론이 틀렸던 이유(I2의 안전 검토가 비용 축을 통째로 누락).
> - **풀 고갈이 먼저다**: `max_size=8`이라 남용은 돈이 새기 전에 **정상 사용자 503**(자기 DoS)로 번진다.
> - **대안 폐기**: 익명 N회 제한은 Cloud Run 다중 인스턴스에서 카운터를 공유 못 해 "3회"가 "3~15회 랜덤"이 된다(scale-to-zero 시 리셋). 진짜로 만들려면 Redis+VPC×2서비스 — 얻는 건 프록시로 뚫리는 IP 상한. **A2 과설계.** 멀티턴 제한은 `req.context`가 클라 자기신고라 `context=[]`로 우회됨(명세 자기모순).
> - **FR58은 살아있다**: 원문이 "매물 열람**과** AI 검색"이라 분해 가능하고, PRD 각주가 이 조정을 이미 예고했다(*"비용이 문제되면 그 전송 버튼에 로그인 게이트/rate-limit을 붙여 쉽게 조인다"*) — 그중 **더 싸고 확실한 쪽**을 택한 것. UX 산출물의 비로그인 여정(`EXPERIENCE.md:145-148`)도 anon 가치를 **열람+찜**으로 그렸지 AI로 그리지 않았다.
> - **계정당 쿼터가 필요해지면** JWT 사용자 ID로 Postgres 카운트 한 줄이면 된다(Redis 불필요). **신원을 지켜두면 나중 제한이 싸진다.**

## Acceptance Criteria

원문(`epics-increment-2026-07-12.md` Story 8.5) BDD — 2026-07-14 개정 반영:

**Given** 비로그인(anon) 사용자
**When** 매물 목록·상세에 접근하면

1. **(FR58 열람 — DB anon SELECT)** 매물 목록(`/search`)·상세(`/listings/[id]`)를 비로그인으로 열람할 수 있다. `listings` RLS에 **anon SELECT 경로**가 확립되되, **`status='on_sale'`만** 노출하고 `sold`는 어느 경로로도 안 보인다(FR11 sold 비노출 유지 — 규칙7 새 조회경로엔 이 필터 필수).
2. **(anon 컬럼 스코프)** RLS는 **행**만 통제하고 **컬럼**은 통제하지 못한다. anon에게 `grant select (필요 컬럼)`을 명시해 **`embedding`(vector 768) 등 비노출 컬럼을 차단**하고, 동시에 플랫폼 기본 GRANT 암묵의존을 없애 마이그레이션을 self-contained하게 만든다(8.6 CI 게이트 대비).
3. **(AI 검색 = 행동, JWT 필수 유지)** `/ai/search`는 **로그인 필수를 유지**한다 — 무토큰 401, 무효 토큰 401. `/ai`는 `proxy.ts`의 `PROTECTED_PREFIXES`에 남는다. (개정 근거는 위 박스. ~~구 AC-2 "JWT 게이트 비로그인 완화(I2)"는 철회~~)
4. **(게이트는 행동에만 + 원위치 복귀)** 로그인 게이트는 **"행동"(AI 검색·문의·매물 등록·찜)에만** 적용됨이 계약으로 확립되고, 비로그인이 행동을 시도하면 로그인으로 보낸 뒤 **원래 있던 위치로 복귀**한다(`redirectedFrom` 실동작 — 현재 로그인 페이지가 무시 중인 것을 고친다).
5. **(범위 경계)** 역할 통합 본체(FR52~54: signup 역할선택 제거·`handle_new_user` 트리거·role CHECK 완화)는 **이 스토리에서 건드리지 않는다**(Epic 14). 여기선 열람 토대만 다룬다(FR58 정책 소유 = Epic 8).
6. **(회귀 0)** 이 변경이 **로그인 사용자의 기존 흐름을 깨지 않는다** — 로그인 상태에서 목록·상세·AI 검색·문의·등록이 그대로 동작한다(web E2E + api 결정론 회귀).

### 핵심 설계 결정 ② — ⚠️ **참이지만 결론이 틀렸던 논증** (2026-07-14 정정 — 반드시 읽을 것)
**당시 논증(사실관계는 지금도 참)**: `/ai/search`의 DB 읽기는 사용자 JWT가 아니라 서버측 `ai_readonly` 롤(`readonly.py`, `BEGIN; SET LOCAL ROLE ai_readonly; …; COMMIT`)로 수행된다. 인증을 완화해도 **DB 권한은 anon 여부와 무관하게 항상 ai_readonly(SELECT 전용)** 다. `sql_guard.py`가 SELECT만·`status='on_sale'`을 강제한다(코드리뷰 실측: `sql_guard.py:152-159`가 on_sale 존재검증 후 없으면 거부, `117-121`이 OR 우회까지 금지 — **"AI로 sold가 샌다"는 리뷰 지적은 오탐이었다**).

**❌ 그런데 결론이 틀렸다**: 위 논증은 *"JWT 완화가 **권한 누수**를 만들지 않는다"* 를 증명하지, *"열어도 **안전하다**"* 를 증명하지 않는다. **`ai_readonly`도 `sql_guard`도 돈을 지키는 장치가 아니다.** 검색 1회 = Gemini 호출 3회 내외 = **실제 과금**이고, JWT는 호출자를 식별하는 **유일한 수단** = 유일한 과금 울타리였다. 그걸 제거하면서 비용 축을 아무도 안 봤다.

**교훈(재발 방지)**: "안전한가?"를 물을 땐 **어느 축의 안전인지** 명시하라. 여기선 DB 권한 축만 검토됐고 비용 축은 질문조차 되지 않았다. 이 누락은 나 혼자의 실수가 아니라 **PRD 각주(`prd.md:138` "비용이 문제되면 로그인 게이트/rate-limit으로 조인다")가 아키텍처 I2·에픽 AC-F14-1·이 스토리로 전파되지 않은 계보 끊김**의 결과다. → `docs/conventions.md` §8에 **"비용이 발생하는 엔드포인트는 신원을 요구한다"** 를 원칙으로 못 박았다(Epic 13 RAG 임베딩 비용에도 동일 적용).

### 핵심 설계 결정 ③ — 열람 vs 행동 게이트 배치 (2026-07-14 개정)
- **분류 기준(신설)**: 서버 자원을 **읽기만** 하면 열람, 상태를 바꾸거나 **외부 유료 API를 호출해 청구서를 만들면** 행동. 화면이 아니라 **대가**로 가른다.
- **web `proxy.ts`(라우트 1차 게이트)**: 열람 경로(`/search`·`/listings`) 개방, 행동/개인 경로(**`/ai`**·`/sell`·`/chat`·`/admin`) 유지. **`/ai`는 위 기준상 행동**이라 보호에 남는다(개정 전엔 열람으로 분류해 열었다). `/chat`=문의 인박스(개인 대화 목록)라 anon 무의미 → 보호 유지. `/sell`=매물 등록(행동) → 보호 유지(역할 2차 게이트는 `(user)/sell` 레이아웃 `requireRole` — 불변).
- **컴포넌트(2차 게이트)**: 문의 버튼처럼 **페이지는 공개지만 특정 액션만 로그인 필요**한 경우, 액션 지점에서 `redirectedFrom` 링크로 게이트. 이게 "게이트는 행동에만"의 실제 구현 모양.

### FR11 재확인 (규칙7 — 새 조회경로엔 sold 필터 필수)
anon SELECT는 **새 조회 경로**다. 이중 방어로 sold를 막는다: ① **RLS** `to anon using (status='on_sale')` — DB가 anon에 sold 행 자체를 안 준다 ② **web** `buyerListingsQuery`가 `.eq('status','on_sale')` 재강제(불변). 상세 페이지는 `maybeSingle()` → sold면 null → "찾을 수 없음"(FR11). ai 경로는 sql_guard가 on_sale 강제. **강제지점 유지, 앱레벨 필터링 추가 금지.**
