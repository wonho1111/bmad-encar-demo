## 잘된 점 (반복할 패턴)

1. **마이그레이션 1개로 에픽 완주.** 0005만 새로 만들고 6-2~6-5는 DB 변경 0 — 기존 RLS·가드·cascade를 재사용. Epic 5 회고가 예고한 "Epic 6는 기존 자산 위에 admin 정책만 얹는 구조"가 그대로 적중. 새 테이블·새 규칙을 만들기 전에 기존 불변식이 답하는지 본 결과.
2. **독립 코드리뷰가 자체 리뷰의 사각을 메움.** 6-3에서 dev 서브에이전트가 지시 없이 자체 code-review를 돌려 done 표기했지만, 오케스트레이터가 **별도 fresh-context 독립 리뷰**를 강제 → 그 리뷰가 매물 삭제 시 채팅방·메시지 cascade 무결성을 **DB rollback 프로브로 실증**(자체 리뷰는 "결함 0"만 선언). 자체 리뷰 ≠ 독립 검증.
3. **FR11 예외(관리자 sold 조회)를 OR 결합으로 깔끔히.** 구매자 비노출 단일규칙(`status='on_sale'`)을 건드리지 않고, `listings_select_admin(using=is_admin())`이 관리자 세션에만 sold를 열어줌. 새 규칙 없이 기존 정책에 admin 분기만 추가 — Epic 2 회고의 "RLS OR-결합" 교훈 재적용.
4. **cascade 삭제를 DB FK에 위임.** 매물 삭제→그 매물 채팅방→메시지가 `0003`의 `ON DELETE CASCADE`로 자동 정리. 앱이 자식 행을 직접 지우지 않아 고아·FK 위반 없음(독립 리뷰가 트랜잭션 rollback으로 실증).
5. **관리자 자기 보호 가드.** 회원 관리에서 관리자 본인 행은 "나" 배지 + 액션 버튼 숨김으로 자기 정지/삭제 차단(UI 1차). 데모의 흔한 실수 동선을 선제 제거.
6. **공유 표시 컴포넌트로 drift 차단.** 에픽 후 보강에서 구매자 상세와 관리자 상세가 `ListingDetailFields` 하나를 공유 — 15필드 마크업을 복붙하지 않아 한쪽만 바뀌는 drift를 원천 차단(5-3의 `lib/messages.ts` 단일출처 패턴 연장).
7. **실사용 피드백 → 분류 체계 → 차등 처리.** 사용자가 데모를 직접 써보고 4개 발견 → party-mode로 "결함/취향/신규기능"으로 가르고 "지금 핫픽스/회고 보류/불필요"로 처리. **무조건 다 고치지 않는 게 데모 규율**(폴링·카피만 즉시, 동선·일관성은 보류).

## 어려웠던 점 / 반복된 함정

1. **dev 서브에이전트가 지시 범위를 넘어 자체 리뷰까지 수행(6-3).** 오케스트레이터 지시는 "dev까지, 상태 review"였는데 스스로 code-review를 돌려 done으로 올림. 문맥 편향 위험 → 이후 6-4·6-5는 "code-review 직접 돌리지 말 것"을 지시에 명시하고 독립 리뷰를 별도 강제.
2. **완전 계정 삭제(auth.users)는 service-role 키 부재로 범위 밖(6-2).** 회원 "삭제"가 `profiles` 행만 제거하고 로그인 계정 자체는 남음. AC "계정이 제거된다"를 profiles 삭제로 해석(클라이언트에 service-role 키 노출 금지 원칙). 의도된 데모 범위 결정 — `deferred-work` 인계.
3. **Turbopack 증분 캐시 꼬임(에픽 후 보강).** 새 동적 라우트 `/admin/listings/[id]` 추가 후 `/login`이 컴파일 에러 없이 404 반복. `.next` 캐시 삭제로 즉시 해결. **코드 문제가 아니라 dev 번들러 캐시 특성** — 라우트 트리 변경 후 재현 가능.
4. **거래일 "간이" 방식의 타임존 흔들림.** `sold_at` 컬럼이 없어 `updated_at`을 거래일로 근사 → 서버(UTC) 렌더 vs 로컬(KST) 차이로 날짜 경계에서 ±1일(로컬 6.22 = preview 6.21). 간이 방식의 알려진 한계로 투명 기록, 정확안(sold_at)은 보류.
5. **로컬 dev HMR 첫 클릭 삼킴(Epic 5부터 계속).** Playwright 첫 클릭이 Fast Refresh에 삼켜짐 → 직접 URL 이동으로 우회. 운영 빌드 preview엔 없음(환경 특성).

## 핵심 교훈

1. **기존 자산을 재사용하면 에픽이 마이그레이션 1개로 끝난다.** 새 규칙·새 테이블을 짜기 전에 기존 RLS·가드·cascade가 이미 답하는지 보라. 0005 하나로 5개 스토리가 완결된 건 Epic 1~5가 쌓은 불변식 덕분(Epic 5 "기존 불변식이 먼저 답했는지 보라"의 직접 연장).
2. **자체 리뷰는 독립 리뷰를 대체하지 못한다.** 같은 문맥이 만든 결론은 같은 사각을 갖는다. fresh-context 적대적 재검증이 "결함 0 선언"을 "cascade 무결성 DB 실증"으로 바꿨다 — 토대·보안 사안일수록 독립 검증을 분리하라.
3. **공유 표시 로직은 컴포넌트로 추출해 drift를 막는다.** 두 번째 복붙이 보이면 단일출처화 신호(Epic 3 교훈). 구매자·관리자 상세가 한 컴포넌트를 쓰면 규칙 변경이 양쪽에 자동 적용된다.
4. **실사용 발견은 "분류"부터 — 전부 고치는 게 능사가 아니다.** 결함(거짓 카피)은 즉시, 취향(정렬)·신규기능(판매자 탐색)은 회고/보류. 데모 마감 직전 회귀 위험 큰 리팩터를 "더 좋게"라는 이유로 손대지 않는 게 규율.
5. **"정확"과 "간이"를 투명하게 구분해 남긴다.** 거래일 `updated_at` 근사는 한계(타임존 ±1일)를 명시하고 정확안(`sold_at` 마이그레이션)을 `deferred-work`에 남겨, 나중에 필요할 때 즉시 판단하게 한다.
