# Story 2.3: 본인 매물 조회·수정·삭제

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 판매자,
I want 내가 올린 매물만 조회·수정·삭제하고 싶다,
so that 내 매물을 안전하게 관리한다.

## Acceptance Criteria

1. **(본인 매물만 조회)** 판매자로 로그인한 상태에서 내 매물 목록을 열면 **본인 매물만** 보인다. `status='sold'`(판매완료)를 포함한 본인 매물 **전부**가 보인다(관리용 목록이므로 비노출 규칙 FR11은 구매자 화면에만 적용). [FR6]
   - ⚠️ **반드시 `.eq('seller_id', user.id)` 명시 필터**를 건다. listings SELECT RLS가 `on_sale ∪ own ∪ admin`의 **OR 결합**이라, 필터 없이 select하면 남의 `on_sale` 매물까지 섞여 들어온다(2-2 E2E에서 실제 발생·수정한 버그). [Source: 2-2 Dev Notes·0002_listings.sql:90-101]
2. **(본인 매물 수정)** 본인 매물을 수정하면 반영된다. 수정 폼은 2-2의 등록 폼(15필드)·검증 로직·`LISTING_OPTIONS`/`LISTING_RANGES` 단일출처를 **재사용**한다(재발명 금지). 수정 시 기존 값이 폼에 미리 채워지고, 저장하면 `listings` 행이 UPDATE된다. `status`는 수정 폼에서 변경하지 않는다(구매완료 전환은 2-4). [FR6]
3. **(본인 매물 삭제)** 본인 매물을 삭제하면 행이 제거되고 목록에서 사라진다. 실수 방지를 위해 삭제 전 확인 단계를 둔다. [FR6]
4. **(타인 매물 차단 — RLS)** 타인 매물 id로 수정/삭제를 시도하면 **RLS로 차단**된다. 앱에서도 안전하게 처리한다: 본인 소유가 아닌 id로 수정 화면 진입 시 매물이 조회되지 않으므로 한국어 안내("매물을 찾을 수 없거나 접근 권한이 없습니다.")를 보여주고, UPDATE/DELETE가 0행이면 한국어로 거절 안내한다(원본 에러·코드는 콘솔만). [FR6·NFR2]
5. **(역할 게이트 유지)** 수정/삭제·본인 매물 관리는 seller 역할만 접근 가능하다(2-2의 `(user)/sell` layout `requireRole(SELLER)` + proxy `/sell` 보호가 그대로 커버). 비로그인→`/login`, 다른 역할→홈. [FR3 기반·2-2 확정]

> **범위 주의(반드시 준수):** 이 스토리는 **본인 매물 조회·수정·삭제(FR6)** 까지다. **구매완료(sold) 전환은 2-4**(스코프 침범 금지 — 수정 폼·관리 UI에 "구매 완료" 버튼/상태 변경을 만들지 마라), **샘플 시드는 2-5**, **구매자용 목록·필터·상세(FR9~11)는 Epic 3** 소관이라 만들지 않는다. 마이그레이션·DB 스키마는 2-1에서 완료됐으므로 **건드리지 않는다**(UPDATE/DELETE RLS·소유권 가드는 이미 라이브). `embedding`은 NULL로 둔다(Epic 4 backfill).

## Dev Notes

### ⚠️ 이건 당신이 아는 Next.js가 아니다 (web/AGENTS.md 경고)
- `web/AGENTS.md`: "코드 작성 전 `node_modules/next/dist/docs/`의 관련 가이드를 읽으라." Next.js **16.2.x**. 동적 라우트 `[id]`, 서버/클라이언트 컴포넌트 경계, `proxy.ts`(구 middleware) 관례를 따른다. 동적 라우트 params는 Next 16에서 **`Promise`로 await** 해야 한다(가이드 확인). [Source: web/AGENTS.md, node_modules/next/dist/docs/]

### 2-1/2-2에서 확정된 DB·앱 계약 (재발명·재설계 금지)
- `listings` 테이블·6개 RLS 정책·CHECK·트리거는 **2-1에서 라이브 Supabase에 적용 완료**. 이번 스토리는 DB를 건드리지 않는다.
- **UPDATE 정책 `listings_update_own`**: `using (auth.uid()=seller_id) with check (auth.uid()=seller_id)` → 타인 매물 수정·status 전환·seller_id 재지정 차단. 본인 매물만 UPDATE 가능. [Source: 0002_listings.sql:111-114]
- **DELETE 정책 `listings_delete_own`**: `using (auth.uid()=seller_id)` → 본인 매물만 삭제. [Source: 0002_listings.sql:117-118]
- **SELECT 정책 OR 결합**(`on_sale ∪ own ∪ admin`): 본인 매물 관리 목록·수정 진입 조회는 반드시 **`.eq('seller_id', user.id)`** 를 명시해야 남의 on_sale이 안 섞인다. 2-2 E2E에서 실제 버그로 확인·수정됨. [Source: 0002_listings.sql:90-101, 2-2 Dev Notes line 143·159]
- **updated_at 트리거 + created_at 불변**: UPDATE 시 `set_updated_at()`가 `updated_at=now()`, `created_at=old.created_at` 강제 → update payload에 created_at을 넣어도 무시·보호된다. [Source: 0002_listings.sql:65-84]
- **앱 쿼리에서 RLS 거부의 표현**: RLS로 막힌 UPDATE/DELETE는 에러가 아니라 **0행 영향**으로 나타나는 게 일반적(PostgREST). 따라서 `.select()`를 붙여 **반환 행 수 0**을 "차단됨"으로 판정하고 한국어 안내한다(AC4 앱측 안전 처리). 권한 에러 코드(42501)가 올 수도 있으니 `toKoreanError`도 함께 둔다. [Source: 0002_listings.sql RLS, 2-2 SellForm toKoreanError]

### CHECK/RLS 위반 → 한국어 에러 (AC4 핵심)
- 정상 흐름(본인 매물만 진입·드롭다운 6필드)에선 위반이 안 생긴다. 그래도 **앱측 방어**:
  - 수정 진입 조회 0건 → "매물을 찾을 수 없거나 접근 권한이 없습니다."
  - UPDATE/DELETE 반환 0행 → "본인 매물만 수정/삭제할 수 있습니다." (RLS 거부 가능성)
  - 권한/CHECK 에러 코드(42501/23514)는 `toKoreanError`로 한국어 변환, 원본은 콘솔만.
