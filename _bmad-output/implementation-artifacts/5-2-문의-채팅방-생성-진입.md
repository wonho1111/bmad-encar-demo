# Story 5.2: 문의 채팅방 생성·진입 (FR19)

## Story

As a 구매자,
I want 매물에 문의를 보내 채팅방을 열고 싶다,
so that 판매자와 직접 대화한다.

## Acceptance Criteria

1. **Given** 매물 상세(`/listings/[id]`)에서, **When** "문의하기" 버튼을 누르면, **Then** 그 매물의 판매자와의 채팅방이 생성되고 해당 채팅방으로 진입한다(상대 선택 화면 없음 — 매물 주인에게 자동 연결). (epics.md L605~609, architecture.md L407)
2. **Given** 같은 매물에 대해 같은 구매자가 다시 "문의하기"를 누르면, **When** 이미 동일 (매물, 구매자, 판매자) 조합의 방이 있으면, **Then** 새 방을 만들지 않고 **기존 방을 재사용**해 그 방으로 진입한다(중복 방 생성 금지 — UNIQUE 의존). (epics.md L609, 5-1 AC#5)
3. **Given** 채팅방 목록(`/chat`)에서, **When** 방을 선택하면, **Then** 해당 대화(`/chat/[roomId]`)로 진입한다. (epics.md L611~613)
4. **Given** 채팅방 목록(`/chat`)에서, **When** 내가 당사자(구매자 또는 판매자)인 방만, **Then** 상대·매물 요약과 함께 목록에 보인다(RLS 참여자 한정으로 제3자 방은 0건). 방이 하나도 없으면 빈 상태 안내.
5. **Given** 채팅방(`/chat/[roomId]`)에 진입했을 때, **When** 내가 그 방의 당사자가 아니거나 존재하지 않는 방이면, **Then** 접근이 차단되어 "찾을 수 없음/접근 불가" 안내가 보인다(RLS 0건). 당사자면 상대·매물 헤더와 **메시지 영역의 빈 골격**(메시지 송수신은 5-3 범위)이 보인다.
6. **Given** 판매자 본인이 자기 매물 상세를 볼 때, **When** "문의하기" 버튼 영역을 보면, **Then** 자기 자신에게 문의할 수 없으므로 버튼이 비노출되거나 시도 시 거부된다(`CHECK(buyer_id <> seller_id)`로 DB도 막음 — 이중 방어).
7. **Given** 비로그인 사용자가 `/chat` 또는 `/chat/[roomId]`에 접근하면, **When** 라우트 가드(proxy)가 평가하면, **Then** `/login`으로 리다이렉트된다(보호 경로). (proxy.ts PROTECTED_PREFIXES)

### 핵심: chat 스키마·트리거 동작(신뢰하고 활용)
- `chat_rooms(id, listing_id, buyer_id, seller_id, created_at)` + `UNIQUE(listing_id, buyer_id, seller_id)` + `CHECK(buyer_id <> seller_id)`.
- **BEFORE INSERT 트리거 `enforce_chat_room_seller`** 가 `seller_id`를 **그 매물(`listing_id`)의 실제 소유자(`listings.seller_id`)로 강제 덮어쓴다.** 따라서:
  - 방을 만들 때 **클라이언트는 `seller_id`를 보낼 필요가 없다**(보내도 무시되고 매물주로 교정됨). → `insert({ listing_id, buyer_id })`만.
  - 존재하지 않는 `listing_id`면 트리거가 거부(code `23503` 계열).
  - 구매자=매물주(본인 매물 문의)면 트리거 후 `seller_id == buyer_id`가 되어 `CHECK(buyer_id <> seller_id)` 위반(code `23514`)으로 거부 → "본인 매물에는 문의할 수 없습니다"로 매핑(AC#6 DB측).
- **재사용 식별키**: seller_id가 트리거로 결정되므로, 같은 (listing, buyer)면 항상 같은 seller로 귀결 → `(listing_id, buyer_id)`만으로 방을 유일하게 select할 수 있다. select-first는 이 두 컬럼으로 조회한다.
- RLS:
  - `chat_rooms_insert_participant` — 만드는 사람이 buyer 또는 seller여야 통과. 구매자가 `buyer_id = 본인`으로 insert → 통과(정합).
  - `chat_rooms_select_participant` — `auth.uid() = buyer_id or seller_id`. 목록/진입 조회 시 내 방만 보인다(제3자 0건 = AC#4·AC#5).

### 방 생성·재사용 구현 방식(확정 — select-first + insert + 경합 fallback)
```
async openOrCreateRoom(supabase, listingId, buyerId):
  # 1) 기존 방 재사용 우선
  existing = supabase.from('chat_rooms')
    .select('id').eq('listing_id', listingId).eq('buyer_id', buyerId).maybeSingle()
  if existing.data: return { roomId: existing.data.id }

  # 2) 없으면 생성 (seller_id 미지정 — 트리거가 매물주로 강제)
  ins = supabase.from('chat_rooms')
    .insert({ listing_id: listingId, buyer_id: buyerId }).select('id').single()
  if ins.data: return { roomId: ins.data.id }

  # 3) UNIQUE 경합(23505) → 그새 다른 탭/요청이 만든 방을 재조회
  if ins.error.code == '23505':
     retry select-first → return that id
  # 4) CHECK 위반(23514, 본인 매물) → 한국어 거부
  if ins.error.code == '23514':
     return { error: '본인 매물에는 문의할 수 없습니다.' }
  # 5) 그 외 → 일반 한국어 에러(원본은 console.error만)
  return { error: '채팅방을 여는 중 오류가 발생했습니다. 잠시 후 다시 시도해주세요.' }
```
- 왜 select-first인가: UNIQUE에 `seller_id`가 포함돼 있어 클라이언트가 upsert `onConflict`를 정확히 지정하기 까다롭고(seller_id를 모름), insert 충돌 시 PostgREST가 행을 돌려주지 않아 재조회가 필요하다. select-first가 가장 단순·견고하고 "기존 방 재사용"(AC#2) 의도를 직접 표현한다.
- 에러 코드는 `error.code`(PostgREST/Postgres SQLSTATE)로 분기. `23505`=unique_violation, `23514`=check_violation, `23503`=foreign_key_violation.

### 폴리글랏 일관성(AR5)
- 통신선/컬럼 snake_case 유지(`listing_id`, `buyer_id`, `room_id`). 이 스토리는 Supabase 직접 접근(클라이언트 RLS 경유)이라 FastAPI 에러 포맷(`{error:{code,message}}`)은 직접 해당 없음 — 단 사용자 표시 에러는 한국어 일반 안내로 통일(원본 메시지 비노출, 기존 ListingActions 규칙).

### Completion Notes List

- **무엇을·왜**: 구매자가 매물 상세에서 "문의하기" 한 번으로 그 매물 판매자와의 채팅방이 자동으로 열리게 만들었다(상대 고르는 화면 없음). 같은 매물에 다시 문의하면 새 방을 만들지 않고 **기존 방을 재사용**한다. 그리고 내 채팅방 목록(`/chat`)과 각 방 진입 화면(`/chat/[roomId]`)을 추가했다. 메시지를 주고받는 기능(입력·전송·폴링)은 다음 스토리(5-3) 몫이라, 진입 화면엔 빈 메시지 골격만 뒀다.
