## Story

As a 운영자,
I want 가입 흐름과 무관하게 미리 존재하는 별도 관리자 계정을 원한다,
so that 일반 가입 경로를 거치지 않고도 관리 기능(Epic 6)에 진입할 수 있다.

## Acceptance Criteria

1. **(seed.sql → admin 계정 1개 생성)** `supabase/seed.sql`이 실행되면, `role='admin'` 프로필을 가진 관리자 계정 **정확히 1개**가 생성된다. (재실행해도 중복 생성되지 않는다 — 멱등.)
2. **(admin 로그인 → 권한 인식)** 생성된 관리자 계정으로 로그인하면 관리자 권한이 인식된다 — 홈에서 "역할 관리자"로 표시되고, `is_admin()`이 `true`이며, `/admin` 보호 경로를 통과한다. (실제 관리자 화면·기능은 **Epic 6** 소관. 여기서는 자리표시 페이지 통과까지만.)
3. **(일반 가입 경로로 admin 선택 불가 — 검증만)** 일반 가입 경로(`/signup`)로는 `admin` 역할을 만들 수 없다. UI는 구매자/판매자만 제시하고, DB 트리거(`handle_new_user`)는 admin 메타를 buyer로 강제한다. **본 스토리는 이 차단을 검증만** 하며, 코드/마이그레이션을 바꾸지 않는다.

> **범위 주의(반드시 준수):** 이 스토리는 **`supabase/seed.sql` 신설 + admin 계정 1건 시드 + 권한 인식 검증**까지다. seed.sql에는 **이번 스토리에 필요한 admin 계정만** 넣는다. 샘플 매물 시드(AR7)는 **Story 2.5**, 가이드 문서 코퍼스는 **Story 4.2** 소관 — 같은 `seed.sql` 파일에 후속 스토리가 섹션을 덧붙이는 구조로 두되, **이번엔 admin 섹션만** 작성한다. 관리자 화면·기능·`0005_admin_policies`(관리자 전권 교차 정책)는 **Epic 6**다(만들지 않음). `0001_profiles.sql`을 수정하지 말 것(이미 적용됨). 새 마이그레이션 파일을 만들지 말 것 — 시드는 **데이터**지 스키마가 아니다.

## Dev Notes

### ⚠️ 가장 중요 — SQL로 admin 계정 시드하기 (트리거 함정 + identities 필수)
중고차 데모는 **호스티드 Supabase**(로컬 스택 없음 — `config.toml` 부재, 0001도 MCP로 적용)를 쓴다. 그래서 admin 계정을 SQL로 직접 심어야 하는데, 두 개의 함정이 있다.

1. **`auth.identities`가 없으면 로그인 불가.** `auth.users`만 넣으면 계정은 보이지만 이메일+비밀번호 로그인이 실패한다. GoTrue(Supabase Auth 서버)는 비밀번호 로그인 시 identities를 참조한다. **최신 버전은 `auth.identities.provider_id`가 NOT NULL 필수** — 값으로 user의 `id`(text)를 넣으면 된다. (`provider_id` 누락이 시딩 실패의 흔한 원인.) [Source: 웹리서치 — Supabase 사용자 시딩 SQL 패턴, GitHub supabase-community/seed#208]
2. **트리거가 admin을 buyer로 깎는다.** `0001_profiles.sql`의 `on_auth_user_created` → `handle_new_user()`는 `raw_user_meta_data.role`이 buyer/seller가 아니면 **buyer로 강제**한다(가입 경로 admin 차단 = AC3의 DB측 집행). 그래서 `auth.users` insert 직후 profiles는 buyer다 → **`update profiles set role='admin'`로 승격**해야 한다. 이 "가입 흐름 밖에서의 직접 승격"이 정확히 FR4가 말한 *별도 경로로 admin 생성*이다. [Source: supabase/migrations/0001_profiles.sql:27-51]

비밀번호는 **반드시 `crypt('<암호>', gen_salt('bf'))`(bcrypt)** 로 해싱한다(`pgcrypto` 확장 필요 — Supabase는 기본 활성). 평문 저장 시 로그인 실패. `email_confirmed_at`을 `now()`로 채워야(또는 콘솔에서 Confirm email OFF) 로그인이 막히지 않는다. [Source: 웹리서치 — laros.io / dev.to 시딩 가이드]

개념 골자(실제 작성 시 컬럼·기본값 재확인):
```sql
-- supabase/seed.sql (개념) — pgcrypto의 crypt/gen_salt 사용
do $$
declare v_id uuid := gen_random_uuid();
        v_email text := 'admin@encar-demo.test';
begin
  if not exists (select 1 from auth.users where email = v_email) then
    insert into auth.users (
      id, instance_id, aud, role, email, encrypted_password,
      email_confirmed_at, raw_app_meta_data, raw_user_meta_data,
      created_at, updated_at)
    values (
      v_id, '00000000-0000-0000-0000-000000000000', 'authenticated', 'authenticated',
      v_email, crypt('<데모비밀번호>', gen_salt('bf')),
      now(), '{"provider":"email","providers":["email"]}'::jsonb, '{}'::jsonb,
      now(), now());
    insert into auth.identities (
      provider_id, user_id, identity_data, provider,
      last_sign_in_at, created_at, updated_at)
    values (
      v_id::text, v_id,
      jsonb_build_object('sub', v_id::text, 'email', v_email),
      'email', now(), now(), now());
  end if;
  -- 트리거가 만든 profiles(buyer)를 admin으로 승격 (멱등)
  update public.profiles set role='admin'
   where id = (select id from auth.users where email = v_email);
end $$;
```
> 위는 **개념**이다. `auth.users`/`auth.identities`의 실제 컬럼·NOT NULL·기본값은 호스티드 DB에서 `list_tables`(schemas=['auth'], verbose)로 확인 후 채운다. GoTrue 버전에 따라 컬럼이 다를 수 있으니 맹신 금지.

### 시드 적용 경로 (호스티드 — db reset 없음)
- 로컬 스택이 없으므로 `supabase db reset`(이게 보통 seed.sql을 자동 실행)을 쓸 수 없다. → **seed.sql 내용을 MCP `execute_sql`로 호스티드 DB에 직접 실행**한다.
- `seed.sql` 파일 자체는 **데이터 시드의 단일 출처(AR7)** 로 저장소에 남긴다(나중에 로컬 스택이 생기거나 재현이 필요할 때 그대로 재실행 가능). 적용 수단만 MCP일 뿐 출처는 파일.
- **`apply_migration` 쓰지 말 것** — 시드는 스키마 변경이 아니라 데이터다. 마이그레이션 이력(`list_migrations`)을 오염시키지 않는다. [Source: bmad-encar-demo/CLAUDE.md §4(마이그레이션은 스키마 변경만)]

- 1.4는 admin 계정이 없어서 `execute_sql`로 **임시** admin을 만들어 `/admin` 통과를 검증하고 지웠다. 1.5는 그 happy-path를 **영속 시드 계정**으로 정식 확정한다. 1.4 Dev Notes의 "관리자 happy-path 딜레마"가 여기서 해소된다. [Source: 1-4 Dev Notes(관리자 happy-path 딜레마), 1-4 Completion Notes]
