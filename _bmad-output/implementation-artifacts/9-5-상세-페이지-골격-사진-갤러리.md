# Story 9.5: 상세 페이지 골격 + 사진 갤러리

Status: ready-for-dev

## Story

As a 구매자,
I want 매물 상세에서 사진 갤러리를 넘겨보고 정보가 신뢰→차량→옵션→판매자 순으로 정리되길,
so that 구매 결정을 순서대로 신뢰 있게 내린다.

**범위: web only** (`web/src/app/(user)/listings/[id]/` + 새 갤러리 컴포넌트 + 이미지 조회 유틸 1개). api·app(Flutter)·DB 마이그레이션은 이 스토리 밖 — [Dev Notes §0](#0-범위-경계--먼저-읽을-것).

1. **섹션 골격 = 신뢰정보 → 차량정보 → 옵션 → 판매자정보** — 위에서 아래로 정확히 이 순서의 `<section>` 4개로 페이지를 세운다.
   - **① 신뢰정보**: Epic 10.2가 채울 **빈 슬롯**. 지금은 뱃지 값이 없으므로 **아무것도 렌더하지 않는다**(빈 제목·빈 테두리·"준비중" 문구 금지). 슬롯의 자리만 코드 구조로 존재하면 된다.
   - **② 차량정보**: 지금 있는 필드 표를 여기로 옮긴다(AC5).
   - **③ 옵션**: 지금 있는 옵션 칩 목록을 여기로 옮긴다. 카테고리 분류·희소옵션 강조는 **Epic 10.3/10.4의 몫이며 여기서 만들지 않는다.**
   - **④ 판매자정보**: Epic 10.6이 채울 **빈 슬롯**(①과 동일 규칙).
   - ⚠️ 목업 `ux-designs/.../mockups/detail-1.html`은 **옛 순서(옵션→차량정보)**다. 목업이 아니라 이 AC가 정답이다.
   (UX-DR24, D9, `epics-increment-2026-07-12.md:539`)

2. **사진 갤러리** — 사진이 1장 이상이면 갤러리를, 0장이면 5:3 "사진 준비중" 플레이스홀더를 그린다.
   - 구성: **대표 사진 5:3** + **하단 썸네일 스트립** + **"1/N" 카운터(대표 사진 우하단)** + **좌우 화살표**.
   - 썸네일 클릭·화살표 클릭으로 대표 사진이 바뀐다. 키보드로도 조작 가능해야 한다(화살표 버튼은 진짜 `<button>`, `aria-label` 한국어).
   - 첫 화면에 **대표 1장 + 썸네일들만** 로드한다. 큰 사진을 N장 미리 받지 않는다(`loading="lazy"`, `decoding="async"`, `aspect-ratio`로 레이아웃 시프트 0). (NFR7)
   - 크롭은 클라 렌더 크롭(`object-fit: cover`, 중앙). **저장본을 다시 만들지 않는다**(I14) — 9.3이 업로드 시 긴 변 ≤1600px·WebP·q0.82로 이미 다운스케일해 뒀다(실측 196~205KB).
   - 라이트박스(확대보기)는 **범위 밖**. 만들지 않는다.
   (FR28, UX-DR12, `DESIGN.md:141`, `EXPERIENCE.md:78`)

3. **갤러리 조회 정렬은 `order by sort_order, id`** — `listing_images`를 읽는 새 조회는 **반드시 2차 정렬 키 `id`까지** 건다.
   - 이유: `sort_order` 동률을 막는 DB 제약이 없다(`docs/tech-debt.md` #47-2 — 유니크 제약을 일부러 안 걸었다). 2차 키를 빼면 **새로고침할 때마다 사진 순서가 바뀐다.**
   - 대표 = 정렬 첫 행. **`is_cover` 컬럼을 보지 않는다**(파생값). 대표 판별 로직을 **새로 만들지 않는다** — 계산 자리는 `web/src/lib/images/coverImages.ts` 하나다.
   - **이 정렬을 고정하는 단위테스트 1건**을 남긴다(`sort_order` 동률 행을 섞어 넣어 순서가 결정론적임을 단언). 선례: `web/src/lib/images/coverImages.test.ts`.
   - ⇒ 이 AC가 통과하면 `docs/tech-debt.md` **#59를 닫는다**(9.4가 목록 카드 절반만 이행하고 열어 둔 항목).
   (`conventions.md` §10.1·§10.2, tech-debt #59)

4. **이미지 로드 실패 폴백은 반드시 2겹** — 갤러리 대표·썸네일 **모두**에 적용한다.
   - ① `onError` 콜백 + ② **ref 콜백에서 `img.complete && img.naturalWidth === 0` 확인**.
   - ⚠️ **`onError` 하나만 복사하면 같은 자리를 다시 밟는다.** 서버가 그린 `<img>`는 HTML 도착 즉시 로드를 시작하고, 하이드레이션 전에 끝난 실패는 React가 이벤트를 재생하지 않아 `onError`가 **영영 발화하지 않는다**. 404가 정확히 이렇게 빨리 실패한다(9.4 실브라우저 실측으로 발견, 리뷰 3개 레이어가 전부 못 잡음).
   - `getPublicUrl`은 **파일 존재를 확인하지 않는다**(순수 문자열 조립). 깨진 이미지를 막는 유일한 장치가 이 2겹이다.
   - 구현 참고: `web/src/components/listings/ListingCardImage.tsx:25-42, 61-69`.
   (`conventions.md` §10.2)

5. **차량정보 재배치 + 디자인 토큰 적용** — 현재 필드를 새 순서·토큰으로 옮긴다. **필드를 새로 추가하지 않는다.**
   - **"15필드"는 FR5 정의이고 실제와 일치한다**(실측 확인). 내역: 표 13행(제조사·모델·차종·연식·가격·주행거리·색상·연료·변속기·배기량·승차인원·지역·사고이력) + **옵션** + **설명** = 15. 옵션·설명은 표 행이 아니라 별도 `<section>`이라 표만 세면 13으로 보인다. 근거: `supabase/migrations/0002_listings.sql`("FR5 15필드" 블록) · `ListingDetailFields.tsx:14, 57`.
   - `seller_name`은 **FR5 15필드 밖**이다(0007이 넣은 비정규화 표시값, 값이 없으면 행 자체를 숨김). 15에 포함해 세지 말 것.
   - 재배치 시 **15개 전부 남아야 한다.** 옵션은 AC1의 ③ 옵션 섹션으로, 설명은 차량정보 섹션 하단에 둔다. **필드를 새로 만들지도, 빠뜨리지도 않는다.**
   - 토큰: `text-section`(섹션 제목) · `text-body`(값) · `text-meta`/`text-ink-muted`(라벨) · `bg-surface-raised` · `border-hairline` · `rounded-card` · `shadow-card`. 가격은 `text-price-emphasis` + **상세 large 변형(최대 30/800)**.
   - ⚠️ 현재 페이지는 `text-zinc-500`·`bg-red-50`·`bg-green-100` 같은 **원시 Tailwind 팔레트**를 쓴다. 전부 `globals.css`의 8.1 토큰으로 교체한다. **새 토큰을 만들지 않는다**(전부 이미 있다).
   (D9, `DESIGN.md` 타이포/색 토큰)

6. **404 중립 화면** — 미존재·삭제·판매완료(sold) ID로 상세 URL 직접 접근 시:
   - 문구 **그대로**: `매물을 찾을 수 없어요. 삭제됐거나 판매완료된 매물일 수 있어요.` + CTA `매물 목록으로`(→ `/search`).
   - **중립 톤이다. `danger`(빨강) 금지.** 지금 코드의 `bg-zinc-100` 손수 마크업을 `EmptyState`(또는 `ErrorState tone="neutral"`) 프리미티브로 교체한다 — **새 에러 컴포넌트를 만들지 않는다**(`web/src/components/ui/`에 이미 있다).
   - 조회 자체 실패(DB 에러)는 이와 **구분**해 `ErrorState tone="danger"`로 남긴다(현재 `bg-red-50` 손수 마크업을 교체).
   - sold 필터는 **손으로 다시 짜지 않는다** — `buyerListingsQuery`(`web/src/lib/listings.ts:29`)가 FR11 단일 출처다. sold는 `!listing`으로 합류해 자동으로 이 화면이 된다.
   (FR11, UX-DR20, `EXPERIENCE.md:60, 95-96`)

7. **sticky 문의 CTA 자리** — 문의 개시 로직은 **기존 Epic 5 코드를 그대로 재사용**한다. 여기서 만드는 건 배치뿐이다.
   - **모바일(<640px)**: 하단 고정 바(가격 + CTA 상시, top-shadow = `shadow-float`).
   - **데스크톱(≥1024px)**: 2열 배치의 **우 요약 컬럼이 `position: sticky`(top = navbar 높이)** — 긴 섹션을 스크롤해도 문의 CTA가 상시 보인다(D9 "구현 필수").
   - 분기 3가지를 **모두** 유지한다(현재 동작 = 회귀 금지선):
     - 비로그인 → `/login?redirectedFrom=...` 링크(FR58: 열람은 열려 있고 **게이트는 행동에만**)
     - 로그인 + 타인 매물 → `InquiryButton`(`openOrCreateRoom` → `/chat/{roomId}`)
     - **로그인 + 본인 매물 → "문의하기" 대신 "내 매물 관리"** 노출(현재는 버튼을 아예 숨긴다 — 이 AC가 바꾸는 지점이다. 링크 목적지는 기존 판매자 관리 화면).
   (UX-DR23, D9, `EXPERIENCE.md:95`)

8. ~~**사진 저작자 표시(`credit`)**~~ — ❌ **제외 (사용자 결정 2026-07-20). 구현하지 않는다.**
   - 이유: 과제·연습용이고 공개 배포를 유지할 의사가 없다 → CC BY 표시 의무의 발생 조건(공중 배포·송신) 자체가 없어진다. 경위·목업·되살아나는 조건은 `docs/tech-debt.md` #70에 기록.
   - **dev가 할 일: 아무것도 없다.** 갤러리 조회 `select`에 `credit`을 넣지 않아도 되고, 캡션·팝오버·출처 섹션도 만들지 않는다. **"있으면 좋을 것 같아서" 넣지 말 것**(A2·A3).
   - `listing_images.credit` 컬럼과 시드 스크립트의 저장 동작은 **그대로 둔다**(건드리지 않는다). 나중에 켤 때 비용이 낮다.
   - 번호는 비워 둔다 — 아래 AC를 당겨 올리면 Tasks·리뷰의 참조가 어긋난다.

9. **D5 반응형 무결성 + 2열/스택 전환**
   - 데스크톱 **2열**(좌 갤러리 / 우 요약·sticky) → 좁아지면 **스택 1열**. 폭 축소는 **열 수로만 흡수**한다.
   - **컴포넌트 내부 가로 배치(meta 한 줄·썸네일 스트립·"1/N" 카운터·버튼 라벨)를 세로로 접지 않는다.** 공간 부족은 `truncate`·가로 스크롤(썸네일 스트립)로만. **줄바꿈 찌그러짐·2줄로 밀리는 버튼 = 금기.**
   - 검증은 **실제 브라우저 3개 뷰포트**(1280 · 800 · 390px)에서 눈으로 + 스크린샷(AC11).
   (D5, project-context 규칙13)

10. **관리자 상세 무회귀** — `ListingDetailFields.tsx`는 **관리자 매물 상세(`web/src/app/(admin)/admin/listings/[id]/page.tsx:116`)와 공유 컴포넌트**다.
    - 관리자 화면 리스킨은 **Epic 15의 몫이며 이 스토리 밖**이다. 이 스토리의 변경으로 관리자 상세가 **깨지거나 의도치 않게 바뀌면 회귀**다.
    - 권장 방향: 공유 컴포넌트를 in-place로 뒤집는 대신 **구매자 상세 전용 레이아웃을 새로 짜고, 공유 컴포넌트는 관리자가 계속 쓰게 둔다.** 다른 길을 택했다면 이유를 Completion Notes에 남긴다.
    - 어느 길이든 **관리자 매물 상세를 실제로 열어 눈으로 확인**하고 결과를 기록한다("코드상 안 건드림"으로 갈음하지 않는다).

### 2. FR11과 사진 URL의 관계 (헷갈리기 쉬운 자리)

- **FR11이 막는 것**: sold 매물이 목록·검색·**상세**·AI 응답에 **나타나는 것**. → 이 스토리에선 `buyerListingsQuery`가 `status='on_sale'`만 통과시키므로 sold는 자동으로 404 화면(AC6)이 된다.
- **FR11이 막지 않는 것**: 판매중일 때 사진 주소를 저장해 둔 사람이 나중에 그 주소를 다시 여는 것. **명시 수용된 결정**이다(`conventions.md` §6.1, tech-debt #65 ⚪). **이걸 "고치려고" 서명 URL을 되살리지 말 것.**
- ⚠️ `conventions.md` §6이 이 스토리를 콕 집어 경고한다: *"9.5(상세 갤러리)·9.6(AI 카드)이 같은 테이블을 열 때가 정확히 규칙7이 경고한 자리다."* → **새 조회 경로를 열면 FR11 강제 지점 목록에 등록**해야 한다. 실무적으로는 **갤러리 조회를 `buyerListingsQuery`로 매물을 먼저 좁힌 뒤 그 `listing_id`로만** 하면 자동 충족된다(9.4의 `attachCoverImages`가 쓴 방식과 동일).

### 7. `credit` 표시 — ❌ 하지 않기로 결정됨 (AC8 제외)

**dev는 이 절을 읽고 "그래도 넣는 게 낫지 않나" 판단하지 말 것.** 결정은 끝났다. 아래는 나중에 되살릴 때를 위한 근거 보관이다.

실제 DB 값(실측, `listing_images.credit` jsonb):
```json
{"author":"Benespit","license":"CC BY-SA 4.0",
 "source":"https://commons.wikimedia.org/wiki/File:0_Kia_Carnival_(KA4).jpg",
 "title":"File:0 Kia Carnival (KA4).jpg"}
```

| 안 | 화면 | 장점 | 단점 |
|---|---|---|---|
| **A** | 갤러리 바로 아래 12px 회색 한 줄 (`사진 © Benespit · CC BY-SA 4.0 · Wikimedia Commons`) | 늘 보여 의무 이행이 확실. 구현 최단 | 시연 화면에 텍스트 한 줄이 항상 붙음 |
| **B** | 사진 위 작은 ⓘ 버튼 → 클릭 시 팝오버 | 평소 화면이 가장 깔끔 | 클릭해야 보임(의무 이행 해석 여지). 팝오버 상태·바깥클릭 닫기 코드가 조금 더 |
| **C** | 갤러리는 그대로, **페이지 최하단 "사진 출처" 섹션**에 N장 모아서 | 갤러리 디자인 무손상. 관행적(위키·언론사 방식) | 사진과 출처가 멀어짐 |

세 안 모두 **코드 부담은 비슷하다**(select에 컬럼 1개 + 렌더 몇 줄). 차이는 시연 화면의 인상뿐.
목업: 세션 스크래치패드 `credit-mockup.html` / `credit-mockup.png`.

**결정 (사용자, 2026-07-20): 하지 않는다.** 과제·연습용이고 공개 배포를 유지할 의사가 없다. CC BY 계열의 표시 의무는 **공중에 배포·송신할 때** 발생하므로, 배포를 내리면 의무의 발생 조건 자체가 없어진다 — **"코드로 막아야 할 버그"가 아니라 "공개 배포를 유지하는 한 지켜야 하는 것"**이었다.

⚠️ 되살아나는 조건: `bmad-encar-demo.vercel.app`(Production)이 공개로 떠 있는 동안은 의무가 실재한다. **공개 배포를 계속 유지하는 쪽으로 방향이 바뀌면** 위 3안 중 하나를 그때 고르면 된다. 경위 정본은 `docs/tech-debt.md` #70.
