## 잘된 점 (반복할 패턴)

1. **DB측 단일 방어선** — 권한·상태 무결성을 앱이 아니라 DB(RLS+CHECK+트리거)에 못박아, 웹·앱 어느 클라이언트가 붙어도 한 곳에서 보장(Winston 선치 원칙).
2. **단일 출처 미러링(drift 0)** — 드롭다운 허용값을 `constants.ts` ↔ `0002` CHECK ↔ architecture.md 바이트 단위 일치. 실 INSERT 성공 + distinct 쿼리로 교차검증.
3. **독립 2차 코드리뷰의 가치 입증** — 2-3에서 dev 자체 1차 리뷰가 놓친 "예외 시 원본 에러 화면 노출"을 새 문맥 2차 리뷰가 적발·수정(`7ddf8ac`).
4. **[Decision]은 멈춰서 사용자 승인** — price `int→bigint`, 연식 상한 하드코딩→동적 등 spec 영향 항목을 임의 결정 없이 Dnjsg님께 확인 후 반영.
5. **상태를 디스크에서 다시 읽는 설계** — 2-4 서브에이전트가 API 오류로 중단됐을 때 git/sprint-status 재확인으로 깨끗이 재시작, 중복·손상 없음.

## 어려웠던 점 / 반복된 함정

1. **RLS OR-결합의 함정(2-2)** — SELECT 정책이 `on_sale ∪ own ∪ admin`이라 "내 매물 목록" 단순 조회 시 남의 on_sale 매물까지 섞임 → 앱 쿼리에 `.eq('seller_id', user.id)` 명시 필터 필수(2-3까지 전파). *교훈: 쓰기는 RLS가 막아도 읽기는 앱이 의도한 필터를 명시해야 한다.*
2. **빌드 성공 ≠ 동작(preview 500)** — 빌드 "Ready"였으나 **Preview 환경 Supabase 환경변수 누락**으로 런타임 500. Dnjsg님이 직접 방문해 발견. 환경변수는 Production/Preview 환경별로 따로 관리됨. → 변수 Preview에도 추가 + 재배포로 해결.
3. **개발 환경 잡음(코드 무관)** — ① Turbopack `.next` 캐시 꼬임으로 전 경로 404(`.next` 삭제로 해결) ② `python3` 이름이 실제 설치본에 없어 MS Store 팝업(`python3.exe` 복사본으로 해결, [[windows-python3-alias]]).
4. **서브에이전트 1회 중단(2-4)** — API 연결 오류(FailedToOpenSocket)로 종료. 디스크 상태 확인 후 재시작해 복구(무손상).

## 핵심 교훈

- **"빌드 성공"은 "동작"이 아니다** — 환경변수는 환경별로 분리 관리되니, preview도 변수를 채우고 런타임 로그/실방문까지 확인해야 한다.
- **RLS의 OR-결합 SELECT는 항상 의심** — 목록 조회에 의도한 소유권/상태 필터를 앱에서 명시.
- **분리된 2차 리뷰는 비용값을 한다** — 작성자가 못 보는 결함을 잡는다(2-3).
