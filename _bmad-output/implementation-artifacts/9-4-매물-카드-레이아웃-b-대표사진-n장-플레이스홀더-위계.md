## Story

As a 구매자,
I want 목록 카드에서 대표사진·가격·핵심 정보를 한눈에 보길,
so that 빠르게 훑고 관심 매물을 고른다.

1. **레이아웃 B 순서** — 카드는 위에서 아래로 정확히 이 순서로 렌더된다:
   **① 사진(5:3) → ② 신뢰속성 행 슬롯 → ③ 차량명 → ④ meta(주행·연료·지역) → ⑤ 가격(최상위) → ⑥ 옵션 칩 슬롯**
   - ③ 차량명 = `text-card-title`(16/600) `text-ink-primary`, 형식 `[제조사] 모델 · 연식`
   - ④ meta = `text-meta`(13/500) `text-ink-muted`, **한 줄 가로 유지**
   - ⑤ 가격 = `text-price`(26/800) `text-price-emphasis` — **카드에서 시각적으로 가장 큰 요소**여야 한다(차량명보다 작으면 실패)
   - ②·⑥ 슬롯은 **값이 없으면 아무것도 렌더하지 않는다**(빈 높이·빈 테두리 금지). 값을 채우는 것은 Epic 10이며 지금은 항상 비어 있다.
   - ⚠️ **현재 `ListingCard.tsx`는 zinc 유틸(`border-zinc-200`·`text-zinc-500`)로 되어 있다.** 8.1 토큰(`bg-surface-raised`·`text-ink-*`·`rounded-card`·`shadow-card`)으로 **교체**한다 — 새 토큰을 만들지 않는다(전부 `globals.css`에 이미 있다).
   (FR32, UX-DR4, DESIGN.md "매물 카드 — 레이아웃 B")

2. **사진 있으면 대표 1장 + "N장" 배지** — `image_url`이 있으면 5:3 비율로 대표사진 1장을 렌더하고, 사진 위 **우하단**에 다크 pill "N장" 배지를 붙인다(`image_count` ≥ 1일 때만).
   - 크롭 = **클라 렌더 크롭**(`object-fit: cover`, 중앙 기준). 저장본을 다시 만들지 않는다(I14).
   - **`backdrop-filter`가 없어도 읽혀야 한다** — 불투명 다크 pill을 기본값으로 깔고 blur는 지원될 때만 얹는다(구형 WebView no-op, UX-DR6).
   - **배지 흰 글자 대비를 실제로 계산해 기록한다**(폴백 상태 기준, ≥4.5:1). 리뷰에서 "폴백 대비 미검증"으로 지적된 항목이다(`review-accessibility.md` [medium]).
   (FR29, UX-DR6, I14)

3. **사진 없으면 "사진 준비중" 플레이스홀더** — `image_url`이 **`null` 또는 빈 문자열(`""`)** 이면 같은 5:3 자리에 `bg-placeholder-bg` + 카메라 아웃라인 글리프 + "사진 준비중"을 렌더한다. **빈 영역·깨진 이미지 아이콘 금지**(의도적으로 보이게).
   - `image_url`이 있어도 **로드에 실패하면**(만료·삭제) 같은 플레이스홀더로 떨어진다 — `onError`로 처리. 깨진 이미지 아이콘이 그대로 노출되면 실패다.
   - ⚠️ **문구 색 대비를 계산해 확인한다.** `--ink-muted`(#676D69) on `--placeholder-bg`(#F1EFE9) = **약 4.4:1로 AA 4.5:1에 미달할 가능성이 높다**(개략 계산, 실측 필요). 미달이면 이 문구만 `text-ink-secondary`(#565F5D, 약 5.9:1)로 올린다. **다크 모드도 같이 잰다.** 토큰 값 자체는 바꾸지 않는다(다른 화면 회귀).
   (FR27, UX-DR7, `conventions.md` §4 정규화 규칙, `review-accessibility.md`)

4. **찜(♡) 버튼은 사진 밖, 위치·시각만** — 정보 영역 **우상단**에 원형 버튼(그림자)으로 자리를 잡는다. **클릭 동작·토글·상태는 Epic 10.5의 몫이며 여기서 만들지 않는다.**
   - 동작이 없으므로 `disabled` + `aria-hidden="true"` + `tabIndex={-1}` 로 두어 **스크린리더·키보드가 죽은 컨트롤을 잡지 않게** 한다.
   - 터치 타깃 **≥44×44px**(`review-accessibility.md` [high] — 아이콘 전용 소형 컨트롤 크기 미명시 지적).
   - ⚠️ **HTML 중첩 함정**: 현재 카드 전체가 `<Link>` 하나다. `<a>` 안에 `<button>`은 유효하지 않다. 카드 루트를 `<article className="relative">`로 바꾸고 `<Link>`는 **내용 영역만** 덮게 한 뒤 찜 버튼을 그 밖에 `absolute`로 놓는다.
   (FR32, DESIGN.md D3)

5. **대표 1장만 로드** — 카드는 대표 1장 외의 이미지를 요청하지 않는다. `loading="lazy"` + `decoding="async"` + `width`/`height`(또는 `aspect-ratio`)로 **레이아웃 시프트 없이** 지연 로드한다.
   (NFR7)

6. **D5 반응형 무결성** — `/search`·홈 미리보기의 카드 목록을 `ResponsiveGrid`로 바꾼다(**≥1100px 4열 · 640~1099px 2열 · <640px 1열**). 현재는 `<ul className="flex flex-col gap-2">` 1열 고정이다.
   - **카드 내부 가로 배치(meta 한 줄·신뢰속성 행·옵션 칩)는 어떤 폭에서도 세로로 접히지 않는다.** 공간 부족은 `truncate`로만 처리한다.
   - 검증은 **실제 브라우저 3개 뷰포트**(1280·800·390px)에서 눈으로 확인 + 스크린샷(AC11).
   (D5, project-context 규칙13, UX-DR21)

   **추측으로 고치지 않는다.** 분기가 정상이면 "측정했고 정상"으로 #57을 닫고, 깨져 있으면 그때 고친다.

8. **[#58] 업로더에 포맷 손실 안내를 넣는다** — `PhotoUploader.tsx`에 **애니메이션 WebP는 정지 이미지가 되고, 투명 PNG는 배경이 채워진다**는 사실을 사용자가 올리기 전에 알 수 있게 안내 문구로 노출한다. 별도 UI 작업을 새로 만들지 않고 **기존 드롭존 보조문 옆에** 붙인다.

9. **[#63] 제출 중 시작된 드래그의 drop을 막는다** — `PhotoUploader`의 `onDrop` 첫 줄에 `if (disabled) return;`.
   ⚠️ **한 줄이지만 먼저 재현한다** — 지금까지는 코드를 읽고 추론한 것이고 실제로 재현해 본 적이 없다. 재현(제출 중 드롭이 실제로 통과하는지) → 수정 → 재현 안 됨 확인, 이 순서를 Debug Log에 남긴다. 재현되지 않으면 고치지 말고 그 사실을 #63에 적고 닫는다.

10. **[#59][#47-4] 이미지 조회 규칙을 코드와 계약에 동시에 박는다**
    - **정렬**: 이미지 조회는 반드시 **`order('sort_order').order('id')`** 로 건다. `sort_order` tie-break가 정의돼 있지 않아(#47-2) 2차 키가 없으면 조회마다 순서가 달라진다. 이 정렬을 **강제하는 테스트 1건**을 남긴다(문서 약속만으로는 강제되지 않는다 — B9).
    - **대표 판별**: 카드는 **`is_cover`를 읽지 않는다.** `sort_order, id` 정렬의 **첫 행**이 대표다. 근거: `conventions.md` §10.1 "`is_cover`는 파생 결과로만 기록한다" + 시드·레거시 행은 `is_cover=false`뿐일 수 있다(#47-4 "대표 0장 허용").
    - **결정 기록**: "**대표 0장은 정상 상태이고, 읽는 쪽은 `sort_order` 첫 행을 대표로 본다**"를 `docs/conventions.md` §10에 명문화하고 #47-4를 닫는다.

    - 실제 차단은 이미 `storage.objects` 읽기 RLS가 한다(sold면 서명 자체가 `null`) — **새 앱 레벨 필터를 만들지 말 것.** 목록 갱신은 "다음 사람이 이 경로의 존재를 알게 하는 것"이 목적이다.

### 2. 데이터 흐름 (E2E — 입력에서 화면까지)

```
[서버 컴포넌트: /search · /]
  ① buyerListingsQuery → listings 행 N개 (status='on_sale'만, FR11)
  ② listing_images.select().in('listing_id', ids).order('sort_order').order('id')
        → anon/authenticated RLS가 sold 매물 사진 행을 애초에 안 준다
  ③ coverImages(rows) → Map{ listing_id → { coverPath, count } }
  ④ getSignedUrls(bucket, coverPath[]) → 배치 1회 (실패 원소는 null)
        → storage.objects 읽기 RLS가 여기서 한 번 더 sold를 막는다 (FR11 4번째 강제 지점)
  ⑤ listing에 image_url·image_count 병합 → ListingCard로 전달
[클라이언트: 브라우저]
  ⑥ <img loading="lazy">가 서명 URL을 직접 로드 (Supabase Storage → 브라우저)
  ⑦ null/''/onError → "사진 준비중" 플레이스홀더
```

**왜 이 구조인가**: 서명은 서버에서만 가능하고(`@/lib/storage`는 서버 클라이언트 사용), 카드는 클라이언트에서도 렌더되므로(`ChatAssistant`) **카드 안에서 서명하면 안 된다.** 그래서 "서버가 서명해서 `image_url`로 내려주고, 카드는 문자열만 받는다"가 §4 계약 그대로다.

### 3. ⚠️ 결정 필요 — `next/image` vs 평범한 `<img>` (착수 시 판단하고 기록할 것)

**아키텍처 문서는 "변형(thumb/card/full) = Next.js Image(web) 원본 파생"이라고 적어 뒀다.** 그런데 실제 코드를 보면 그대로 하기 어렵다:

1. **`web/next.config.ts`에 `images.remotePatterns`가 없다**(실측). `next/image`에 Supabase 호스트의 원격 URL을 넣으면 **런타임에서 바로 에러**가 난다. 쓰려면 설정 추가가 선행이다.
2. **서명 URL은 1시간마다 쿼리스트링이 바뀐다.** Next 이미지 최적화기는 URL 단위로 캐시하므로 **캐시가 거의 안 맞는다**(매 갱신마다 재최적화).
3. **FR11 관점 위험**: 최적화된 바이트가 `/_next/image` 캐시에 남으면, 원본 서명이 만료되거나 매물이 `sold`로 바뀐 뒤에도 캐시 TTL 동안 접근될 수 있다. FR11 강제를 스토리지 RLS에 몰아둔 설계(ADR-IMG-01)와 어긋난다.
4. **저장본이 이미 작다** — 9.3이 업로드 전 **긴 변 ≤1600px · WebP · q0.82**로 다운스케일한다(실측 196~205KB). 최적화기가 더 줄일 여지가 크지 않다.

**권고: 평범한 `<img loading="lazy" decoding="async">`.** 설정 추가 없이 동작하고, FR11 캐시 누수가 없고, A2(단순함 우선)에 맞는다.
**단, 이건 아키텍처 문장과 다른 선택이므로** — 어느 쪽을 택하든 **이유를 Completion Notes에 적고**, `next/image`를 택했다면 `remotePatterns` 설정과 위 3번 캐시 위험을 어떻게 처리했는지 함께 적는다. 말없이 고르지 말 것(A1).

> 참고: 9.7 시드 사진은 Wikimedia Commons **원본**을 올릴 수 있어 9.3의 리사이즈를 안 거친다. 그러면 목록에서 수 MB 다중 다운로드가 날 수 있다(NFR7). **이 스토리에서 고치지 말고**, 검증 중 실제로 큰 파일이 관찰되면 사실만 `docs/tech-debt.md`에 새 항목으로 등재해 9.7이 받게 한다.

### 4. 비용 — 알고 넘어갈 것 (고치지 말 것)

`/search`는 **limit이 없어** on_sale 전체(시드 100건+)를 한 번에 가져온다. 그래서 이미지 조회 1회 + 서명 배치 1회가 **매 요청마다 100건 규모**로 돈다.
- 서명은 배치 1회이므로 왕복은 1번이다(NFR7 "배치 발급" 충족).
- 브라우저 다운로드는 `loading="lazy"`로 화면에 들어온 것만 받는다.
- **페이지네이션은 이 스토리 범위가 아니다.** 관찰된 지연이 실제로 문제면 사실만 대장에 등재하고 넘긴다(A3 — 요청에 없는 개선 금지).
