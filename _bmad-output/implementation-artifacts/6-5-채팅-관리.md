# Story 6.5: 채팅 관리 (FR25)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자,
I want `/admin` 안에서 **전체 채팅방 목록을 보고, 한 방의 대화 내용을 열람하고, 문제 방을 삭제**하고 싶다,
so that 운영 관점에서 부적절한 대화를 확인·정리할 수 있다(FR25).

## Acceptance Criteria

1. **Given** 관리자(role=admin)가 채팅 관리 화면(`/admin/chats`)을 열면, **When** 목록을 조회하면, **Then** 당사자 여부와 무관하게 **모든 채팅방**이 보인다(참여자 한정 정책과 OR로 결합되는 `chat_rooms_select_admin`(0005, using=`is_admin()`)이 전체 행을 연다). 각 행은 매물 요약(없으면 플레이스홀더)·구매자/판매자 식별·생성일을 보여준다. (FR25 — 채팅 관리)
2. **Given** 관리자가 목록에서 한 채팅방을 열면(`/admin/chats/[roomId]`), **When** 진입하면, **Then** 그 방의 **대화 내용(메시지 전문)** 을 시간순으로 열람할 수 있다(`chat_messages_select_admin`(0005)이 당사자가 아니어도 메시지를 연다). 메시지는 보낸 사람(구매자/판매자/관리자 식별)·본문·시각으로 표시된다. 메시지가 0건이면 "메시지가 없습니다." 안내. (AC① 대화 열람)
3. **Given** 관리자가 한 채팅방을 삭제하면, **When** 삭제를 확정(window.confirm)하면, **Then** 방(`chat_rooms`)이 제거되고, 0003의 `on delete cascade`로 그 방의 메시지(`chat_messages`)도 함께 제거된다. 성공 시 목록에서 즉시 사라진다(router.refresh). (AC② 방 삭제 시 방+메시지 제거, FR25)
4. **Given** 삭제가 RLS로 막히거나(권한 없음) 그새 사라진 방이면, **When** DELETE `.select('id')`가 0행을 반환하면, **Then** 에러가 아니라 한국어로 "삭제할 수 없습니다(권한/없음)" 안내한다(6-2·6-3 액션 컴포넌트 규율 동일: 원본 에러는 콘솔, 0행은 한국어 거부, 네트워크 예외는 별도 안내).
5. **Given** 비관리자/비로그인이 `/admin/chats`·`/admin/chats/[roomId]`에 접근하면, **When** 진입을 시도하면, **Then** 기존 `(admin)` 가드(proxy 1차 + layout `requireRole(admin)` 2차)로 차단된다(별도 가드 추가 없이 `(admin)` 그룹 상속).
6. **Given** 관리자 홈(`/admin`)에서, **When** 화면을 보면, **Then** "채팅 관리"(`/admin/chats`) 진입 링크가 노출되고(회원·매물·거래 링크와 동일 형식), 하단 "채팅 관리는 추후 제공됩니다." 자리표시 문구는 제거된다(Epic 6 관리 4종 전부 구현 완료).
7. **Given** 본 스토리 구현 후, **When** 마이그레이션/RLS 상태를 확인하면, **Then** 신규 마이그레이션은 없다. 관리자 SELECT(`chat_rooms_select_admin`·`chat_messages_select_admin`)·DELETE(`chat_rooms_delete_admin`)·cascade(0003)가 모두 0005/0003에 이미 있어, 기능은 순수 web(Next.js) 화면 추가로 달성된다.

## Dev Notes

### 핵심 요지(가장 먼저 이해할 것)
- 이 스토리는 **Epic 5(문의 채팅) 화면의 "관리자 열람·삭제" 버전**이다. (user)/chat의 "내 방만, 송수신·폴링"과 달리, admin은 "전체 방, 조회 전용 열람 + 방 삭제"다.
- 차이점: ① 필터 없음(admin RLS가 전체 방·메시지를 연다). ② 송수신·폴링 없음(관리자는 대화에 끼지 않고 읽기만 → 클라 메시지 컴포넌트 불필요, 서버에서 1회 로드). ③ 방 삭제 액션 추가(6-3 ListingAdminActions 패턴). 나머지(서버 컴포넌트 구조·에러/빈목록 분기·admin 링크)는 기존 패턴을 그대로 따른다.
- **DB 작업 0건.** `chat_rooms_select_admin`·`chat_messages_select_admin`·`chat_rooms_delete_admin`(0005)와 `on delete cascade`(0003)가 이미 있다. 방을 지우면 메시지는 DB가 자동 정리하므로 클라에서 메시지를 따로 삭제하지 않는다.

### 반드시 먼저 읽을 것 (기존 패턴 — drift 금지)
- `web/src/app/(admin)/admin/listings/page.tsx` + `ListingAdminActions.tsx` — ⭐ **목록+행 삭제의 직접 본보기**. 목록 page.tsx + 클라이언트 삭제 컴포넌트(window.confirm·`.select('id')` 0행 거부·router.refresh).
- `web/src/app/(user)/chat/page.tsx` — 채팅방 목록 조회·매물 임베드 조인·null 플레이스홀더 패턴(admin은 필터를 빼고 식별만 admin식으로 바꾼다).
- `web/src/app/(user)/chat/[roomId]/page.tsx` + `ChatRoomMessages.tsx` — 방 단건 조회·메시지 표시 패턴. **단, admin 열람은 폴링·전송이 없으므로 ChatRoomMessages(클라)는 쓰지 않고 서버에서 chat_messages를 1회 조회해 정적으로 렌더**한다(조회 전용).
- `web/src/app/(admin)/admin/page.tsx` — 진입 링크 추가 위치 + 자리표시 문구 제거.

- `web/src/lib/constants.ts` — `UNITS`(price 단위) 재사용. 신규 상수 없음.

### 설계 결정(확정)
- **조회 전용 열람(폴링 없음)**: 관리자는 대화에 참여하지 않고 감독만 한다. 송수신·폴링(ChatRoomMessages)은 당사자 동선이라 admin엔 부적절·불필요. 서버 컴포넌트에서 `chat_messages`를 시간순 1회 조회해 렌더하고, `force-dynamic`으로 매 진입 시 최신 상태를 반영한다.
- **삭제는 방 단위(메시지는 cascade)**: AC②는 "방과 메시지가 제거된다". 0003 `chat_messages.room_id ... on delete cascade`가 보장하므로 클라는 방만 DELETE한다. 메시지 단건 삭제(0005 chat_messages_delete_admin)는 본 스토리 범위 밖(방 단위 관리로 충분).
- **보낸 사람 식별**: profiles엔 이메일이 없고(6-2 결정) admin anon-key로 타인 이메일 조회 불가 → 메시지 sender_id를 방의 buyer_id/seller_id와 대조해 "구매자/판매자"로 라벨링, 그 외(예: 과거 admin이 보낸 경우)는 "기타" 표기. id 축약(앞 8자)도 보조 표기로 둘 수 있다.
- **매물 null 플레이스홀더**: (user)/chat과 동일하게 매물 임베드가 null이면(삭제·sold·RLS) 상세 대신 플레이스홀더. 단 admin은 listings_select_admin으로 sold도 보이므로 보통 null은 "매물 삭제됨" 케이스다.
- **회귀 안전**: 신규 파일 3개(chats/page.tsx, chats/[roomId]/page.tsx, ChatAdminActions.tsx) + admin 홈 링크/문구 수정. 기존 (user)/(auth)/(admin) 동선·RLS·마이그레이션·constants 무변경. 비관리자는 is_admin()=false라 admin 정책으로 아무 행도 안 열리고, 화면은 (admin) 가드로 진입 자체 불가.

- **Epic 5 화면의 "관리자 열람·삭제" 버전**: (user)/chat은 "내 방만, 송수신·폴링"이지만, admin은 "전체 방(필터 없음), 조회 전용 열람 + 방 삭제"다. 그래서 ① 방 조회에 필터를 빼 admin RLS가 전체를 열게 했고, ② 폴링·전송 클라이언트(ChatRoomMessages)를 쓰지 않고 서버에서 메시지를 시간순 1회 로드해 정적 렌더했고(관리자는 대화에 끼지 않음), ③ 6-3의 행 삭제 패턴(ListingAdminActions)으로 방 삭제 액션을 붙였다.
- **방 삭제 = 방+메시지 제거**: AC②대로 방만 DELETE하면 0003의 `chat_messages.room_id on delete cascade`가 그 방의 메시지를 DB에서 자동 정리한다(클라가 메시지를 따로 안 지운다). E2E + DB 교차검증으로 cascade 작동을 확인(삭제 방의 메시지 0건).

- **회귀 안전**: 신규 파일 3개 + admin 홈 링크/문구 수정 1개. 기존 (user)/(auth)/(admin 회원·매물·거래) 동선·RLS·마이그레이션·constants 무변경. 비관리자는 is_admin()=false라 admin 정책으로 아무 행도 안 열리고, 화면은 (admin) layout 가드로 진입 자체 불가.
