## Story

As a 구매자,
I want 어떤 질문을 하든 의도에 맞는 경로로 처리되고 자연어 설명 + 매물 카드로 답을 받고 싶다,
so that 한 입력창에서 자연스럽게 매물을 검색한다(구조형이든·의미형이든·매물과 무관한 질문이든 적절히 응대).

> **이 스토리의 본질 — "지휘자(라우터)를 꽂아 흩어진 노드를 한 흐름으로 묶는다".** 4.3이 경로 A(정확한 조건 SQL), 4.4가 경로 B(의미형 임베딩 검색)를 **각각 호출 가능한 함수**로 만들어 뒀다. 하지만 `/ai/search`는 지금 **모든 질의를 무조건 경로 A로** 보낸다. 4.5는 그 앞단에 **라우터(router_node)** 를 꽂아 질문 의도를 (A)구조형 / (B)질적·의미형 / (C)매물 무관 셋으로 분류하고, A→sql_rag_node·B→doc_rag_node·C→가드(정중한 거절)로 **갈래를 나눈다**(FR13·FR16). 그리고 결과를 **공통 계약 `{answer, listings[]}`** 으로 다듬는 답변 조립(answer_node)을 통과시킨다(FR17). 이로써 4.1~4.4의 조각들이 architecture가 그린 **`질의 → 라우터 → (A SQL / B 문서RAG / C 가드) → answer → {answer, listings[]}`** 단일 파이프라인으로 완성된다.
>
> *라우터(router): 들어온 질문의 의도를 보고 어느 처리 경로로 보낼지 결정하는 분기 장치.*
> *가드(guard): 처리하면 안 되는(매물과 무관한) 질문을 막아 정중히 돌려보내는 안전장치.*
> *StateGraph(LangGraph): 노드(처리 단위)를 간선(흐름)으로 이어 만든 상태 기반 처리 그래프.*

## Acceptance Criteria

1. **(AC1 — 라우터 3분류, FR13)** `app/graph/router_node.py`의 라우터가 질의를 받아 **정확히 셋 중 하나**로 분류한다 — `"A"`(구조형: 가격·차종·연식 등 명시 조건) / `"B"`(질적·의미형: "패밀리카로 무난한 거" 류) / `"C"`(매물 무관: 날씨·상식·잡담 등). 분류는 **LLM 구조화 출력(Pydantic 스키마, OI2)** 으로 하되, **결정론적 안전장치**로 출력이 A/B/C 외 값이면 안전한 기본값(B 또는 C)으로 보정한다(환각 방어). [FR13, research §8 #2]
2. **(AC2 — 경로 C 가드/거절, FR16·CM1)** 라우터가 `"C"`(매물 무관)로 분류한 질의는 `app/graph/guard_node.py`(또는 answer 분기)가 **정중히 거절하고 매물 검색으로 유도**하는 한국어 `answer`를 반환하며 `listings: []`를 돌려준다. **순수 상식 Q&A는 제공하지 않는다**(예: "오늘 날씨?" → "저는 중고차 매물 검색을 도와드려요. 원하시는 차의 조건이나 용도를 말씀해 주세요."). [FR16, CM1]
3. **(AC3 — 전체 그래프 조립 + /ai/search 연결)** `app/graph/`에 **LangGraph `StateGraph`** (또는 동등한 명시적 오케스트레이션 함수)로 `라우터 → (A: sql_rag_node / B: doc_rag_node / C: guard) → answer_node` 흐름을 조립하고, `routers/ai.py`의 `/ai/search`가 **이 그래프를 호출**하도록 바꾼다(더는 sql_rag_node를 직접 호출하지 않는다). 인증(`get_current_user`)·응답 계약(`{answer, listings[]}`)·에러 포맷·`SqlGuardError→400` 처리는 **기존 동작을 보존**한다(회귀 0). [architecture.md#AI 데이터 흐름]
4. **(AC4 — 답변 조립 + 0건 안내, FR17)** `app/graph/answer_node.py`가 각 경로 결과를 **공통 계약 `{answer, listings[]}`** 으로 최종 정리한다. 검색 결과가 있으면 자연어 설명 + 매물 카드, **0건이면 빈 목록 대신 조건 완화/재질문 유도 안내**를 반환한다(FR17). (경로 A·B 노드가 이미 만든 answer를 살리되, 그래프 레벨에서 계약 일관성을 보장.)
5. **(AC5 — OI5 데모 질의셋 작성)** **OI5 데모 질의셋**(구조형 / 질적·의미형 / 회색지대 / 매물 무관 + 차형 용어 매핑 예)을 문서로 작성해 라우터·경로의 동작 기대를 명세한다(예: `api/docs/ai-demo-queries.md` 또는 `docs/` 하위). 각 질의에 기대 분류(A/B/C)와 기대 결과 요지를 적는다. [epics.md AC, OI5]
6. **(AC6 — 결정론적 검증 + 라이브)** 라우터 분류·가드 거절·그래프 분기·0건 FR17을 **LLM/DB 모킹 단위 테스트**로 네트워크 없이 검증한다(라우터가 A→sql 경로, B→doc 경로, C→가드로 가는지 / 환각 라벨 보정 / C는 listings=[] + 유도 문구). 추가로 **실 LLM·실 DB 라이브**로 OI5 질의셋 대표 4건(A/B/C/회색) 이상이 기대 경로로 흐르고 `/ai/search` HTTP가 `{answer, listings[]}`를 돌려줌을 눈으로 확인한다.

## Dev Notes

### ⚠️ 결정적 함정 — 반드시 숙지

**함정 #1 — `/ai/search`는 지금 sql_rag_node를 "직접" 호출한다(라우터 미경유).**
`routers/ai.py:28`이 `result = sql_rag_node(req.query)`로 **모든 질의를 경로 A로** 보낸다(4.3이 의도적으로 그렇게 둠). 4.5의 핵심은 이 한 줄을 **그래프 호출로 교체**하는 것이다. 교체할 때 **기존 에러 처리(`SqlGuardError → 400`, 그 외 → 전역 500)** 를 반드시 보존해야 한다 — 경로 A가 그래프 안에서 실행돼도 `SqlGuardError`가 똑같이 `/ai/search`까지 전파돼 400으로 잡혀야 회귀가 없다. (그래프가 예외를 삼키지 않도록 주의: 경로 노드 예외를 그래프가 감싸 버리면 400이 500으로 바뀐다.)
[Source: api/app/routers/ai.py(24-37); story 4.3 범위밖(28-33)]

**함정 #2 — LLM 분류 출력을 그대로 믿지 마라(결정론적 보정 필수).**
라우터도 LLM이라 환각·형식이탈 가능성이 있다. 구조화 출력(`with_structured_output` + `Literal["A","B","C"]`)으로 1차 방어하되, **그래도 A/B/C 외 값/파싱 실패가 나면 코드가 안전 기본값으로 보정**한다(sql_guard·doc_rag의 fail-loud/fallback 철학과 동일). 보정 정책: 매물 신호가 조금이라도 있으면 **B**(빈손보다 추천), 명백 무관이면 C. 단 **키 부재(GEMINI_API_KEY)는 보정이 아니라 `require()` fail-loud**(조용한 오답 금지).
[Source: research §8 #2(구조화 출력); 4-3 sql_guard fail-closed 사상; 4-4 fail-loud 패턴]

**함정 #3 — answer_node에서 LLM으로 답을 다시 쓰지 마라(과잉구현).**
경로 A·B 노드는 이미 충분한 한국어 answer를 만든다(4.3 `_ANSWER_FOUND/_EMPTY`, 4.4 동일). answer_node의 역할은 **새 답 생성이 아니라 "공통 계약 보장 + 0건 fallback 통일"** 이다. 여기서 또 LLM을 호출하면 비용·지연·비결정성만 늘고 데모 가치는 없다. FR17(0건 안내)이 모든 경로에서 보장되는지만 한 곳에서 점검한다.

### 설계 결정 — OI2: 라우터 구조 (확정 제안: **LLM 구조화 출력 분류 + 결정론 보정**)

- **채택: 단일 LLM 분류 노드 + Pydantic `Literal["A","B","C"]` 구조화 출력.** 질의 1건을 한 번의 LLM 호출로 A/B/C 판정. 가볍고(1콜) 결정적 보정으로 안전. architecture OI2가 말한 "라우터 구조화 출력(Pydantic 스키마)" 그대로.
- **미채택: 규칙기반(키워드 매칭) 라우터** — "3천만원"·"SUV" 같은 키워드로 A/B/C 가르기. 빠르지만 자연어 변형에 취약("패밀리카"·"무난한"을 다 못 잡음)하고 회색지대 처리가 빈약 → 데모 품질 저하. (보정 fallback에서 보조적으로 신호로 쓸 수는 있음.)

### 라우터 프롬프트 설계 (요지)

```
너는 중고차 매물 검색 어시스턴트의 "의도 분류기"다. 사용자 질의를 아래 셋 중 하나로만 분류한다.
- A (구조형): 가격·차종·연식·색상·지역·주행거리 등 명시적 조건으로 매물을 거르는 질의.
    예) "3천만원 이하 흰색 SUV", "2020년 이후 제네시스", "10만km 미만 디젤"
- B (질적·의미형): 정확한 조건 대신 용도·느낌·추천을 묻는 질의.
    예) "패밀리카로 무난한 거", "초보운전자에게 좋은 차", "연비 좋은 전기차 추천"
- C (매물 무관): 중고차 매물 검색과 관계없는 잡담·상식·다른 주제.
    예) "오늘 날씨 어때?", "파이썬 코드 짜줘", "안녕"
출력은 route 필드에 "A"/"B"/"C" 중 하나만. 애매하면(차 얘기긴 한데 조건이 흐릿) B로 분류한다.
```
- 허용값(차종·연료 등)은 sql_rag_node 프롬프트의 단일출처를 재인용(중복 정의 금지 — drift 방지). 라우터는 "분류"만 하므로 전체 스키마까진 불필요, 카테고리 정의·예시 위주.
- temperature=0(재현성), 구조화 출력으로 형식 강제.
