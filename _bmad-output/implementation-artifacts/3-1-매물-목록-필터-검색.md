# Story 3.1: 매물 목록 + 필터 검색

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 구매자,
I want 키워드/필터(가격·연식·차종·색상·연료·변속기·지역)로 매물을 검색하고 싶다,
so that 원하는 조건의 매물을 빠르게 찾는다.

## Acceptance Criteria

1. **(필터·키워드 검색 → on_sale만)** 매물 목록 화면에서 필터(가격·연식·차종·색상·연료·변속기·지역) 또는 키워드를 적용하면 **조건에 맞는 `on_sale` 매물만** 반환된다. `status='sold'` 매물은 어떤 필터 조합으로도 노출되지 않는다(FR11 — listings SELECT RLS가 구매자에게 on_sale만 통과시킴). [FR9·FR11]
   - 필터 7종 = 가격(범위 min/max)·연식(범위 min/max)·차종(`body_type`)·색상(`color`)·연료(`fuel`)·변속기(`transmission`)·지역(`region`). 드롭다운 옵션은 **`LISTING_OPTIONS` 단일 출처**를 그대로 쓴다(재발명·하드코딩 금지). [Source: constants.ts:47-63]
   - 키워드는 `model`(모델명) 부분일치(대소문자 무시)로 검색한다. (제조사는 드롭다운 후보였으나 본 스토리 필터셋은 AC에 명시된 7종 + 키워드로 한정 — 제조사 필터는 범위 컷.)
2. **(빈 상태 안내)** 조건에 맞는 매물이 없으면 한국어 빈 상태 안내("조건에 맞는 매물이 없습니다. 필터를 완화해 보세요." 류)를 표시한다. 에러(조회 실패)와 "0건"은 **구분**해 표시한다(조회 실패는 별도 한국어 안내 — 2-2/2-3 패턴). [FR9]
3. **(단위 표시 규칙)** 가격·주행거리는 conventions §3 단위 규칙대로 천단위 구분 + 단위로 표시한다(예: `29,800,000원` / `103,000km`). 연식은 `2021년` 형태. [Source: conventions.md §3, constants.ts UNITS]
4. **(역할·접근)** 매물 탐색은 로그인 사용자가 접근한다. 구매자가 기본 사용자다. 구매자/판매자 모두 `on_sale` 공개 매물을 볼 수 있다(RLS상 on_sale은 모두에게 공개). 비로그인은 `/login`으로 보낸다(보호 경로 추가). [FR3 기반]
5. **(목록 항목 표시)** 각 매물 카드는 `ListingCard` 필드 계약대로 핵심 요약을 보여준다 — `manufacturer·model·year·price·mileage·region`(사진 없음). 카드 클릭 시 상세로 가는 진입점(링크)을 둔다(상세 화면 자체 구현은 **3-2**). [Source: conventions.md §4 ListingCard]

> **범위 주의(반드시 준수):** 이 스토리는 **구매자용 매물 목록 + 필터 검색(FR9)** 까지다.
> - **매물 상세 화면(FR10, `listings/[id]`)은 3-2** — 카드의 상세 링크 `href`는 만들되, 상세 페이지 자체는 만들지 않는다(혹은 최소 placeholder 금지 — 링크만 두고 3-2가 채움). 링크가 404가 되지 않도록 하려면 상세 라우트를 **만들지 말고** 카드 링크는 두되, 자체 테스트에서는 목록·필터·빈상태까지만 검증한다(상세 진입 클릭은 3-2 검증).
> - **FR11 단일 규칙 검증 전담은 3-3** — 본 스토리에서도 sold 비노출은 당연히 성립해야 하나(AC1), 별도 단일규칙 회귀 스위트는 3-3 소관.
> - **AI 검색(Epic 4)·채팅(Epic 5)·관리자 전체매물(Epic 6)** 미구현.
> - **마이그레이션·DB 스키마는 건드리지 않는다**(2-1에서 listings + RLS 라이브 완료). `embedding`은 NULL 그대로(Epic 4 backfill).

## Dev Notes

### ⚠️ 이건 당신이 아는 Next.js가 아니다 (web/AGENTS.md 경고)
- `web/AGENTS.md`: "코드 작성 전 `node_modules/next/dist/docs/`의 관련 가이드를 읽으라." Next.js **16.2.x**.
- **`searchParams`는 Promise** — 서버 컴포넌트에서 `const sp = await searchParams`로 await 후 사용한다(동적 라우트 `params`와 동일 규약, 2-3에서 확인). 가이드 확인 후 정확한 시그니처 사용. [Source: web/AGENTS.md, node_modules/next/dist/docs/]
- 서버/클라이언트 컴포넌트 경계: 필터 폼(상호작용·`router.push`)은 `'use client'`, 데이터 조회·렌더는 서버 컴포넌트.

### 2-1에서 확정된 DB·RLS 계약 (재발명·재설계 금지 — DB 안 건드림)
- `listings` 테이블·RLS·CHECK·트리거는 **2-1에서 라이브 Supabase에 적용 완료**. 본 스토리는 마이그레이션을 만들거나 고치지 않는다.
- **SELECT RLS = `on_sale ∪ own ∪ admin` OR 결합.** 구매자는 own·admin이 아니므로 **on_sale만** 통과 → sold가 어느 경로로도 안 보임(FR11 핵심, AC1). [Source: 0002_listings.sql:88-101]
  - 단, **판매자**가 `/search`에 들어오면 본인 sold 매물이 RLS `own` 정책으로 통과될 수 있다 → 그래서 앱 쿼리에 **`.eq('status','on_sale')` 명시**로 "구매자 관점"을 강제한다(Task 3.2). 이게 FR11 단일 규칙을 앱·RLS 양쪽에서 못박는 방식.
- **단위 규칙(conventions §3):** `price`=원(KRW) 정수, `mileage`=km 정수. 저장 단위 그대로 표시 시 `toLocaleString('ko-KR')` + `UNITS.price`/`UNITS.mileage`. [Source: conventions.md §3, constants.ts:32-37]
