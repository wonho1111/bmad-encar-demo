# Story 3.2: 매물 상세 조회

Status: done

## Story

As a 구매자,
I want 매물 상세를 보고 싶다,
so that 구매 판단에 필요한 정보(설명·옵션·상태 포함)를 확인한다.

## Acceptance Criteria

1. **(상세 표시 — 15필드 + 설명·옵션·상태)** 목록(3-1 `ListingCard`)에서 매물을 선택하면(`/listings/[id]`), 상세 화면에 **FR5 15필드 전부**가 표시된다 — `manufacturer·model·body_type·year·price·mileage·color·fuel·transmission·displacement·seats·region·accident_free·options·description`. 설명(`description`)·옵션(`options`)·상태(판매중/판매완료 라벨)도 보인다. **사진은 없다**(서비스 전체가 사진 미사용). [FR10, Source: epics.md#Story-3.2, 0002_listings.sql:34-57]
   - 단위·표시는 conventions §3 규칙대로: `price` → `29,800,000원`, `mileage` → `103,000km`, `displacement` → `1,998cc`, `year` → `2021년`. 천단위 콤마(`toLocaleString('ko-KR')`) + `UNITS`. [Source: conventions.md §3, constants.ts:32-37]
   - `accident_free`는 한국어("무사고"/"사고이력 있음")로, `options`(text[])는 빈 배열이면 "옵션 정보 없음" 류 안내, `description`이 비면 "설명 없음" 류 안내로 표시한다(null/빈값 방어).
2. **(sold 매물은 구매자에게 비노출 — FR11)** `status='sold'` 매물 id로 상세 접근을 시도하면 **구매자에게는 노출되지 않는다**. 상세를 못 찾는 경우(존재하지 않는 id·sold·접근 권한 없음)는 한국어 "찾을 수 없음" 안내를 띄운다(목록으로 돌아가는 링크 포함). [FR11, Source: epics.md#Story-3.2, conventions.md §6]
   - **이중 방어**: RLS는 구매자에게 `on_sale`만 통과시키지만(`listings_select_on_sale ∪ own ∪ admin` OR 결합), **판매자**가 본인 sold 매물 상세 URL로 들어오면 RLS `own` 정책으로 통과될 수 있다 → 앱 쿼리에 **`.eq('status','on_sale')` 명시**로 "구매자 관점(판매중만)"을 강제한다(3-1 `/search`와 동일 방식). 이게 FR11 단일 규칙을 앱·RLS 양쪽에서 못박는다. [Source: search/page.tsx:11-14·95-98, 0002_listings.sql:90-101]
3. **(역할·접근)** 매물 상세는 로그인 사용자(구매자가 기본, 판매자도 가능)가 접근한다. 비로그인은 `/login`으로 보낸다 — `proxy.ts`의 `PROTECTED_PREFIXES`에 **`/listings`를 추가**한다(현재 미보호 상태로, 3-1이 만든 카드 링크 `/listings/[id]`가 비로그인에게 1차 차단되지 않는 갭). [FR3 기반, Source: proxy.ts:18]
4. **(에러 vs 못 찾음 구분)** 조회 자체가 실패(네트워크·RLS·DB 오류)하면 별도 한국어 에러 안내("불러오지 못했습니다" 류, 빨강)를, 결과가 0건(없음·sold·권한 없음)이면 "매물을 찾을 수 없습니다" 안내를 띄워 **둘을 구분**한다(2-3 edit·3-1 패턴). 원본 에러는 `console.error`만. [FR10, Source: sell/[id]/edit/page.tsx:43-64, conventions.md §4]

> **범위 주의(반드시 준수):**
> - 이 스토리는 **구매자용 매물 상세(FR10)** = `(user)/listings/[id]/page.tsx` 한 화면이다.
> - **FR11 단일 규칙 회귀 스위트 전담은 3-3** — 본 스토리도 상세 경로에서 sold 비노출을 보장해야 하나(AC2), 목록·필터·상세를 한꺼번에 검증하는 회귀 스위트는 3-3 소관.
> - **문의하기(채팅, Epic 5)·AI 검색(Epic 4)** 진입 버튼은 만들지 않는다(미구현 기능 링크 금지). 상세 화면은 "정보 표시 + 목록으로 돌아가기"까지.
> - **마이그레이션·DB 스키마는 건드리지 않는다**(2-1에서 listings + RLS 라이브 완료). `embedding`은 NULL 그대로(Epic 4 backfill).
> - 판매자 본인 매물 관리(수정·삭제·구매완료)는 `/sell`(2-3·2-4) 소관 — 상세 화면에 그 버튼을 두지 않는다(구매자 관점 화면).

## Dev Notes

### ⚠️ 이건 당신이 아는 Next.js가 아니다 (web/AGENTS.md 경고)
- `web/AGENTS.md`: "코드 작성 전 `node_modules/next/dist/docs/`의 관련 가이드를 읽으라." Next.js **16.2.x**.
- **동적 라우트 `params`는 Promise** — 서버 컴포넌트에서 `const { id } = await params`로 await 후 사용한다(2-3 edit·3-1에서 확인된 규약). 가이드 확인 후 정확한 시그니처 사용. [Source: sell/[id]/edit/page.tsx:21-27, web/AGENTS.md]
- 상세는 **데이터 조회·렌더만** 하므로 전부 **서버 컴포넌트**로 충분하다('use client' 불필요 — 폼·상호작용 없음). 3-1의 `SearchFilters`(클라이언트)와 달리 추가 클라이언트 컴포넌트 없음.

### 2-1에서 확정된 DB·RLS 계약 (재발명·재설계 금지 — DB 안 건드림)
- `listings` 테이블·RLS·CHECK·트리거는 **2-1에서 라이브 Supabase에 적용 완료**. 본 스토리는 마이그레이션을 만들거나 고치지 않는다.
- **SELECT RLS = `on_sale ∪ own ∪ admin` OR 결합.** 구매자는 own·admin이 아니므로 **on_sale만** 통과 → sold가 어느 경로로도 안 보임(FR11 핵심, AC2). [Source: 0002_listings.sql:90-101]
  - 단, **판매자**가 본인 sold 매물의 `/listings/[id]`에 들어오면 RLS `own` 정책으로 통과될 수 있다 → 그래서 앱 쿼리에 **`.eq('status','on_sale')` 명시**로 "구매자 관점"을 강제한다(Task 2.2). 3-1 `/search`가 쓴 것과 동일한 이중 방어. [Source: search/page.tsx:11-14·95-98]
- **15필드 = FR5 전부**(사진 제외): manufacturer·model·body_type·year·price·mileage·color·fuel·transmission·displacement·seats·region·accident_free·options·description. 컬럼 정의·CHECK는 0002. [Source: 0002_listings.sql:34-57]
- **단위 규칙(conventions §3):** `price`=원(KRW) bigint, `mileage`=km int, `displacement`=cc int, `year`=4자리 연도. 저장 단위 그대로 표시 시 `toLocaleString('ko-KR')` + `UNITS`. [Source: conventions.md §3, constants.ts:32-37]
- **null/빈값 주의:** `options`는 `text[] default '{}'`(빈 배열 가능), `description`은 nullable(null 가능). 둘 다 빈값 방어 안내 필요(AC1). [Source: 0002_listings.sql:56-57]
