# Story 7.3: 판매자 — 매물 등록 (재현 FR5)

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 판매자(모바일),
I want 앱에서 15필드 폼을 채워 매물을 등록하고 싶다,
so that 폰으로 차를 내놓을 수 있다(FR5 재현). 사진은 없다(업로드 화면 없음).

## Acceptance Criteria

1. **(AC1 — 매물 등록, FR5 재현)** 판매자로 로그인한 앱에서 진입하는 등록 화면에서 **15필드 폼**(제조사·모델·차종·연식·가격·주행거리·색상·연료·변속기·배기량·인승·지역·무사고·옵션·설명)을 채워 제출하면 `listings` 행이 **`status='on_sale'`로 생성**된다(관리자 승인 없이 즉시). 사진 없음(업로드 위젯 없음). 생성 성공 시 한국어 성공 안내 + 폼 초기화.
   - *근거*: epics.md 7.3 "15필드 폼을 채워 제출하면 listings 행이 on_sale로 생성된다". web `SellForm.tsx`의 INSERT(create 모드) 동작 이식. 사진 제외(0002_listings 주석 "사진 제외 — Supabase Storage 미사용").

2. **(AC2 — 한국어 검증 거절, CHECK 위반)** **CHECK 위반 값**(목록 밖 값·범위 위반·빈 필수값)으로 제출하면 **한국어 검증 오류로 거절**된다. 클라이언트 검증(드롭다운 단일출처·범위·정수)과 DB CHECK 둘 다로 막는다(이중 방어). 고정 6필드(제조사·차종·색상·연료·변속기·지역)는 드롭다운이라 목록 밖 값 선택 자체가 불가. 숫자(연식·가격·주행·배기량·인승)는 정수·범위를 클라이언트가 먼저 거르고, 그래도 DB까지 닿은 위반은 한국어로 변환해 노출(원본 코드·메시지는 콘솔/로그에만).
   - *근거*: epics.md 7.3 "CHECK 위반 값으로 제출하면 한국어 검증 오류로 거절". web `SellForm.tsx`의 `validateAndBuild`(클라이언트) + `toKoreanError`(DB 23514/42501 → 한국어) 이식.

3. **(AC3 — 소유권, FR6 RLS 공유)** 생성된 행의 `seller_id` = 현재 로그인 사용자다. 등록 시 `seller_id`를 현재 `user.id`로 명시하고, 위조는 DB RLS(`listings_insert_own`: `auth.uid()=seller_id`, 마이그 0002)가 차단한다(web과 동일 Supabase 프로젝트·RLS). RLS 거부(42501)는 한국어 안내로 변환.
   - *근거*: 0002_listings.sql `listings_insert_own` 정책. web `SellForm.tsx`가 `insert({...payload, seller_id: user.id})`로 명시하는 패턴 이식.

4. **(AC4 — 역할 가드)** **판매자(seller)만** 등록 화면에 진입한다. buyer·admin은 진입 차단(admin은 7.1 AuthGate에서 이미 모바일 제외). 홈(`home_screen.dart`)에서 판매자에게만 "매물 등록" 진입 동선을 노출하고, 진입점/화면 진입 시 역할을 한 번 더 확인한다(이중 방어).
   - *근거*: 7.1 `user_role.dart`(역할 enum)·`AuthGate`(admin 차단). web `(user)/sell/layout.tsx requireRole(seller)` 선례를 모바일 역할 가드로 이식. nav-ia §2 판매자 전용(등록·관리)은 7.3·7.4 소관.

5. **(AC5 — 검증·테스트)** `flutter analyze`(0 error)·`flutter test`(검증/페이로드 빌드 순수 함수 단위 테스트)·`flutter build web --dart-define-from-file=.env.json`이 통과한다. 가능하면 판매자 토큰으로 `listings` INSERT→on_sale 생성, CHECK 위반→거절을 live Supabase HTTP(REST) 스모크로 실증(생성한 테스트 행은 정리, 못 하면 표기). 브라우저 픽셀 렌더는 시스템 라이브러리 미비 시 빌드+테스트로 갈음(그 사실 표기).
   - *근거*: 7.2 테스트 전략(실폰·에뮬 없음 → build web + test + live HTTP 스모크). 프롬프트 §검증.

### 단일 출처 / 계약 (drift 금지)

- **15필드·CHECK·범위의 출처** = `supabase/migrations/0002_listings.sql`. web `src/lib/constants.ts`(LISTING_OPTIONS·LISTING_RANGES)·web `SellForm.tsx`가 이 출처를 미러. **Flutter도 같은 값을 바이트 단위로 복사**한다. 어긋나면 폼에서 고른 값이 DB CHECK에 걸려 거절된다.
  - 제조사(15): 현대,기아,제네시스,쉐보레,르노코리아,KG모빌리티,BMW,벤츠,아우디,폭스바겐,토요타,혼다,렉서스,테슬라,기타
  - 차종(12)·색상(9)·연료(5)·변속기(2)·지역(17): 이미 `ListingOptions`에 존재(재사용).
  - year: 1990 ~ (올해+1), seats: 2~11, price/mileage/displacement: ≥0, accident_free: 기본 true, options: text[]·기본 빈배열, description: nullable, status: 'on_sale'.
- **단위**: price=원(KRW)·mileage=km·displacement=cc(0002 주석·docs/conventions §3). 저장은 정수.
- **status 'on_sale'은 DB 기본값**이지만 web과 동일하게 페이로드에 명시(의도 명시·회귀 안전).

- 검증 순수 함수 분리 패턴: 7.2 `listing_filters.dart`(parseIntFilter·pickOption)가 선례. 같은 스타일로 `validateAndBuildListing`을 순수 함수로 빼 단위 테스트.
- Riverpod 3 `Notifier`: 7.2 `SearchController`(NotifierProvider) 패턴 그대로.
- 화면 폼 위젯: `signup_screen.dart`(ConsumerStatefulWidget + SingleChildScrollView + 에러/성공 텍스트 + 로딩 중 비활성) 패턴 그대로.
- 숫자 표시 콤마: `lib/core/format/number_format.dart`(필요 시).
- 한국어 에러 변환 선례: `auth_errors.dart`(code/message 매칭). DB 에러는 `PostgrestException.code` 매칭.
