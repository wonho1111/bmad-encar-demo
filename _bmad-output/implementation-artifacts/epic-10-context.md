## Goal

매물 카드·상세가 신뢰도 있게 읽히도록 만든다. 무사고·1인소유·비흡연 같은 "신뢰속성"을 옵션(장비)과 별개 데이터로 분리하고, 전부 판매자 자기신고임을 밝히는 면책 라벨을 뱃지와 한 몸으로 붙인다. 옵션은 통제어휘로 표준화하고 희소한 옵션을 우선 노출한다. 로그인 사용자는 매물을 찜해 모아볼 수 있고, 상세에는 판매자를 가볍게 신뢰할 수 있는 경량 정보 섹션을 둔다. Epic 9가 세운 상세 페이지 골격(신뢰→차량→옵션→판매자 순서의 빈 섹션)을 이 에픽이 채운다.

## Stories

- Story 10.1: 신뢰속성 스키마 + 락스텝 반영
- Story 10.2: 신뢰 뱃지 표시 + 면책 라벨
- Story 10.3: 옵션 통제어휘 + 우선순위 상수 + 저장구조 정비
- Story 10.4: 옵션 하이브리드 피커 (등록)
- Story 10.5: 찜 (wishlist)
- Story 10.6: 판매자 정보 섹션
- Story 10.7: 신뢰속성·옵션 통합 검증 (SM-C)

## Requirements & Constraints

- 신뢰속성(무사고·1인소유·비흡연)은 옵션(장비)과 분리해 별도 컬럼으로 관리·표시한다. 매물 카드는 희소성 있는 옵션을 우선 노출한다.
- 찜: 로그인 사용자가 매물을 찜(토글)하고 "찜한 매물" 목록에 도달할 수 있다. 재방문을 유도하는 핵심 고리다. 평판·비교표는 범위 밖.
- 판매자 정보: 상세에 닉네임 + 가입 시점 + "이 판매자의 다른 매물 N건"만 노출하는 경량 섹션. 평판 점수·응답률·인증 배지는 데이터가 없어 범위 밖이며, 없는 걸 있는 것처럼 보이면 신뢰속성 정직성 원칙과 충돌하므로 만들지 않는다.
- CM-C(성공 지표): 신뢰속성을 "검증됨"으로 오도하지 않는다 — 면책 라벨은 옵션이 아니라 필수 요구사항이다. 근거: 객관 검증 인프라(보험이력·성능기록부) 연동이 이 데모엔 없으므로, 무사고·1인소유·비흡연 셋을 전부 "판매자 자기신고" 티어로 통일하고 정직하게 면책 표시한다(가짜 검증 UI/API 금지). "무사고"만 초록 신뢰 뱃지, "단순교환"·"사고"는 초록이 아닌 가치중립 상태칩으로 구분한다.
- 옵션은 정규화 테이블을 새로 만들지 않고 기존 `text[]` 저장을 유지한다(과설계 회피). 대신 표준 옵션명 상수(통제어휘)로 쓰기를 검증하고, 옵션명→우선순위 점수 상수로 희소 옵션을 가려낸다. 카드는 우선순위 상위 3~4개만(희소 옵션이 없으면 보편 옵션으로 대체), 상세는 카테고리별 전량을 보여준다.
- 기존 옵션 저장 관련 기술부채(`SellForm.tsx`가 옵션을 쉼표로 join/split해 값에 쉼표가 있으면 데이터가 손실되는 문제)를 이 에픽에서 배열/정규화 저장으로 정비한다.
- 등록 폼의 옵션 선택은 인기 옵션 8개를 칩으로 우선 노출해 대부분의 입력을 여기서 끝내고, 희소 옵션은 "전체 옵션 더보기"(카테고리 아코디언 + 검색창)로 구조적으로 찾게 한다.
- 찜은 서버(DB) 단일 저장이며 실시간 동기화 대상이 아니다(재사용 우선 원칙 — 찜 개수를 반영한 인기 신호도 이번 범위 밖, 지금 인기 정렬은 조회수만 사용).

## Technical Decisions

- 마이그레이션 파일 번호는 여기에 적지 않는다 — **착수 시점에 `supabase/migrations/` 파일 목록을 실제로 보고 다음 빈 번호로 정한다.** 계획 문서에 적혔던 번호(신뢰속성용·wishlists용 각각 예정됐던 번호)는 다른 스토리가 먼저 그 번호를 써버려 이미 소진됐다 — 문서에 적힌 숫자를 그대로 믿고 파일을 만들면 충돌한다.
- 신뢰속성 스키마: `accident_status`는 `text + CHECK('무사고'·'단순교환'·'사고')`(네이티브 enum이 아니라 기존 `body_type`/`color`/`region`과 같은 관례). `is_single_owner`·`is_non_smoker`는 bool. 셋 다 **nullable**이며, NULL은 "미입력"을 뜻하는 제3상태다(초록 뱃지도 사고 표시도 아님). 기존 100건은 NULL로 유지하고 backfill하지 않는다. 기존 `accident_free` 값은 승격 시 보존한다(드롭 금지, additive만).
- 이 스키마 변경은 `sql_guard` 화이트리스트, AI 프롬프트 스키마, ListingCard 계약(`docs/conventions.md` §4/§4.1), web/app 매퍼까지 **락스텝**으로 동시에 반영해야 drift가 안 생긴다. 같은 락스텝 작업을 도는 김에 카드 meta에서 빠져 있던 연료(fuel) 필드 누락도 함께 닫는다(대장 이월 #67).
- RLS(행 단위 접근 제어)는 신규 컬럼에 별도 정책을 만들지 않고 기존 `listings` 정책을 상속한다.
- `wishlists(user_id, listing_id, created_at, PK(user_id, listing_id))` + 본인 전용 RLS. 인기 신호(찜 개수 집계)는 이번 범위 밖으로 미루되, `wishlists`가 그 COUNT의 원천이 될 수 있게 스키마를 남겨둔다.
- ListingCard wire 계약(단일 출처 `docs/conventions.md` §4)에 `accident_status`·`is_single_owner`·`is_non_smoker` 필드는 이미 nullable로 선언돼 있고 이 에픽이 실제 값을 채우는 주체다. 값을 채울 땐 §4.1 체크리스트(문서·web·api·app 4곳 동시 갱신 + DB 컬럼이 실제로 생기면 sql_guard 화이트리스트까지)를 그대로 따른다.
- 렌더 소비처(web/app 공통)는 계약-외 정규화를 방어적으로 처리해야 한다: `accident_status`가 3값 밖(또는 빈 문자열)이면 `null`과 동일하게 신뢰 뱃지 없음으로 처리, `is_single_owner`/`is_non_smoker`는 `true`/`false`/`null`(미상) 3상태를 구분해 `null`을 `false`로 오해하지 않는다.
- **찜 여부는 ListingCard wire 필드가 아니다.** "내가 찜했는지"는 사용자별 오버레이라 별도 조회/조인으로 처리한다(계약 오염 방지) — 이 스토리(10.5)가 그 조회 경로를 구현한다.
- 옵션 통제어휘·우선순위 상수(`COMMON_OPTIONS`, `OPTION_PRIORITY`)는 web/app 이원화 없이 `conventions.md`를 단일 출처로 선언한다.
- 마이그레이션은 additive·forward-only가 원칙이다(기존 걸 지우거나 바꾸지 않고, 새 컬럼·테이블만 더한다).

## UX & Interaction Patterns

- 색 체계: 신뢰속성 전용 색은 차분한 초록(`trust-green-bg`/`trust-green-ink`)이며 가격/CTA에 쓰는 브랜드 앰버와 분리한다 — "가격=amber, 신뢰=green" 두 신호가 섞이지 않게 한다. 신뢰 뱃지에 amber를 쓰는 것은 금지.
- 카드 레이아웃(레이아웃 B): 사진 바로 아래 신뢰속성 전용 행(초록 칩 + "판매자 제공 정보" 면책, 11px 톤 다운 텍스트) → 차량명 → meta → 가격(최상위 강조) → 희소 옵션 칩(3~4개, 밀도 높으면 "대표 1개 + 외 N개"). 찜(♡) 버튼은 사진 밖 우상단 원형.
- 신뢰 뱃지는 색만이 아니라 색+아이콘(✓)+텍스트로 동시에 표기한다(비색 신호 중복 — 접근성 요구). "판매자 제공 정보" 면책 라벨은 뱃지와 **한 몸으로** 렌더돼야 한다(뱃지만 있고 면책이 빠지는 경로가 있으면 안 됨). 상세 신뢰 섹션에는 "판매자가 직접 입력한 정보예요. 차장님이 검증한 내용은 아니니, 계약 전 꼭 직접 확인하세요." 문구를 표시한다.
- 상세 페이지 섹션 순서는 **신뢰정보 → 차량정보 → 옵션 → 판매자정보**가 정답이다(구버전 목업 `detail-1.html`은 옵션↔차량 순서가 다르므로 따르지 않는다).
- 옵션 하이브리드 피커(등록 폼): 기본은 인기 옵션 8칩(스마트키·내비·후방카메라·열선시트·통풍시트·선루프·크루즈·어라운드뷰). "전체 옵션 더보기"는 엔카 5분류(외관/내장·안전·편의/멀티미디어·시트·기타) 아코디언 체크리스트 + 검색창이며, 희소 옵션엔 "희소" 태그를 붙인다. 선택 상태는 칩 요약 + 개수로 보여준다.
- 찜 토글은 낙관적 반영(누르면 즉시 채워짐) 후 서버 확정, 실패하면 롤백 + 조용한 토스트. 비로그인 사용자가 누르면 로그인 게이트로 보냈다가 로그인 후 원 위치로 복귀시키고 찜을 반영한다. 신뢰 뱃지는 비검증 판매자 신고값이라 클릭 액션이 없다(정보 표시 전용).
- 찜 목록 빈 상태 문구: "아직 찜한 매물이 없어요. ♡를 눌러 관심 매물을 모아보세요." 목록 안의 sold/삭제된 매물은 회색 처리 + "판매완료"/"삭제된 매물" 비활성 배지로 표시하고 상세 진입은 막는다(기존 찜 관계 자체는 유지).
- 접근성: 아이콘 전용 버튼(찜♡ 등)은 시각 크기와 무관하게 히트 영역 ≥44×44px를 확보한다. 찜 토글은 `aria-pressed` + "찜하기"↔"찜 취소" 텍스트 라벨 전환으로 스크린리더에 상태를 알린다.
- 이 에픽(10.2)이 상세 신뢰 섹션을 채우는 자리이므로, 대장에 이월돼 있던 상세 갤러리 접근성 문제 3종(사진 전환이 스크린리더에 고지되지 않는 문제 등)도 함께 처리한다.
- 이 에픽(10.6)이 판매자 정보 슬롯을 문의 CTA와 같은 카드에 넣는 자리이므로, 대장에 이월돼 있던 `InquiryCta` 컴포넌트의 이중 마운트 문제도 함께 처리한다.

## Cross-Story Dependencies

- Epic 9가 만든 상세 페이지 골격(빈 섹션 순서: 신뢰→차량→옵션→판매자)을 전제로 한다 — 이 에픽은 그 슬롯을 채우는 것이지 골격을 새로 설계하지 않는다.
- 10.1(스키마 락스텝)이 끝나야 10.2(뱃지 렌더)가 실제 값을 그릴 수 있고, 10.3(옵션 상수·저장구조)이 끝나야 10.4(등록 피커)가 그 상수를 소비할 수 있다. 10.5(찜)와 10.6(판매자 정보)은 상호간 강한 의존은 없지만 둘 다 Epic 9 상세 골격에 얹힌다. 10.7은 이 에픽 전체(카드·상세·면책·희소 옵션)가 실제로 눈에 구분돼 보이는지를 마지막에 통합 검증하는 자리다.
- Epic 8이 세운 반응형·상태·접근성 프리미티브를 재사용한다. Epic 11(AI 히어로 랜딩)은 이 에픽이 완성한 카드(신뢰 뱃지·희소 옵션·찜 버튼 포함)를 그대로 재사용하므로 카드 계약이 흔들리면 영향을 받는다. Epic 16(Flutter)은 이 기능을 앱에 후속 미러링하며 계약(conventions.md)만 공유하고 위젯은 별도 구현이다.
- 대장(열린 일 장부, `docs/tech-debt.md`)에서 이 에픽이 실제로 닫는 항목: 옵션 쉼표 라운드트립 손실(#11, 10.3) · 카드 meta 연료 필드 누락(#67, 10.1) · 상세 갤러리 접근성 3종(#79, 10.2) · `InquiryCta` 이중 마운트(#82, 10.6) · 시드 재실행 시 사진·채팅방이 함께 삭제되는 문제(#89 — wishlists라는 새 자식 테이블이 하나 더 붙으므로 10.5에서 다시 판단하고 근거를 대장에 남긴다).
