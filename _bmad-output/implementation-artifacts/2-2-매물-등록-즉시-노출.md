# Story 2.2: 매물 등록 + 즉시 노출

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 판매자,
I want 차량 정보를 입력해 매물을 등록하고 싶다,
so that 구매자에게 바로 노출된다.

## Acceptance Criteria

1. **(등록 → 즉시 노출)** 판매자로 로그인한 상태에서 15필드 등록 폼을 채워 제출하면, `listings` 행이 `status=on_sale`로 생성되고 **즉시 목록에 노출**된다(관리자 사전 승인 없음). [FR5·FR7]
2. **(고정 목록 검증 — CHECK 위반 거절)** CHECK 제약을 위반하는 값(고정 목록 6필드 외 차종·제조사 등)으로 제출하면, **한국어 검증 오류**로 거절된다(DB CHECK 위반이 사용자에게 노출되지 않고 한국어 메시지로 변환). 폼의 6필드는 드롭다운으로 제공해 애초에 목록 밖 값을 못 고르게 하고, 서버측에서도 재검증한다.
3. **(단위 규칙 저장)** 단위 규칙(`price`=원, `mileage`=km, `displacement`=cc)대로 저장된다. 입력란에 단위가 표기되고, 저장값은 정수(원·km·cc)다. [docs/conventions.md §3]
4. **(역할 게이트 — seller만 접근)** seller 역할만 등록 화면(`/sell`)에 접근할 수 있다. 비로그인은 `/login`으로, 다른 역할(buyer/admin)은 홈(`/`)으로 리다이렉트된다(앱/프록시 책임). RLS는 본인(`auth.uid()=seller_id`) 명의 INSERT만 허용한다(2-1에서 확정). [FR3 기반·2-1 설계결정 2]
5. **(드롭다운 단일출처 정합)** 고정 목록 6필드(manufacturer·body_type·color·fuel·transmission·region)의 드롭다운 허용값이 `0002_listings.sql` CHECK·architecture.md 확정표와 **바이트 단위 일치**(drift 금지)한다. 공유 constants로 미러링한다.

> **범위 주의(반드시 준수):** 이 스토리는 **판매자 매물 등록 폼 + 제출 + 즉시 노출 확인**까지다. 본인 매물의 **수정·삭제는 2-3**, **구매완료(sold) 전환은 2-4**, **샘플 시드는 2-5**, **구매자용 목록·필터·상세(FR9~11)는 Epic 3** 소관이라 만들지 않는다. "즉시 노출"(FR7) 검증은 **등록 직후 `/sell` 화면에 본인 등록 매물이 보이는 것**으로 충족한다(구매자 전체 목록 화면은 Epic 3). `embedding`은 NULL로 둔다(Epic 4 backfill). 마이그레이션·DB 스키마는 2-1에서 완료됐으므로 **건드리지 않는다**.

### 등록 경로·구조는 architecture.md 확정본을 따른다
- 등록 화면 = **`web/src/app/(user)/sell/`** (라우트 그룹 `(user)`). [Source: architecture.md line 346·351]
- 현재 web에는 `(user)` 그룹이 **아직 없다**(이번에 처음 생성). `(auth)`·`(admin)` 그룹은 이미 있으니 같은 패턴을 따른다.
- 매물 목록 `(user)/page.tsx`·상세 `listings/[id]`는 **Epic 3** 소관 → 여기서 만들지 않는다. "즉시 노출"은 `/sell`의 본인 매물 섹션으로 검증(범위 컷). [Source: architecture.md line 347-349, epics.md Epic 3]

### 2-1에서 확정된 DB 계약 (재발명·재설계 금지)
- `listings` 테이블·6개 RLS 정책·CHECK·트리거는 **2-1에서 라이브 Supabase에 적용 완료**. 이번 스토리는 DB를 건드리지 않는다.
- **INSERT 정책 `listings_insert_own`**: `with check (auth.uid() = seller_id)` → seller_id를 타인으로 위조 불가. 폼에서 `seller_id`를 현재 user.id로 명시 세팅한다. [Source: 0002_listings.sql:104-105]
- **status 기본값 `on_sale`** (CHECK `in ('on_sale','sold')`). INSERT 시 생략하면 자동 on_sale → FR7 즉시 노출. [Source: 0002_listings.sql:29]
- **price = `bigint`**, year·mileage·displacement·seats = `int`. JS `number`로 충분(가격도 안전 정수 범위 내). [Source: 0002_listings.sql:42-51]
- **고정 목록 6필드 CHECK 값(바이트 단위 일치 대상):**
  - manufacturer: `현대, 기아, 제네시스, 쉐보레, 르노코리아, KG모빌리티, BMW, 벤츠, 아우디, 폭스바겐, 토요타, 혼다, 렉서스, 테슬라, 기타`
  - body_type: `경차, 소형차, 준중형차, 중형차, 대형차, 스포츠카, SUV, RV, 경승합차, 승합차, 화물차, 기타`
  - color: `흰색, 검정, 회색, 은색, 파랑, 빨강, 갈색, 녹색, 기타`
  - fuel: `가솔린, 디젤, 하이브리드, 전기, LPG`
  - transmission: `자동, 수동`
  - region: `서울, 부산, 대구, 인천, 광주, 대전, 울산, 세종, 경기, 강원, 충북, 충남, 전북, 전남, 경북, 경남, 제주`
  - 숫자 범위: year 1990~2027, seats 2~11, mileage≥0, price≥0, displacement≥0(전기차 0). [Source: 0002_listings.sql:35-54]
- **`accident_free`** default true, **`options text[]`** default `{}`, **`description`** nullable. [Source: 0002_listings.sql:55-57]

### CHECK 위반 → 한국어 에러 변환 (AC2 핵심)
- 드롭다운으로 6필드를 막으면 정상 흐름에선 CHECK 위반이 안 생긴다. 그래도 **서버측 방어**로 INSERT가 CHECK/RLS로 실패할 수 있으니 Postgres 에러를 한국어로 변환한다.
- Supabase JS 에러 `code`: CHECK 위반 = `23514`, RLS/권한 = `42501`. signup의 `toKoreanError`처럼 code/메시지로 분기해 한국어 메시지(예: "입력값이 허용 범위를 벗어났습니다. 항목을 확인해주세요.")를 띄우고 원본은 콘솔에만.
- 범위·필수 위반은 **클라이언트 검증으로 먼저** 막아 UX를 좋게 한다(signup 패턴).
