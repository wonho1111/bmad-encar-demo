# Story 7.5: 구매자·판매자 문의 채팅 (재현 FR19~21)

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 구매자/판매자(모바일),
I want 앱에서 매물 상세의 "문의하기"로 그 매물 판매자와 1:1 채팅방을 열고(있으면 재사용), 폴링으로 메시지를 주고받고 싶다,
so that 이동 중에도 매물에 대해 대화하고 거래를 진행한다(FR19~21 재현). 같은 매물·구매자 조합은 항상 같은 방을 쓰고, 메시지는 DB에 영속되며 양측이 ≤5초 내 상대 메시지를 본다.

## Acceptance Criteria

1. **(AC1 — 문의 진입 → 방 생성/재사용, FR19 재현)** 구매자가 **매물 상세(`listing_detail_screen.dart`)에서 "문의하기"** 를 누르면, 그 매물의 판매자와의 채팅방이 **없으면 생성·있으면 재사용**되어 채팅방 화면으로 진입한다. **"같은 매물(listing_id) × 같은 구매자(buyer_id)" 조합은 항상 같은 방**(중복 방 생성 없음). seller_id는 **클라이언트가 보내지 않는다** — DB 트리거(`0003c enforce_chat_room_seller`)가 매물 실소유자로 강제하므로 insert에는 `listing_id`·`buyer_id`만 넣는다. **본인 매물(buyer=seller)에는 문의 버튼을 숨긴다**(`listing.sellerId == 현재 user.id`); 우회해 시도하면 DB CHECK(`buyer_id<>seller_id`, 23514)가 한국어로 거부한다.
   - *근거*: web `lib/chat.ts` `openOrCreateRoom`(select-first→insert, 경합 23505 재조회, 23514 본인매물·23503 없는매물 한국어 분기) 이식. 마이그 `0003_chat`(UNIQUE listing_id,buyer_id,seller_id)·`0003c`(seller 강제 트리거). 7.2 `ListingDetail.sellerId` 주석이 이미 "본인이면 문의 버튼 숨김(7.5)" 예고.

2. **(AC2 — 채팅방 목록·진입, FR19)** "문의 채팅" 목록 화면에서 **내가 당사자(구매자 또는 판매자)인 방만** 최신순(created_at desc, id desc 2차)으로 나열된다(RLS `chat_rooms_select_participant`가 제3자 0건 보장 — 별도 필터 불필요). 각 행에 매물 요약 + 내 역할 기준 상대 표기(상대 이름 = 이메일 @앞부분, `0008` 비정규화 `buyer_name`/`seller_name`)가 보이고, 누르면 그 방으로 진입한다. 방이 없으면 **역할별 빈 상태**(구매자: "매물 상세에서 문의하기를 눌러보세요" / 판매자: "구매자가 문의하면 채팅방이 생깁니다"). 조회 실패는 "없음"과 구분해 한국어 오류. **매물 임베드가 null(판매완료 sold·구매자 RLS상 조회불가)** 이면 상세 정보 대신 "판매 완료되었거나 조회할 수 없는 매물" 플레이스홀더(FR11 — 구매자에게 sold 비노출). 진입 동선은 홈(`home_screen.dart`)에 구매자·판매자 공통으로 노출.
   - *근거*: web `(user)/chat/page.tsx`(RLS만으로 내 방, 임베드 listings, 역할별 빈상태, sold=플레이스홀더) 이식. nav-ia §1 공통 동선.

3. **(AC3 — 폴링 송수신·영속, FR20·21 재현)** 채팅방 화면에서 메시지를 입력·전송하면 `chat_messages`에 **INSERT(영속 저장)** 되고 내 화면에 즉시(낙관적) 반영된다. **3~5초 폴링**(web과 동일 하단값 사용 — 4초 또는 3초)으로 **커서 뒤 새 메시지만 증분 조회**(`gte` 커서 + id dedupe)해 상대 메시지를 **≤5초 내 갱신**한다. 메시지는 created_at 오름차순(동시각 id 2차)으로, **내 메시지(오른쪽)·상대(왼쪽)** 로 구분 표시. 0건이면 빈 상태 안내. 빈 본문은 클라 1차 차단(trim) + DB CHECK(23514) 최종 방어. 전송 실패 시 한국어 에러 + 입력 복원. 화면 이탈 시 **폴링 타이머 정리**(메모리 누수·유령 요청 방지).
   - *근거*: web `lib/messages.ts`(`fetchMessages` gte 커서·`order(created_at,id)`·`sendMessage` trim·23514·`dedupeById`)+`ChatRoomMessages.tsx`(낙관 전송·setInterval·cleanup) 이식. `gt(>)` 가 아니라 `gte(>=)` 인 이유(동시각 누락 방지)는 web 주석에 실측 근거. NFR1 폴링 3~5초.

4. **(AC4 — 채팅방→매물 상세 링크)** 채팅방 화면에서 **매물이 살아있는(on_sale, 임베드 non-null)** 방이면 그 매물 상세(`ListingDetailScreen(listingId)`)로 가는 링크/버튼을 보인다. **판매완료(임베드 null)면 링크를 숨긴다**(FR11 — 구매자가 sold 상세를 못 봄, 상세도 어차피 0건). web 커밋 `8c51edf`의 "on_sale일 때만 매물 상세 링크" 규칙을 모바일에 동일 적용.
   - *근거*: web `(user)/chat/[roomId]/page.tsx`(`l && <Link href=/listings/[id]>`) + 커밋 `8c51edf`. 7.2 `ListingDetailScreen`이 이미 존재(`listingId` 받음).

5. **(AC5 — 단일출처·계약 일치)** 방 생성/재사용 규칙은 **한 곳**(`chat_repository.dart` 또는 동급)에, 메시지 조회·전송·dedupe·정렬도 **한 곳**에 모은다(web의 `lib/chat.ts`·`lib/messages.ts` 단일출처 정신). **테이블·컬럼명·RLS·폴링 주기·이름 규칙은 web/DB와 바이트 일치**: 테이블 `chat_rooms`/`chat_messages`, 컬럼 snake_case(`listing_id`·`buyer_id`·`seller_id`·`buyer_name`·`seller_name`·`room_id`·`sender_id`·`body`·`created_at`), 표시 이름 = 이메일 @앞부분(`0008`). React/CSS는 이월하지 않고 **Flutter 위젯 신규 작성**. SQLSTATE 분기(23505/23514/23503)도 web과 동일.

6. **(AC6 — 검증·테스트)** `flutter analyze`(0 error)·`flutter test`(레포·dedupe·정렬 등 순수 로직 단위 테스트)·`flutter build web --dart-define-from-file=.env.json`이 통과한다. 가능하면 구매자 토큰으로 **문의→방 생성 / 같은 조합 재요청→재사용(같은 id) / 메시지 INSERT→양측 조회 / 폴링 증분(gte)** 을 live Supabase HTTP(REST) 스모크로 실증(생성한 테스트 행은 정리, 못 하면 표기). 브라우저 픽셀 렌더는 시스템 라이브러리 미비 시 빌드+테스트로 갈음(그 사실 표기).

## Dev Notes

### 핵심 계약(반드시 일치 — web/DB 단일출처)
- **테이블/컬럼(snake_case, 변환 없이 그대로 통신선에):**
  - `chat_rooms(id, listing_id, buyer_id, seller_id, buyer_name, seller_name, created_at)` — UNIQUE(listing_id, buyer_id, seller_id), CHECK(buyer_id<>seller_id).
  - `chat_messages(id, room_id, sender_id, body, created_at)` — CHECK(length(btrim(body))>0).
- **RLS(신뢰·우회 금지):** `chat_rooms_select/insert_participant`(당사자만), `chat_messages_select/insert_participant`(방 당사자 + sender_id=auth.uid()). 제3자는 **예외가 아니라 0건**. seller_id는 클라가 못 넣음 — `0003c` 트리거가 매물주로 강제(insert에 listing_id·buyer_id만).
- **이름 표시(0008):** `buyer_name`/`seller_name` = 이메일 @앞부분, BEFORE INSERT 트리거 자동 기록(이미 백필됨). 앱은 **읽기만**(조인·재계산 금지). 없으면(예전 방) 역할만 폴백.
- **폴링 커서:** `gte('created_at', cursor)` + `dedupeById`(누락0·중복0). **`gt` 쓰면 동시각 메시지 영구 누락**(web 실측 버그). 정렬은 항상 `(created_at asc, id asc)`.
- **SQLSTATE:** 23505=UNIQUE 경합(재조회 흡수)·23514=CHECK(본인매물/빈본문)·23503=FK(없는 매물 트리거). `PostgrestException.code`로 분기(supabase_flutter는 PostgrestException에 `code` 노출).
