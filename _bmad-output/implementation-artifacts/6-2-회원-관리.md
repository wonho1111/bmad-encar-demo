# Story 6.2: 회원 관리 (FR22)

Status: review

## Story

As a 관리자,
I want `/admin` 안에서 전체 회원을 조회하고 한 회원을 정지(`profiles.status`)하거나 삭제하고 싶다,
so that 이상 회원을 운영 차원에서 관리할 수 있다(FR22).

## Acceptance Criteria

1. **Given** 관리자(role=admin)가 회원 관리 화면(`/admin/members`)을 열면, **When** 목록을 조회하면, **Then** 전체 회원(`profiles`)이 보인다(본인 포함, 역할·상태·가입일 표시). 비관리자/비로그인은 기존 `(admin)` 가드(proxy 1차 + layout `requireRole(admin)` 2차)로 차단된다. (epics.md L659~661, FR22)
2. **Given** 회원 목록에서 한 회원을, **When** "정지"하면, **Then** 그 회원의 `profiles.status`가 `suspended`로 바뀌고 화면에 즉시 반영된다. 이미 정지된 회원은 "정지 해제"(→`active`)로 되돌릴 수 있다. (epics.md L663~665, FR22 — `profiles_update_admin` RLS 사용)
3. **Given** 회원 목록에서 한 회원을, **When** "삭제"하면(확인 후), **Then** 그 회원의 `profiles` 행이 제거되고 목록에서 사라진다. 삭제는 되돌릴 수 없으므로 `window.confirm`으로 실수를 방지한다. (epics.md L663~665, FR22 — `profiles_delete_admin` RLS 사용)
4. **Given** 관리자 본인 계정 행에 대해, **When** 화면을 보면, **Then** 본인에 대한 정지/삭제 버튼은 노출되지 않는다(관리자가 스스로를 정지·삭제해 운영 권한을 잃는 사고 방지). 다른 관리자 행은 데모 범위상 일반 회원과 동일하게 다룬다(정책상 RLS는 허용).
5. **Given** 정지/삭제 시도가 RLS·네트워크로 막히면(예: 권한 상실, 행 없음), **When** 결과가 0행이거나 에러면, **Then** 원본 에러는 콘솔에만 남기고 사용자에겐 한국어 일반 안내를 보여준다(기존 `ListingActions` 규율과 동일). 비관리자 클라이언트로는 애초에 화면 진입이 불가하다.
6. **Given** 본 스토리 구현 후, **When** 마이그레이션/RLS 상태를 확인하면, **Then** 신규 마이그레이션은 없다(`profiles.status` 컬럼은 0001에 이미 존재, 정지/삭제 RLS는 0005에 이미 존재). 기능은 순수 web(Next.js) 화면 추가로 달성된다.

### 설계 결정(확정)
- **정지 = `profiles.status` UPDATE**: payload에 `{ status }`만 보낸다(role 등 다른 필드 위조·부수변경 차단). `profiles_update_admin`의 `with check = is_admin()` 덕분에 관리자가 **타인** 행을 수정한 결과가 통과한다(본인 한정이 아님). 양방향(active↔suspended) 토글 지원 → 정지 해제도 같은 경로.
- **삭제 = `profiles` 행 DELETE**: `profiles_delete_admin`(using=`is_admin()`)로 허용. `profiles.id`는 `auth.users(id) on delete cascade` 참조이지만, **방향이 반대**다 — `profiles` 행을 지운다고 `auth.users`(실제 로그인 계정)가 사라지지는 않는다. 즉 본 스토리의 "삭제"는 **프로필(역할·상태) 제거**까지다.
  - **계정 로그인 자체의 완전 삭제**(auth.users 행 제거)는 `supabase.auth.admin.deleteUser()` = **service-role 키 + admin API**가 필요한데, 현재 `web/.env.local`에 `SUPABASE_SERVICE_ROLE_KEY`가 없다(키 노출 위험 때문에 클라이언트에 두면 안 됨). 데모 범위에서는 프로필 삭제로 FR22 "계정이 제거된다"의 운영 의미(서비스 이용 차단·식별정보 제거)를 충족하고, **완전 계정 삭제(auth.users)는 후속/콘솔 작업으로 인계**한다(아래 "사용자 직접 처리 항목"). 6-1 Dev Notes(L67)가 "6-2에서 서비스롤 vs RLS 결정"이라 했고, **여기서 RLS(프로필 삭제)로 결정**한다. 무인 진행 중 누락 env로 멈추지 않기 위한 합리적 기본값.
- **본인 보호(AC#4)**: 관리자가 자기 자신을 정지/삭제하면 즉시 운영 권한을 잃거나(가드 통과 불가) 화면이 깨질 수 있어, **본인 행에는 액션 버튼을 숨긴다**(앱측 1차 방어). RLS는 본인도 허용하므로 이는 UX 가드다.
- **이메일 비노출**: `profiles`에 이메일이 없다(이메일은 `auth.users` 소관, anon-key로는 타인 이메일 조회 불가). 목록은 회원 id(축약)·역할·상태·가입일로 식별한다. 이메일까지 보이려면 service-role 조회가 필요 → 본 스토리 범위 밖(위 계정 삭제와 동일 사유). 데모 식별엔 id로 충분.
- **회귀 안전**: 신규 파일 위주(page/MemberActions) + 기존 admin/page.tsx 문구 갱신 + constants 상수 1개 추가. 기존 (user)/(auth) 동선·RLS·마이그레이션 무변경. 비관리자는 `(admin)` 가드로 화면 자체에 못 들어온다.

### Completion Notes List

- **무엇을·왜**: 관리자가 전체 회원을 보고 한 명을 정지·삭제할 수 있는 화면(`/admin/members`)을 추가했다(FR22). DB는 한 줄도 바꾸지 않았다 — 정지 상태를 담는 `profiles.status` 컬럼은 0001에, 관리자 정지(UPDATE)/삭제(DELETE)를 허용하는 RLS(`profiles_update_admin`·`profiles_delete_admin`)는 6-1의 0005에 이미 있었기 때문. 즉 6-1이 만든 "DB 자물쇠" 위에 "관리자용 버튼"만 web으로 얹었다.
- **정지 = `profiles.status` 토글**: 정지(active→suspended)·정지 해제(suspended→active)를 한 버튼이 상태에 따라 라벨을 바꿔 처리. payload는 `{ status }`만 보내 다른 필드(역할 등) 위조·부수변경을 차단. RLS가 막으면 0행 → 한국어 거부.
- **삭제 = `profiles` 행 DELETE**: `window.confirm`으로 실수 방지 후 삭제. ⚠️ **이건 프로필(역할·상태) 제거까지다** — 로그인 계정 자체(`auth.users`)는 service-role 키 + admin API가 있어야 지울 수 있는데 `web/.env.local`에 해당 키가 없어(클라이언트에 두면 보안 위험) 본 스토리 범위 밖이다. E2E에서 삭제 후 `profile_rows=0 / auth_rows=1`로 이 경계를 실제로 확인했다. 6-1 Dev Notes가 미뤘던 "service-role vs RLS" 결정은 **RLS(프로필 삭제)** 로 확정. 완전 계정 삭제는 콘솔/후속 작업으로 인계(아래 사용자 처리 항목).
- **본인 보호(AC#4)**: 관리자 본인 행에는 정지/삭제 버튼을 렌더하지 않는다(자기 정지·삭제로 운영 권한을 잃는 사고 방지). "나" 배지로 식별.
- **이메일 비노출**: profiles에 이메일이 없어(타인 이메일은 anon-key로 조회 불가) 식별은 역할 + id 앞 8자로 한다. 이메일까지 보이려면 service-role 조회 필요 → 범위 밖.
- **회귀 안전**: 신규 파일 2개 + 기존 admin 홈 문구 갱신 + constants 상수 1개. 기존 (user)/(auth) 동선·RLS·마이그레이션 무변경. 비관리자는 `(admin)` 가드로 화면 진입 자체가 불가(6-1 검증). lint·typecheck·E2E 통과.
