## Story

As a 시연자(과제 제출자),
I want 핵심 시연 시나리오(SM1 3종 + SM2·SM3·SM4)를 앱·웹에서 처음부터 끝까지(end-to-end) 통과시키고 그 증거를 한 곳에 남기고 싶다,
so that 과제 제출 기준(SM1: 핵심 시연 3종이 끝까지 동작)을 충족함을 **객관적으로 증명**한다. 이 스토리는 신규 기능이 아니라 **Epic 7(앱) + Epic 1~6(web)이 합쳐진 전체 데모가 깨지지 않고 완주됨을 통합 검증**하는 스토리다.

## Acceptance Criteria

> SM 정의(PRD 단일출처 — `_bmad-output/planning-artifacts/prds/prd-bmad-encar-demo-2026-06-17/prd.md` §SM):
> - **SM1**: ① 구매자 AI검색→상세→문의 채팅, ② 판매자 등록→문의 응대→구매완료, ③ 관리자 조회·삭제 — 3종 모두 end-to-end 동작.
> - **SM2**: 기본 검색 — 구매자가 키워드/필터(가격·연식·차종·색상·연료·변속기·지역 등, FR9)로 매물을 찾는다.
> - **SM3**: AI 검색이 **두 경로(경로 A Text-to-SQL · 경로 B 문서 RAG)** 를 모두 시연.
> - **SM4**: 대표 시연 질의(구조형·질적형)에 적절한 매물 카드가 반환된다.

1. **(AC1 — 앱 구매자 여정 ①, 데이터 레벨 end-to-end)** 앱 구매자 여정 "AI검색 → 매물 상세 → 문의 채팅"이 **끊김 없이 완주**됨을 데이터·계약 레벨로 실증한다.
   - 매물 조회는 **`on_sale`만** 노출(구매자 RLS·FR11). 특정 `on_sale` 매물의 상세 정보(15필드)가 조회된다.
   - 그 매물에 대해 구매자가 문의하면 채팅방이 **(listing_id × buyer_id) 단위로 생성·재사용**(중복 방 없음, seller_id는 트리거가 강제)된다.
   - 방에 메시지 INSERT → 양측(구매자·판매자) SELECT로 영속·가시성 확인.
   - AI검색 경로는 **API base URL이 있으면 라이브 1회**(`{answer, listings[]}` 수신), 없으면 **escalate 표기 + 코드↔계약 일치로 갈음**(AC5 참조).

2. **(AC2 — 앱 판매자 여정 ②, 데이터 레벨 end-to-end)** 앱 판매자 여정 "등록 → 문의 응대 → 구매완료"가 **끊김 없이 완주**됨을 실증한다.
   - 판매자가 매물을 등록(`on_sale` 생성, 소유자=판매자 — DB 트리거가 owner 강제)한다.
   - 구매자 문의방에 판매자가 응답 메시지를 INSERT(양측 가시).
   - 그 매물을 **구매완료(`sold`)** 로 전환하면, **구매자 화면(목록·상세·AI·채팅 임베드)에서 비노출**(FR11 — 구매자 RLS로 `sold` 0건)된다. **판매자 본인·관리자에게는 보임**(FR11 예외).

3. **(AC3 — web SM1 3종 + SM2·SM3·SM4)** web에서 SM1 ①구매자 ②판매자 ③관리자 3종 시나리오가 end-to-end 동작하고, SM2(필터검색)·SM3(AI 2경로)·SM4(대표 질의) 합격이 확인된다.
   - web은 Epic 1~6 **done·운영 배포 검증 완료** 상태(에픽 회고·핸드오프 증거 존재).
   - **브라우저 픽셀 E2E가 가능하면 실행**(기존 e2e 테스트 또는 Playwright 리허설). 시스템 라이브러리 미비로 불가하면 **그 사실을 명시**하고, web과 **동일한 live Supabase 데이터 계약 + 기존 운영배포 검증 증거**로 합격을 확인(데이터레벨 갈음).
   - SM3·SM4는 web 4.8 스토리(SM3·CM1·CM2 합격 판정)에서 이미 done 판정된 자산을 근거로 재확인. AI 라이브 호출은 AC5 정책을 따름.

4. **(AC4 — 검증 산출물)** 위 SM1~SM4 항목별 **합격 / 데이터레벨갈음 / 미수행(+사유)** 표를 포함한 **검증 보고서**를 이 스토리 파일(Dev Agent Record) 또는 `_bmad-output/implementation-artifacts/`에 남긴다. 앱 빌드·테스트 결과(핵심 로그)·AI 라이브 성공 여부·escalate 항목·사용자 직접 처리 항목을 명시한다. **생성한 테스트 행은 정리**(못 하면 표기).

5. **(AC5 — AI /ai/search 라이브 시도 또는 escalate)** AI 검색은 앞 스토리에서 코드↔계약 대조만 했고 라이브 미수행이다. **API base URL을 찾아 라이브 1회 시도**한다:
   - 후보 순서: `app/.env.json`의 `API_BASE_URL` → `web/.env*`의 `NEXT_PUBLIC_API_BASE_URL` → `api/` 배포 설정 → `gcloud run services list`로 `encar-ai-api-dev` URL.
   - 성공하면 `{answer, listings[]}` 수신을 실증(SM3·SM4 라이브 보강).
   - **URL을 못 구하면 escalate(사용자 직접 처리)로 명시하고 코드↔계약 일치로 갈음** — **Halt하지 말고 계속**.

6. **(AC6 — 앱 빌드 건강성)** `export PATH="$HOME/flutter/bin:$PATH"` 후 `flutter analyze`(0 error)·`flutter test`(전체 green)·`flutter build web --dart-define-from-file=.env.json`(성공)이 재확인된다.

## Dev Notes

### 이 스토리의 성격
- **통합 검증(end-to-end) 스토리** — 신규 기능 거의 없음. 7.1~7.5에서 완성된 앱과 Epic 1~6 web이 합쳐진 **전체 데모가 깨지지 않고 완주됨**을 데이터·계약 레벨로 실증한다.
- **네이티브 픽셀 E2E 불가**: 실폰/에뮬레이터/Chrome 데스크톱 없음(CLAUDE.md §6, 7.1~7.5 전 스토리 동일 제약). 따라서 `flutter analyze`+`flutter test`+`flutter build web`+**live Supabase HTTP/MCP 스모크**로 데이터·계약 레벨 end-to-end 체이닝을 하고 **그 한계를 명시**한다.

### 환경(반드시 숙지)
- **현재 git 브랜치 = `develop`** (main 금지, push 금지 — 오케스트레이터가 처리).
- **Flutter SDK**: `~/flutter/bin`. 셸 상태 비유지 → **모든 Bash 호출 앞에 `export PATH="$HOME/flutter/bin:$PATH"`**. (Flutter 3.44.3, web 활성화)
- 앱 비밀값: `app/.env.json`(gitignore). 확인된 값: `SUPABASE_URL=https://psrnsasxpkpwqdukjdmt.supabase.co`, `API_BASE_URL=""`(빈 값 — AI 라이브 escalate 사유).
- live Supabase 프로젝트 = web과 동일(`psrnsasxpkpwqdukjdmt`). MCP `mcp__supabase__execute_sql` 사용 가능(service_role — RLS 우회 주의).

### 핵심 계약(검증 기준 — web/DB 단일출처)
- **listings**: 15필드 + `status`(`on_sale`/`sold`) + `embedding`(768). 소유권·status 전환·FR11 비노출 RLS는 `0002_listings`에 동거. 구매자에겐 `sold` 0건(FR11), 관리자·소유자는 예외.
- **chat_rooms**: `(id, listing_id, buyer_id, seller_id, buyer_name, seller_name, created_at)` — UNIQUE(listing_id, buyer_id, seller_id), CHECK(buyer_id<>seller_id), seller_id는 트리거(`0003c`)가 매물 owner로 강제. 클라는 `listing_id`·`buyer_id`만 INSERT.
- **chat_messages**: `(id, room_id, sender_id, body, created_at)`. 폴링 `gte` 커서 증분. 참여자 RLS(제3자 0건).
- **/ai/search 계약**: `POST {base}/ai/search`, JWT Bearer(사용자 토큰 — api가 읽기전용 롤로 RLS 검증), 응답 `{answer, listings[]}`(listings 원소=카드 7필드). 앱 `app/lib/features/ai_search/ai_search_api.dart` ↔ web `web/src/lib/api/aiSearch.ts` ↔ api `/ai/search` 일치.

### 현재 DB 상태(MCP 확인됨)
- 테이블: `chat_messages, chat_rooms, guide_documents, listings, profiles`.
- listings: `on_sale` 94건, `sold` 6건(시드 존재 — 새 시드 불필요, 기존 데이터로 체이닝 가능).

### Halt 정책
- 멈추고 보고할 것만: ① 여정이 실제로 깨져 완주 불가, ② 키/URL 누락 외 치명적 차단, ③ AC 모호.
- **AI URL 미보유는 escalate 표기만 하고 계속(Halt 아님).**

### 앞 스토리 인텔리전스(7.1~7.5)
- 테스트 전략 확립: 실폰·에뮬 없음 → `build web` + `flutter test` + live HTTP/REST 스모크. 7.5에서 채팅 방생성·재사용·메시지 INSERT·gte 증분을 이 방식으로 실증.
