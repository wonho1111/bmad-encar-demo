# Story 6.3: 전체 매물 관리 (FR23)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자,
I want `/admin` 안에서 **판매완료(sold) 포함 전체 매물**을 조회하고 부적절한 한 매물을 삭제하고 싶다,
so that 서비스 건전성을 유지할 수 있다(FR23).

## Acceptance Criteria

1. **Given** 관리자(role=admin)가 매물 관리 화면(`/admin/listings`)을 열면, **When** 목록을 조회하면, **Then** **판매완료(sold) 포함 전체 매물**(`listings`)이 보인다(관리자는 FR11 "판매완료 비노출"의 예외 — `listings_select_admin` RLS로 전체 행을 본다). 비관리자/비로그인은 기존 `(admin)` 가드(proxy 1차 + layout `requireRole(admin)` 2차)로 차단된다. (epics.md L667~681, FR23)
2. **Given** 매물 목록의 각 행에서, **When** 화면을 보면, **Then** 제조사·모델·연식·가격과 함께 **상태 배지(판매중=초록 / 판매완료=회색)**가 표시되어 관리자가 sold 매물도 식별할 수 있다. (FR23 — 전체 조회의 관리 식별성)
3. **Given** 매물 목록에서 한 매물을, **When** "삭제"하면(확인 후), **Then** 그 매물의 `listings` 행이 제거되고 목록에서 사라진다. 삭제는 되돌릴 수 없으므로 `window.confirm`으로 실수를 방지한다. (epics.md L676~681, FR23 — `listings_delete_admin` RLS 사용)
4. **Given** 삭제 시도가 RLS·네트워크로 막히면(예: 권한 상실, 행 없음), **When** 결과가 0행이거나 에러면, **Then** 원본 에러는 콘솔에만 남기고 사용자에겐 한국어 일반 안내를 보여준다(기존 `ListingActions`/`MemberActions` 규율과 동일). 비관리자 클라이언트로는 애초에 화면 진입이 불가하다.
5. **Given** 관리자 홈(`/admin`)에서, **When** 화면을 보면, **Then** "매물 관리"(`/admin/listings`) 진입 링크가 노출된다(6-2의 "회원 관리" 링크와 동일 형식).
6. **Given** 본 스토리 구현 후, **When** 마이그레이션/RLS 상태를 확인하면, **Then** 신규 마이그레이션은 없다(관리자 전체 조회 `listings_select_admin`은 0002, 관리자 삭제 `listings_delete_admin`은 0005에 이미 존재). 기능은 순수 web(Next.js) 화면 추가로 달성된다.

## Dev Notes

### 핵심 요지(가장 먼저 이해할 것)
- 이 스토리는 **DB 작업이 0건**이다. 관리자 전체 조회 RLS(`listings_select_admin`)는 0002에, 관리자 삭제 RLS(`listings_delete_admin`)는 0005(6-1 산출물)에 이미 들어 있다. **이번 작업은 그 토대 위에 web 화면만 얹는 것**이다 — 6-2(회원 관리)와 동일 구조의 "전체 매물판".
- 6-2가 "회원" 목록 + 삭제였다면, 6-3은 "매물" 목록(판매완료 포함) + 삭제다. 정지/수정 같은 부가 액션은 없다(부적절 매물 제거가 목적).

### 설계 결정(확정)
- **전체 조회 = 필터 없는 `listings` select**: `SellPage`는 "내 매물"만 보려고 `seller_id` 필터를 명시했지만(SELECT 정책이 "on_sale ∪ 본인 ∪ 관리자" OR이라 필터 없으면 남의 on_sale까지 섞임), **관리자는 정반대로 전부 보는 게 목적**이다. 관리자 세션에선 `listings_select_admin`(using=is_admin())이 모든 행을 열어주므로, 필터를 빼고 `select`하면 sold 포함 전체가 그대로 온다(FR11 예외). 별도 status 필터도 두지 않는다.
- **삭제 = `listings` 행 DELETE**: `listings_delete_admin`(using=`is_admin()`, 0005)로 허용. `MemberActions`의 삭제와 동일하게 `.select('id')`로 행 수를 확인 → 0행이면 한국어 거부(권한 상실/행 없음). 매물의 `seller_id`는 `profiles(id) on delete cascade`지만 방향이 반대라, 매물을 지워도 판매자 프로필/계정은 그대로다(매물만 제거).
- **부가 액션 없음(범위 컷)**: 6-3은 "부적절 매물 삭제"가 FR23의 관리 동작이다. 회원의 정지(suspend) 같은 토글이나 매물 수정은 관리 요구에 없어 넣지 않는다(0005 주석도 "listings UPDATE는 현재 관리 요구에 없어 추가 안 함"이라 명시). 판매완료 처리(2-4)는 판매자 본인 동선이지 관리자 동선이 아니다.
- **회귀 안전**: 신규 파일 2개(page/ListingAdminActions) + 기존 admin/page.tsx에 링크 1줄 추가. 기존 (user)/(auth)/(admin/members) 동선·RLS·마이그레이션 무변경. 비관리자는 `(admin)` 가드로 화면 진입 자체가 불가(6-1 검증).

- **부가 액션 없음(범위 컷)**: FR23의 관리 동작은 "부적절 매물 삭제"다. 정지/수정 같은 토글은 관리 요구에 없어 넣지 않았다(0005 주석도 listings UPDATE 미추가 명시). 판매완료 처리(2-4)는 판매자 동선이라 관리자 화면에 없다.
- **회귀 안전**: 신규 파일 2개(page/ListingAdminActions) + admin 홈 링크 1개 추가. 기존 (user)/(auth)/(admin/members) 동선·RLS·마이그레이션·constants 무변경. 비관리자는 `(admin)` 가드로 화면 진입 자체가 불가(6-1 검증).
- **테스트 표준 참고**: 이 repo엔 단위 테스트 프레임워크가 없어 검증은 eslint + tsc + Playwright E2E(프로젝트 표준).
