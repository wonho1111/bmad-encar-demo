## 잘된 점 (반복할 패턴)

1. **이전 회고 액션의 직접 이행** — Epic 2 회고 액션 #2("Epic 3 구매자 화면에서 RLS OR-결합 함정 때문에 sold/타인 비공개가 새지 않는지 E2E 검증")를 3-1·3-2·3-3 세 스토리가 전부 검증으로 이행. 회고가 장식이 아니라 다음 에픽의 AC로 살아 움직였다.
2. **단일 출처화 = "가드를 흩뿌림 아니라 모음"(3-3)** — 3-1·3-2가 각자 복붙했던 `.eq('status','on_sale')`를 `buyerListingsQuery` 헬퍼 한 곳으로 수렴. drift(한쪽만 바뀜) 위험 제거. 과한 추상화(repository 패턴 등) 없이 상수 1 + 함수 1로 절제.
3. **CM3 실DB 전환 검증** — on_sale 매물(아반떼 CN7)을 실제 `update set status='sold'` → 목록·필터·상세 세 경로 즉시 비노출 확인 → 원복. mock 아닌 실검증으로 캐시 잔존 없음(force-dynamic 효과)을 증명. **(리드 소감: "검증이 믿을만했다")**
4. **독립 코드리뷰의 실질 적발(3-1)** — 키워드 `ilike`의 LIKE 메타문자(`%`·`_`) 미이스케이프(High)·`asInt` 초과값 통과·정렬 불안정을 3레이어 리뷰가 잡아 자동 수정.
5. **스코프 규율** — 각 스토리가 범위를 명확히 컷(상세는 3-2, 회귀스위트는 3-3)하고 미구현 기능(문의·AI) 링크를 만들지 않음. DB·마이그레이션 불변(2-1 라이브 재사용).
6. **auto-epic 흐름 만족** — 스토리마다 create-story→dev-story→code-review가 무인으로 매끄럽게 진행, escalate·중단 0건. **(리드 소감: "자동개발 흐름이 만족스러웠다")**

## 어려웠던 점 / 반복된 함정

1. **복붙 drift 위험** — 3-1·3-2가 같은 FR11 가드를 복붙 → 3-3이 단일출처화로 수습. *교훈: 같은 가드가 두 번째로 복붙되는 순간이 단일출처화 신호다(첫 등장에선 과한 추상화를 피하되, 두 번째에서 모은다).*
2. **명세-실제 구조 불일치(3-1)** — 스토리 노트가 홈을 `(user)/page.tsx`로 가정했으나 실제는 `web/src/app/page.tsx`. dev가 정정 후 진행(기능 동일). 작은 마찰이나 create-story가 실제 파일 경로를 더 확인할 여지.
3. **min>max 역전 UX 미결(3-1)** — 가격/연식 최소>최대 입력 시 "0건 + 필터 완화" 안내가 오해 소지. 코드리뷰가 **제품 결정**으로 escalate(스왑/경고/현행유지). 데모 영향 작아 미결 보류.
4. **단위 테스트 프레임워크 부재** — 회귀 검증이 매번 수동 Playwright E2E. 데모 범위라 수용했으나 자동 회귀망은 없음(에픽이 쌓일수록 수동 회귀 비용 증가).

## 핵심 교훈

- **두 번째 복붙이 단일출처화의 신호** — 3-1·3-2 복붙 → 3-3 수습. 처음부터 추상화하지 말고, 반복이 확인되면 모은다.
- **데이터 정합(CM3)은 실전환으로 검증** — "동적 렌더면 되겠지"로 끝내지 않고 실제 sold 전환→재조회→원복으로 캐시 잔존을 눈으로 확인.
- **회고 액션은 다음 에픽 AC로 심는다** — Epic 2 #2가 Epic 3 검증으로 실현된 것이 가장 큰 성과. 추상적 다짐보다 다음 스토리의 구체 검증 항목으로 박는 게 이행률을 높인다.
