# Story 3.3: 판매완료 비노출 단일 규칙 (FR11)

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 서비스,
I want 판매완료(`status='sold'`) 매물을 구매자의 모든 경로(목록·필터검색·상세)에서 단 하나의 규칙으로 일관되게 숨기고 싶다,
so that 거래 불가능한 매물이 어디서도 노출되지 않고, 매물이 sold로 바뀌면 즉시 사라진다(데이터 정합 CM3).

## Acceptance Criteria

1. **(전 경로 sold 비노출 — FR11)** 구매자가 **목록(`/search`)·필터검색(`/search?…`)·상세(`/listings/[id]`)** 어디로 접근하든 `status='sold'` 매물은 보이지 않는다. 이는 RLS(구매자에게 `on_sale`만 통과) + 앱 쿼리의 "판매중만" 강제가 함께 보장한다. [FR11, Source: epics.md#Story-3.3(405-411), 0002_listings.sql:90-101, conventions.md §6]
   - 이 스토리는 **새 중복 가드를 흩뿌리지 않는다**. 3-1·3-2가 각각 손으로 박아둔 `.eq('status', LISTING_STATUS.ON_SALE)`를 **하나의 단일 출처(공통 헬퍼/상수)** 에서 비롯되게 가볍게 단일화하고(데모 범위 내), 모든 구매자 경로가 그 단일 출처를 거치는지 확인·고정한다(회귀 방지). [Source: search/page.tsx:98, listings/[id]/page.tsx:83]
2. **(sold 전환 시 즉시 비노출 — CM3)** 어떤 매물이 `on_sale`에서 `sold`로 바뀌면, 구매자가 **재조회(목록·필터·상세)** 했을 때 **즉시** 사라진다(캐시·지연으로 잔존하지 않음). [CM3, Source: epics.md#Story-3.3(409-411)]
   - 구매자 경로 페이지는 매 요청마다 DB를 다시 읽는 **동적 렌더**여야 한다(정적 캐시로 sold가 잔존하면 CM3 위반). 현재 패턴(서버 컴포넌트 + `auth.getUser()`로 인한 동적 렌더)이 이를 만족하는지 **검증**하고, 필요 시에만 최소 보강한다. [Source: search/page.tsx, listings/[id]/page.tsx]
3. **(단일 규칙의 단일 출처화 — 가벼운 리팩터)** "구매자에게는 `on_sale`만" 규칙이 **한 곳**에서 정의되어 목록·필터·상세 쿼리가 모두 그것을 사용한다. 새 상수/헬퍼는 기존 `web/src/lib/constants.ts`(`LISTING_STATUS`) 또는 작은 신규 `web/src/lib/listings.ts` 헬퍼 수준에 그친다 — **DB/RLS·마이그레이션은 건드리지 않는다**(2-1 라이브 완료). 과도한 추상화·레이어 신설 금지(데모 범위). [Source: constants.ts:10-15, 0002_listings.sql(2-1 완료)]
4. **(회귀 고정 + 정적 검증)** 리팩터 후에도 3-1(목록·7필터·키워드·빈상태·에러vs0건)·3-2(15필드 상세·sold 비노출·비로그인 보호·에러vs못찾음)의 동작이 **그대로** 유지된다. `npx tsc --noEmit` 0 에러, `npx eslint`(변경 파일) 0 경고. 관리자(FR11 예외, Epic 6)·AI(Epic 4)·채팅(Epic 5)은 **건드리지 않는다**. [Source: 3-1·3-2 스토리, epics.md(677 관리자 예외)]

> **범위 주의(반드시 준수):**
> - 이 스토리는 **확인·고정(회귀 방지) + 가벼운 단일출처화**가 본질이다. 기능을 새로 만들지 않는다.
> - **새 중복 가드 금지** — 경로마다 또 `.eq('status','on_sale')`를 흩뿌리는 게 아니라, **한 출처에서 비롯되게** 정리한다.
> - **DB·RLS·마이그레이션 불변**(2-1에서 라이브 적용 완료). 앱 계층만 정리·검증한다.
> - **관리자 전체조회(FR11 예외)·AI 경로 SQL guard(Epic 4)·문서 RAG 필터(Epic 4)는 본 스토리 범위 밖** — conventions §6이 그 강제 지점들을 "Epic 2~4 구현"으로 명시한다. 여기선 **구매자 웹 경로(목록·필터·상세)** 만 다룬다.
> - sold→재조회 즉시 비노출(CM3)은 **실제로 검증**한다(Supabase MCP로 임시 on_sale→sold 전환 후 재조회, 검증 끝나면 원복).

## Dev Notes

### ⚠️ 이건 당신이 아는 Next.js가 아니다 (web/AGENTS.md 경고)
- `web/AGENTS.md`: "코드 작성 전 `node_modules/next/dist/docs/`의 관련 가이드를 읽으라." Next.js **16.2.x**.
- 본 스토리는 **캐싱·동적 렌더**가 CM3 핵심이다 — `node_modules/next/dist/docs/`의 caching/`dynamic` 관련 가이드를 확인하고, 구매자 경로가 매 요청 DB를 다시 읽는지 못박는다. 서버 컴포넌트가 `cookies()`/`auth.getUser()`를 호출하면 Next.js가 동적 렌더로 처리하는 것이 통례지만, 가이드로 확인 후 필요 시 `force-dynamic` 최소 추가.

### 이 스토리의 본질: "단일 규칙" 확인·고정 + 가벼운 단일출처화 (재발명 금지)
- FR11 비노출은 **이미 동작 중**이다. 두 곳에서 강제된다:
  1. **RLS(2-1, 0002_listings):** `listings_select_on_sale ∪ own ∪ admin` OR 결합 → 구매자는 own·admin이 아니므로 **on_sale만** 통과. sold가 어느 경로로도 안 보임. **이게 단일 규칙의 진짜 근원.** [Source: 0002_listings.sql:90-101]
  2. **앱 쿼리 `.eq('status','on_sale')`:** 3-1 `/search`(line 98)·3-2 `/listings/[id]`(line 83)에 각각 손으로 박혀 있다. 이유 = **판매자**가 본인 sold 매물의 목록/상세에 들어오면 RLS `own` 정책으로 통과될 수 있어, "구매자 관점(판매중만)"을 앱에서 한 번 더 강제(이중 방어). [Source: search/page.tsx:11-14·98, listings/[id]/page.tsx:12-14·83]
- **문제(이 스토리가 고치는 것):** 같은 규칙이 두 파일에 **복붙**돼 있다 → 한쪽만 바뀌면 drift(어긋남). 그래서 **한 출처에서 비롯되게** 가볍게 정리한다. 새 가드를 더 흩뿌리는 게 아니라 **모으는** 작업이다. (AC3)
- **단일 출처 선택지(작게 — 데모 범위):**
  - (권장) `web/src/lib/listings.ts` 신규에 작은 헬퍼. 예: `buyerListingsQuery(supabase, columns)` = `supabase.from('listings').select(columns).eq('status', LISTING_STATUS.ON_SALE)` 까지 적용한 빌더 반환. 두 페이지가 이걸 호출 → 이후 각자 `.ilike`/`.eq('id', id)`/`.order`/`.maybeSingle` 등 체이닝. "판매중만"은 한 곳에서만 정의됨.
  - (대안·더 가벼움) `constants.ts`에 `BUYER_VISIBLE_STATUS = LISTING_STATUS.ON_SALE` 상수 한 개 + 짧은 주석. 두 페이지가 `.eq('status', BUYER_VISIBLE_STATUS)`. 헬퍼보다 추상화 적음.
  - 둘 중 **헬퍼 방식**이 "단일 규칙이 한 함수에서 비롯"을 더 분명히 표현하나, 과하면 상수 방식으로 충분. dev가 코드량·명료성 보고 택1. **레이어 신설·repository 패턴 등 과한 추상화 금지.**
- **이중 방어를 유지하는 이유(제거 금지):** RLS만으로도 구매자(non-own)에겐 sold가 안 보이지만, 판매자가 본인 sold를 구매자 경로 URL로 열면 RLS `own`으로 통과 → 앱 `.eq('on_sale')`이 막는다. 단일출처화하되 **이 강제 자체는 두 경로 모두에 유지**돼야 한다(헬퍼/상수를 통해). [Source: 3-2 Dev Notes(72-73)]

### CM3 — sold 전환 즉시 비노출(데이터 정합)
- 정의: on_sale→sold 전환 후 구매자 재조회 시 **즉시** 목록·상세에서 사라져야 한다. [Source: epics.md(409-411)]
- 메커니즘: 페이지가 매 요청 DB를 다시 읽으면(동적 렌더) 자동 충족. RLS가 sold를 구매자에게 안 주므로, 재조회만 동적이면 잔존 불가.
- 위험: Next.js가 페이지를 **정적 캐시**하면 옛 on_sale 스냅샷이 남아 CM3 위반. → Task 3에서 동적 렌더 보장.
- **실검증(필수):** Supabase MCP로 시드 on_sale 1건을 임시 sold 전환 → 재조회로 사라짐 확인 → **원복**. (구매완료 처리 2-4 UI를 쓰지 않고 DB 직접 전환으로 빠르게 CM3만 격리 검증. baseline은 원복으로 보존.)
