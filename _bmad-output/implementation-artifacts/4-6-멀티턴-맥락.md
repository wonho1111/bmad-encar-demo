# Story 4.6: 멀티턴 맥락 (FR18)

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 구매자,
I want 직전 대화에 이어 후속 질문을 하고 싶다,
so that "그 중 더 싼 거", "그럼 흰색은?" 같이 앞 대화를 가리키는 맥락 대화를 한 입력창에서 자연스럽게 이어간다.

> **이 스토리의 본질 — "서버는 기억하지 않는다. 클라이언트가 들고 온 맥락을 후속 질의에 녹여 넣는다."** 4.5까지 `/ai/search`는 **단일 `query`만** 본다(`context`는 4.1에서 받아두되 무시). 그래서 "패밀리카 추천해줘" 다음에 "그 중 더 싼 거"라고 하면, 서버엔 "그 중"이 무엇인지 알 도리가 없다(서버 무상태 — AI 대화 이력을 서버·DB에 저장하지 않음). 4.6은 이 단절을 메운다: **클라이언트가 직전 대화(이전 질의/답변 요약)를 `context`에 담아 후속 질의와 함께 보내면**, 서버는 그 맥락을 읽어 **"그 중 더 싼 거" → "패밀리카로 무난한 거 중 더 싼 거"** 처럼 **혼자서도 말이 되는 독립 질의(standalone query)로 재작성(맥락화)** 한 뒤, 그 재작성된 질의를 **기존 그래프(라우터→A/B/C→answer)에 그대로** 흘린다. 핵심은 (1) **재작성 노드 1개를 그래프 진입 앞단에 얹는 것**과 (2) **서버·DB에 그 어떤 대화 이력도 저장하지 않는 것**(새로고침/세션 종료 시 초기화 — FR18·NFR4 무상태)이다.
>
> *멀티턴(multi-turn): 한 번 묻고 끝이 아니라 앞 대화를 이어 후속 질문을 주고받는 대화.*
> *무상태(stateless): 서버가 요청 사이에 아무 상태(대화 기억)도 보관하지 않음 — 필요한 맥락은 매 요청에 클라이언트가 동봉.*
> *맥락화/질의 재작성(query contextualization/rewrite): 앞 대화를 참고해 "그 중 더 싼 거"처럼 생략·지시가 섞인 후속 질의를, 그것만 봐도 뜻이 통하는 완결된 질의로 다시 쓰는 것.*

## Acceptance Criteria

1. **(AC1 — `context` 스키마 강제, 서버 무상태 입력 계약)** `SearchRequest.context`가 **원소 타입·최대 길이를 강제하는 구조**로 바뀐다(현재 `list | None`은 원소 타입·길이 제한이 없어 대용량 중첩 배열을 그대로 수용 — deferred-work 4.5 #1). 한 턴(turn)을 `{role: "user"|"assistant", content: str}` 형태로 정의하고(또는 동등한 최소 스키마), **최대 턴 수(예: 12)와 각 content 최대 길이(예: 2000자)를 Pydantic으로 제한**한다. 빈/누락 `context`(`None` 또는 `[]`)는 **단일 턴**으로 정상 동작한다(4.5까지의 동작과 동일 — 회귀 0). [Source: deferred-work 4.5 #1; api/app/schemas/ai.py:28-29]
2. **(AC2 — 맥락 반영, FR18 핵심)** `context`가 비어있지 않은 후속 질의가 오면, 서버가 **그 맥락을 반영해 답한다**. 구현은 **질의 맥락화(재작성) 노드**(`app/graph/contextualize_node.py` 신규, 또는 `run_search`에 맥락화 단계 추가)로 한다 — 직전 대화 + 현재 `query`를 받아 **혼자서도 뜻이 통하는 독립 질의(standalone query)** 를 만들고, 그 재작성 질의를 **기존 라우터→A/B/C→answer 그래프에 그대로** 흘린다. 예: context=[패밀리카 추천 대화] + query="그 중 더 싼 거" → 재작성 "패밀리카로 무난한 차 중 더 저렴한 매물" → 경로 B(또는 A) 정상 처리. [Source: epics.md#Story 4.6(525-535); FR18]
3. **(AC3 — 서버·DB 미저장, 무상태 검증)** 4.6 어디에서도 **AI 대화 이력을 서버 메모리(모듈 전역 등)나 DB에 저장하지 않는다**. `/ai/search`는 매 요청을 독립적으로 처리하고, 맥락은 **오직 요청 본문의 `context`** 에서만 온다. 새 마이그레이션·새 테이블·전역 대화 캐시를 만들지 않는다(AI 대화 전용 테이블 없음 — architecture "무상태 전제"). 같은 서버 인스턴스라도 `context` 없이 보낸 요청은 직전 요청을 전혀 기억하지 못함을 테스트로 보인다(FR18 둘째 AC: 새로고침/세션 종료 시 초기화). [Source: architecture.md(52,398,421); epics.md#Story 4.6(537-539)]
4. **(AC4 — 단일턴 회귀 0)** `context`가 없거나 빈 단일턴 질의는 **4.5와 정확히 동일하게** 동작한다 — 맥락화 노드는 **맥락이 없으면 `query`를 그대로 통과**시킨다(불필요한 LLM 호출·재작성 금지). 기존 `/ai/search` 계약(200 `{answer, listings[]}` / 401 미인증 / 422 빈질의 / 400 SqlGuardError)과 라우터·경로 A/B/C 동작이 깨지지 않는다. [Source: 4-5 story AC3; api/tests/test_ai_search.py]
5. **(AC5 — 맥락화 안전장치·fail-loud)** 맥락화 노드는 LLM을 쓰되 **결정론적 안전장치**를 둔다 — (a) `context`가 비면 LLM 호출 없이 원 `query` 반환(비용·지연 0), (b) 재작성 결과가 비거나 형식이탈이면 **원 `query`로 안전 폴백**(조용한 빈 결과 금지, 4.3·4.5 fail-loud/fallback 철학 계승), (c) `GEMINI_API_KEY` 부재는 폴백이 아니라 `require()` fail-loud(단, 맥락 없는 단일턴은 LLM을 아예 안 부르므로 키 없이도 기존 경로는 살아야 함). temperature=0으로 재현성 확보. [Source: 4-3·4-5 _llm() 호출시점 키·fail-loud 패턴; 함정 #2 계승]
6. **(AC6 — 결정론적 검증 + 라이브)** 멀티턴 맥락화·단일턴 통과·`context` 스키마 검증을 **LLM 모킹 단위 테스트**로 네트워크 없이 검증한다(맥락 있으면 재작성 호출·없으면 통과 / 재작성 실패 시 원질의 폴백 / context 과대 입력 422 / 무상태: 서버가 직전 요청 안 기억). 추가로 **실 LLM·실 DB 라이브**로 대표 멀티턴 2건 이상("패밀리카 추천"→"그 중 더 싼 거", "3천만원 SUV"→"흰색만")이 맥락을 반영해 흐르고, `/ai/search` HTTP가 `{answer, listings[]}`를 돌려줌을 눈으로 확인한다. 전체 pytest(기존 74 + 신규) 통과. [Source: 4-5 AC6 검증 방식 계승; CLAUDE.md §6]

## Dev Notes

### ⚠️ 결정적 함정 — 반드시 숙지

**함정 #1 — 서버에 대화를 "기억"시키지 마라(무상태 위배 = FR18·NFR4 직접 위반).**
멀티턴을 구현한다고 모듈 전역 dict나 사용자별 캐시, 새 DB 테이블에 대화를 쌓고 싶은 유혹이 있다. **절대 금지.** 맥락은 **오직 요청 본문 `context`** 에서만 온다 — 서버는 매 요청을 독립 처리하고 끝나면 잊는다. 이래야 "새로고침/세션 종료 시 AI 대화가 초기화"(FR18 둘째 AC)되고, Vercel 무상태 서버리스(NFR4)와 정합한다. AC3 테스트가 "context 없는 요청은 직전을 기억 못 함"을 못 박는다.
[Source: architecture.md(52,398,421) 무상태 전제; epics.md#Story 4.6(537-539)]

**함정 #2 — 단일턴(맥락 없음)에서 불필요한 LLM 호출·재작성 금지(회귀·비용).**
`context`가 비면(None·[]) 맥락화 노드는 **LLM을 아예 부르지 않고 원 query를 그대로 반환**해야 한다. 이유: (1) 4.5까지의 단일턴 동작이 토씨 하나 안 바뀌어야 회귀 0(AC4), (2) `gemini-flash` 호출 비용·지연을 매 단일 질의에 더하면 데모 품질 저하, (3) 키 없이도 기존 경로가 살아야 한다(맥락화가 단일턴에 키를 요구하면 4.5 동작을 깬다). LLM 호출은 **오직 context가 있을 때만**.
[Source: 4-5 단일턴 라우팅; 4-3 _llm() 호출시점 키 로드; CLAUDE.md 단순화 원칙]

**함정 #3 — 재작성 결과를 그대로 믿지 마라(폴백 필수). 그리고 매물을 "지어내게" 하지 마라.**
맥락화도 LLM이라 환각·빈 출력·형식이탈이 가능하다. 재작성 결과가 공백이거나 호출이 실패하면 **원 query로 안전 폴백**(조용한 빈 결과 금지 — 4.3 sql_guard·4.5 router fallback 철학). 또한 재작성 프롬프트는 **"앞 대화에 없는 새 조건·매물을 지어내지 말고 지시어만 치환"** 하도록 박는다 — "그 중 더 싼 거"를 "1천만원 이하 빨간 스포츠카"처럼 없는 조건으로 부풀리면 엉뚱한 결과가 난다. temperature=0으로 재현성·보수성 확보.
[Source: 4-5 함정 #2 결정론 보정; 4-4 fail-loud; research §8]

**함정 #4 — 다운스트림 노드 시그니처를 건드리지 마라(라우터·경로는 단일 query만).**
맥락화의 출력은 **"혼자서도 말이 되는 독립 질의 한 문장"** 이다. 이 한 문장을 기존 `COMPILED_GRAPH.invoke({"query": ...})`에 그대로 넣으면, 라우터·sql_rag·doc_rag·guard·answer가 **평소처럼** 처리한다(그들은 맥락을 몰라도 됨 — 이미 완결된 질의를 받으니까). 그래서 4.6은 router_node·경로 노드·answer_node·SearchState를 **수정하지 않는다**. 맥락 처리는 전부 "그래프 앞단" 한 곳에 격리(관심사 분리 — 다운스트림 회귀 0).
[Source: 4-5 graph.py 구조; 노드 시그니처 node(query)->{answer,listings} 불변]

### 설계 결정 — 맥락 반영 방식 (확정 제안: **그래프 앞단 질의 재작성(contextualize) 전처리**)

- **채택: 질의 맥락화(재작성) 전처리.** `run_search(query, context)`가 그래프 호출 **전에** `contextualize_query(query, context)`로 "독립 질의"를 만들고, 그 결과를 기존 그래프에 넣는다. 장점: (1) 라우터·경로·answer 전부 불변(회귀 격리), (2) 무상태 자연 보장(맥락은 인자로만 흐름), (3) 재작성된 질의가 A/B/C 어느 경로로든 평소처럼 흐름(예: "그 중 더 싼 거"→재작성→B 정렬 추천, "흰색만"→재작성→A color 필터). architecture의 "질의+맥락 → 라우터 → ..."(421) 흐름과 정합 — "맥락"을 라우터 앞에서 query에 흡수시키는 가장 단순한 형태.
- **미채택(데모상): `context` 전체를 라우터·각 노드에 그대로 넘겨 노드마다 맥락 해석.** 모든 다운스트림 노드 프롬프트에 대화 이력을 끼워야 해 토큰·복잡도↑, 노드 4개 회귀 위험↑. 재작성 1곳 격리가 더 단순·안전(데모 목표 정합).
- **그래프 진입 노드 vs run_search 전처리:** 둘 다 가능. SearchState에 context를 싣고 `contextualize`를 entry node로 두는 정공법도 좋지만, **데모 단순성·그래프/테스트 영향 최소화** 위해 `run_search` 전처리를 우선 제안. dev 판단으로 진입 노드 방식 채택 시 SearchState 확장·테스트 조정하고 근거를 남길 것(결과 계약·무상태는 동일해야 함).

### 맥락화 프롬프트 설계 (요지)

```
너는 중고차 검색 대화의 "질의 재작성기"다.
아래 [직전 대화]를 참고해, [현재 질의]를 그것만 읽어도 뜻이 통하는
독립적인 중고차 검색 질의 한 문장으로 다시 써라.
규칙:
- "그 중", "그거", "아까 그", "방금" 같은 지시어를 직전 대화의 구체 내용으로 치환한다.
- 직전 대화에 없는 새 조건·차종·매물을 지어내지 않는다(있는 맥락만 흡수).
- 현재 질의가 이미 독립적이면 거의 그대로 둔다.
- 출력은 재작성된 질의 "한 줄"만. 설명·따옴표 없이.

[직전 대화]
{context를 user/assistant 턴으로 직렬화}
[현재 질의]
{query}
```
- temperature=0. 출력 한 줄 → `.strip()` 후 비면 원 query 폴백.
- context 직렬화는 마지막 N턴만(스키마가 이미 12턴 제한). 토큰 절약 위해 필요 시 최근 4~6턴만 써도 무방(데모).

- **함정 #2 단일턴 무비용:** context가 None·[]이면 LLM을 아예 안 부르고 원 query 반환 → 4.5까지 단일턴 동작 토씨 불변(회귀 0), 키 없이도 기존 경로 동작. `test_no_context_returns_query_unchanged_and_no_llm`이 LLM 호출 시 실패하도록 심어 검증.
