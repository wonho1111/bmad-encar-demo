## 잘된 점 (반복할 패턴)

1. **"프롬프트는 못 믿는다 — 결정론적 검증을 덧댄다"가 에픽 전반의 일관된 방어선.** 4-3·4-4·4-5·4-6·4-7 다섯 스토리가 모두 같은 원칙을 명시: LLM 출력은 항상 코드가 한 번 더 검사·보정한다. (예: sql_guard `validate_select_sql`, 라우터 A/B/C 외 값 보정, 맥락화 폴백, 멀티턴 주제전환 순수함수 가드.)
2. **이중 방어(defense in depth).** 읽기전용은 `sql_guard`(SELECT만) **+** `ai_readonly` 롤(쓰기 DB 거부) 두 겹. 한쪽이 뚫려도 다른 쪽이 막는다.
3. **단일출처(single source of truth) 철저.** `SELECT_COLUMNS`(경로 A·B 공유), 허용값을 `0002_listings.sql` CHECK와 프롬프트가 일치, FR11(sold 비노출)을 모든 AI 경로 WHERE에 직접 박음 — drift 차단.
4. **fail-loud, 조용한 빈 결과 금지.** 키 부재는 `require()`로 명확한 한국어 에러, 0건은 실패가 아니라 FR17 안내. 버그가 빈손으로 위장되지 않음.
5. **Claude 자체 E2E 검증이 실제로 작동.** Playwright로 로컬·preview에서 로그인→질의→카드까지 직접 확인(보호된 preview는 Vercel 공유링크 우회). "빌드 성공 ≠ 동작"을 E2E로 메움.
6. **데이터 기반 의사결정(Phase B).** 모델 선택을 감이 아니라 44항목×3회 실측 + 사전식 채점으로 결정. 채점 하니스 단위테스트(15개)로 채점기 자체의 오류도 방어.

## 어려웠던 점 / 반복된 함정

1. **배포 모델을 중간에 갈아엎음(Vercel→Cloud Run).** Python 의존성 번들이 Vercel 서버리스 250MB 한도를 초과(480MB)해 배포 실패 → 컨테이너(Cloud Run)로 전환. 웹(Vercel)/API(Cloud Run) 이원 배포 구조가 됨.
2. **LLM 비결정성이 운영 버그로 표출.** temperature=0이어도 라우팅·재작성이 흔들림. 멀티턴 주제전환 시 **옛 조건이 새 질의에 계승되는 오염**(중형세단→초보첫차에 중형 조건 잔존) → 안건2(프롬프트 리셋 + 결정적 코드가드)로 수습.
3. **운영에서야 드러난 크래시 — `uuid > int`.** "지금 말고 다른 거 더 보여줘"에서 LLM이 id(UUID)로 페이지네이션 시도 → `psycopg UndefinedFunction(uuid > integer)` 500. 게다가 500이 CORS 미들웨어 바깥에서 생성돼 **CORS 헤더 누락 → 브라우저가 진짜 원인 대신 "연결 실패"로 오인**(이중 혼란). 프롬프트 규칙 + 라우트 예외변환으로 수습.
4. **경로 경계의 모호함(Path B 범위).** 금융·세금·보험 일반지식을 어디로? → "이 답이 보여줄 매물을 바꾸나?" 기준으로 C(거절)로 확정(안건3). 부수효과로 일부 상담형 질문(전기차 유지비 등)도 C로 가는데, **이는 의도된 동작으로 사용자 확정**(버그 아님).
5. **외부 의존 블로커 — Gemini 선불크레딧.** 임베딩 backfill·A/B 실측이 크레딧 0이면 429로 막힘. 코드 문제가 아니라 계정 단위 선결 조건이라 작업이 한때 보류됨.
6. **새 스택 학습 비용.** FastAPI+LangGraph(첫 별도 백엔드)·pgvector·Gemini content가 list로 오는 변형 처리·LangChain 콜백 미전파(토큰 집계를 `_generate` 모니터패치로 우회) 등 프레임워크 함정.

## 핵심 교훈

1. **LLM은 "확률적 제안기", 시스템 신뢰는 코드가 책임진다.** Epic 4의 정체성. 모든 LLM 출력 경계에 결정론적 검증·폴백·보정을 둔 게 운영 안정성의 핵심이었다. 향후 AI 기능도 이 패턴을 기본값으로.
2. **번들/런타임 제약은 착수 전에 못 박아라.** Vercel 250MB 한도를 늦게 만나 배포를 갈아엎었다. 새 런타임·무거운 의존성은 PoC 단계에서 배포 타깃의 한도부터 확인.
3. **운영 버그는 "흔치 않은 사용자 표현"에서 나온다.** "다른 거 더", 멀티턴 주제전환처럼 정상 경로 밖의 자연어가 크래시·오염을 유발. 질의셋에 경계·전환 케이스를 의도적으로 심어야 잡힌다(Phase B 질의셋이 이걸 반영).
4. **에러는 CORS 안쪽에서 만들어야 프런트가 진실을 본다.** 전역 핸들러 500은 헤더가 빠져 원인을 가린다. 예외를 라우트에서 변환.
5. **모델·아키텍처 선택은 측정으로 끝내라.** Phase B에서 "더 싼 모델이 품질 동급인가?"를 실측으로 답했고, 회귀 게이트로 보수적으로 결정. 감 대신 근거.
