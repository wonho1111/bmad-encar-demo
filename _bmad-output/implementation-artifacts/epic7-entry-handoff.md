## 3. Epic 7 착수 전 — 사용자가 직접 준비할 것

| 준비물 | 왜 / 어디서 |
|---|---|
| Android Studio + Flutter SDK 설치 | 모바일 빌드·실행 기반 / 로컬 PC |
| **실제 폰 USB 연결** | 이 PC RAM 6GB라 에뮬레이터 부팅 불가(Pixel_7 offline) → 실기기 필요. 개발 중엔 Flutter 웹 빌드 + Playwright 스모크로 대체 가능 |
| mobile MCP 승인 | 최종 네이티브 자체검증(mobile-mcp v0.0.60 연결됨) / Claude Code 프롬프트 |
| `.env` 키값 | 모바일도 Supabase·`/ai/search` 계약 재사용 → 동일 키 주입 |

> 모바일 테스트 전략: 개발 중 **Flutter 웹 + Playwright 스모크** → Epic 7 종료 후 **실제 폰 USB로 mobile-mcp 자동검증**.

## 4. Epic 7 슬라이스 계획 (epics.md 기준)

- **범위(AR9)**: 모바일 = **구매자 + 판매자 여정만**. **관리자는 web 전용(제외).** web이 확정한 Supabase 스키마/RLS + `/ai/search` 계약 **재사용**(코드가 아니라 계약·규칙이 이월).
- **슬라이스**: 7.1 골격(flutter create+인증 연동) → 7.2 구매자(목록·상세·AI검색) → 7.3 판매자 등록(15필드 폼) → 7.4 본인매물 관리+구매완료 → **7.5 SM1 통합 시연 검증**.
- **크기 통제 기준**: "이 화면이 SM1 시나리오 ①②의 한 스텝인가?" 아니면 컷(풀패리티 금지).
- **이월되는 자산**: 스키마·RLS·`/ai/search` 응답 포맷 + `nav-ia-rules.md`의 역할별 내비/정보구조/복귀 규칙. **React/CSS 코드는 이월 안 됨(위젯 트리 새로 짬).**

### 7.1 골격 들어가면 바로 쓸 입력
1. `nav-ia-rules.md` — §1 역할×진입점(로그인 후 역할별 착지), §4 공유 계약
2. 공유 계약: `0001~0009` 마이그레이션(스키마·RLS·트리거), `/ai/search` 응답 `{answer, listings[]}`
3. buyer≠seller 무결성(3중 차단)은 DB가 보장 → 모바일에서도 그대로 유효
