# Story 2.5: 샘플 매물 시드

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 개발자/시연자,
I want 텍스트 필드가 채워진 샘플 매물 다수를 시드하고 싶다,
so that 탐색·AI 검색을 실제 데이터로 검증·시연할 수 있다.

## Acceptance Criteria

1. **(≥35건 생성)** `supabase/seed.sql`이 실행되면 `0002_listings.sql`의 CHECK 목록값을 따르는 샘플 매물 **≥35건**이 생성된다(사진 없음). 모든 행의 INSERT가 성공하고 CHECK 위반은 0이다. [AR7·FR5]
2. **(embedding NULL)** 모든 샘플 매물의 `embedding`은 우선 `NULL`이다 — Epic 4(Story 4.2)에서 일괄 적재(backfill)한다. 본 스토리에서 임베딩을 계산·삽입하지 않는다. [AR7·Epic 4 분리]
3. **(허용값 단일출처 정합)** 시드의 고정 목록 6필드(manufacturer·body_type·color·fuel·transmission·region) 값은 **UI 드롭다운·Text-to-SQL 허용값과 바이트 단위로 동일**하다 = `0002_listings.sql` CHECK ∩ `web/src/lib/constants.ts`(LISTING_OPTIONS) ∩ `architecture.md` 확정표. 수치 필드(year·seats·price·mileage·displacement)는 `LISTING_RANGES`/CHECK 범위를 만족한다. [AR7·정합성 주의]
4. **(분포 풍성)** 차종(body_type)·가격대(price)·연료(fuel)·지역(region)이 고루 분포해 필터·AI 검색 시연이 풍성하다. 특정 한 값에 쏠리지 않고 다수 카테고리에 걸쳐 분포한다. [AR7]
5. **(자연어 코퍼스)** `description`과 `options` 텍스트는 Epic 4 RAG 코퍼스①(설명·옵션 임베딩 대상)이 되므로, 각 매물의 실제 사양·상태를 반영한 **자연스러운 한국어**로 충실히 작성한다(빈/형식적 문자열 금지). [architecture.md L181-182·OI6]
6. **(유효 판매자 참조 + 멱등)** 모든 샘플 매물의 `seller_id`는 유효한 판매자 계정(`profiles.id`, role=seller)을 참조한다. 시드용 판매자 계정이 없으면 seed.sql에서 함께 생성한다(기존 admin 시드 패턴 재사용). seed.sql 재실행 시 매물이 중복 누적되지 않도록 **멱등**하게 작성한다(시드 전용 판매자의 기존 매물 정리 후 재삽입, 또는 동등 전략). 기존 admin 시드(Story 1.5)와 기존 데이터를 깨지 않는다. [AR7·멱등성]

> **범위 주의(반드시 준수):** 이 스토리는 **`supabase/seed.sql`에 샘플 매물 섹션 추가 + 라이브 DB 적용·검증**까지다.
> - **마이그레이션·DB 스키마는 2-1에서 완료** — `0002_listings.sql`을 비롯한 어떤 마이그레이션도 건드리지 않는다(시드는 **데이터**지 스키마가 아니다 → `apply_migration` 미사용, `execute_sql`로 적용).
> - **기존 admin 시드(Story 1.5) 섹션을 덮어쓰지 말 것** — 같은 파일에 매물 섹션을 **추가/병합**한다.
> - **embedding backfill·가이드 문서 코퍼스는 Epic 4(4.2)** — 만들지 않는다(embedding은 NULL).
> - **구매자 화면/필터/상세는 Epic 3** — 시드는 데이터만 채운다(화면 변경 없음).
> - **constants.ts·마이그레이션 등 허용값 단일출처를 수정하지 말 것** — 시드 값을 단일출처에 **맞춘다**(drift 시 시드 값을 고친다, 단일출처를 고치지 않는다).

### 멱등 전략 (권장)

- 시드 전용 판매자 계정(예: `seller-seed@test.com`)을 멱등 생성 → 그 계정 소유 매물만 `delete` 후 재삽입.
- 장점: 재실행 시 정확히 시드 매물만 갱신, 다른 계정·잔여 테스트 데이터·admin 시드는 절대 건드리지 않음.
- `ON CONFLICT`는 id가 `gen_random_uuid()` 기본값이라 충돌 키가 없어 부적합 → "삭제 후 삽입"이 단순·안전.

### 테스트 방식 (CLAUDE.md §6)

- 시드는 화면이 아니라 **DB 검증 중심**: `execute_sql`로 적용 후 SQL group-by/count로 건수·분포·CHECK·embedding NULL·판매자 참조를 검증.
- CHECK 위반이 나면 **시드 값을 단일출처에 맞게** 고친다(단일출처를 고치지 않는다).
- 적용 상태로 두어도 됨(데모 데이터) — 무엇을 적용했는지 Completion Notes에 명확히 기록.
