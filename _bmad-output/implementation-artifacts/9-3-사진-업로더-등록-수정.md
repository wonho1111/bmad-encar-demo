# Story 9.3: 사진 업로더 (등록/수정)

Status: ready-for-dev

## Story

As a 판매자,
I want 매물 등록·수정 시 사진을 여러 장 올리고 대표를 지정하길,
so that 구매자가 내 차를 사진으로 먼저 본다.

**범위: web(`web/src/app/(user)/sell/`)만.** Flutter 앱 업로더는 이 스토리 밖이다 — 근거는 [Dev Notes §0 범위 경계](#0-범위-경계-먼저-읽을-것).

## Acceptance Criteria

원문 출처: `_bmad-output/planning-artifacts/epics-increment-2026-07-12.md:489-506` (Story 9.3). AC1~AC7이 원문, AC8~AC10은 9.1·9.2 코드리뷰 이월 부채가 이 스토리를 **트리거로 지목**해서 추가된 것(`docs/tech-debt.md` #43·#47·#49·#51).

1. **업로드 기본** — 매물 등록/수정 폼에서 드롭존 + "N/10" 카운터로 **최대 10장 · 장당 원본 5MB**까지 업로드된다. **첫 장이 대표**가 되고, 순서 변경·대표 재지정이 된다. (FR26, UX-DR13)
2. **사진은 선택** — 사진 0장이어도 등록·수정이 정상 완료된다. 폼에 안내 문구를 그대로 노출한다: **"사진은 선택이에요. 없어도 등록되지만, 있으면 문의가 훨씬 잘 와요."**
3. **실패는 폼을 막지 않는다** — 업로드 실패·용량초과·미지원 포맷은 **해당 사진 항목의 인라인 오류 + 재시도 버튼**으로 처리하고, **폼 제출 자체는 차단하지 않는다**. 대표사진을 삭제하면 **다음 순서 사진이 자동으로 대표 승격**한다.
4. **업로드 전 클라이언트 리사이즈/압축** — 원본 5MB 상한과 별개로, 저장본을 카드/갤러리에 적합한 크기·품질로 **다운스케일해서 올린다**. (앱은 서명 **원본**을 받으므로(ADR-IMG-01) 목록에서 수 MB 다중 다운로드가 나면 NFR7 위반) 저장본은 **긴 변 ≤1600px · WebP · quality 0.82**로 고정한다 — 근거·구현법은 Dev Notes §4.
5. **경로 규약** — 저장 경로는 9.1 규칙 **`{user_id}/{listing_id}/{filename}`** 을 따른다. **신규 등록은 `listings` insert로 `listing_id`를 받은 뒤** 업로드한다(스테이징 경로 없음).
6. **처리중·실패는 로컬 상태** — 업로드 진행/실패는 업로더 본인 세션의 화면 상태일 뿐이고, **wire(DB·JSON)에는 `null` 또는 확정된 값만** 나간다(I8). `listing_images` 행은 **업로드가 실제 성공한 사진에 대해서만** 만든다.
7. **이탈 가드** — 변경(특히 업로드한 사진)이 있는 채 이탈하려 하면 확인 다이얼로그가 뜬다: **"저장하지 않고 나가시겠어요? 작성한 내용이 사라져요."** + [계속 작성]/[나가기]. **변경이 없으면 경고 없이 이탈**한다. (UX-DR23)
8. **대표 교체는 2문장으로** — 대표를 바꿀 때 `is_cover`를 **단일 UPDATE로 뒤집지 않는다**(부분 유니크 인덱스에 걸려 `duplicate key`로 죽는다 — 도커 실측, #47-1). **① 해당 매물 전체 `is_cover=false` → ② 대상 1장만 `true`** 순서로 실행하고, 이 순서가 필요한 이유를 코드 주석에 남긴다.
9. **10장 상한을 클라이언트가 먼저 막는다** — DB 트리거는 **INSERT만** 보고 동시 삽입 경합에서 새는 것이 실측돼 있다(#43·#49). 따라서 (a) UI가 10장 초과 선택을 애초에 막고, (b) **행 INSERT는 순차(직렬)로** 보내 병렬 경합을 만들지 않는다. ⚠️ 이건 **UX 층의 1차 차단**이지 강제가 아니다 — 강제는 계약대로 서버(트리거·버킷 설정)에 남는다(`docs/conventions.md:193` "클라 검증은 우회 가능하다"). 클라가 막는다고 서버 검증을 빼지 말 것.
10. **삭제가 실제로 지워지는지 실측** — 사진 삭제는 **`listing_images` 행 + Storage 오브젝트 둘 다** 제거해야 한다. 그런데 **인증 사용자의 Storage DELETE/LIST가 RLS·GRANT가 맞는데도 실패하는 현상(#51)이 열려 있다.** 착수 시 **원격에서 먼저 재현 여부를 측정**하고(Task 0), 결과에 따라 경로를 정한 뒤 구현한다. **측정 없이 "된다/안 된다"를 코드나 문서에 적지 않는다.**

## Dev Notes

### 0. 범위 경계 (먼저 읽을 것)

**이 스토리는 web만 만든다.** AC 원문은 "매물 등록/수정 폼"이라고만 하고 플랫폼을 못 박지 않지만, 이 레포의 증분 구조는 **Epic 9~15 = web·api, Epic 16 = Flutter 앱 미러링**이다(`epics-increment-2026-07-12.md` Epic 16: 16-1 토큰·내비 … 16-2 이미지 카드 재설계-앱). 또 앱에는 사진 선택 패키지(`image_picker` 등)가 `app/pubspec.yaml`에 **아예 없다** — 앱 업로더는 새 의존성·권한(카메라/갤러리) 설계가 붙는 별개 크기의 일이다.

⚠️ **다만 Epic 16에도 "앱 사진 업로더" 스토리가 없다** — 즉 현재 계획대로면 앱 판매자는 사진을 못 올린다. 이건 이 스토리가 조용히 삼킬 문제가 아니라 **계획의 구멍**이다. Task 7에서 `docs/tech-debt.md`에 등재하고, 사용자 판단(Epic 16에 스토리 추가 vs 앱은 읽기 전용 유지)을 받는다.

**이 스토리 밖(하지 말 것):** 목록 카드 렌더(9.4) · 상세 갤러리(9.5) · AI 응답 카드 사진(9.6) · 시드 사진(9.7) · 이미지 모더레이션(Epic 9 Non-goal).

### 1. 지금 코드가 어떤 상태인가 (실제로 읽고 적음)

| 파일 | 현재 상태 | 이 스토리가 하는 것 |

| `web/src/app/(user)/sell/[id]/edit/page.tsx` | 서버 컴포넌트. 본인 매물만 조회, `sold`면 수정 차단. `SellForm mode="edit"` 재사용 | 기존 사진 조회 + **서버측 서명 URL 발급**해서 전달 |

**업로드는 브라우저(anon key + Storage RLS)에서 한다.** `SellForm`이 이미 브라우저 클라이언트로 insert하고 있고, `storage_path` 위조는 마이그레이션 `0013`의 트리거가 **DB에서** 막는다(소유자를 클라 입력이 아니라 `listings`에서 조회해 대조). `service_role`은 어디에도 두지 않는다.

### 2. 계약 (값의 정본은 `docs/conventions.md` §10 — 여기 사본을 만들지 말 것)

- 버킷 `listing-images`(비공개) · 경로 `{user_id}/{listing_id}/{filename}`(첫 세그먼트=소유자) · 상한 10장/5MB/MIME 3종 · TTL 3600
- `listing_images(id, listing_id, storage_path unique, sort_order, is_cover, credit)` — `supabase/migrations/0012_listing_images.sql`
- **대표는 매물당 최대 1장**(부분 유니크 인덱스 `listing_images_one_cover_per_listing`)
- `ListingCard.image_url`은 **null 또는 빈 문자열 모두** "사진 준비중"으로 취급(§4) — 이 스토리는 값을 쓰기만 하고 렌더는 9.4/9.5 몫
- ⚠️ **UX 목업 HTML(`mockups/forms-2.html:882`)의 "JPG · PNG, 장당 최대 20MB"는 낡았다.** 정본은 5MB · MIME 3종(WebP 포함)이다. **목업 문구를 그대로 베끼지 말 것**

### 4. 라이브러리 결정 — **새 의존성 0개**

`web/package.json`의 dependencies는 `@supabase/ssr`, `@supabase/supabase-js`, `next`, `react`, `react-dom`뿐이다. 이 스토리는 **여기에 아무것도 더하지 않는다.**

- **리사이즈/압축**: `canvas.drawImage()` → `canvas.toBlob(cb, 'image/webp', 0.82)`. 브라우저 네이티브다. `browser-image-compression` 같은 패키지 불필요
- **드래그 순서변경**: HTML5 native drag&drop(`draggable` + `dragstart`/`dragover`/`drop`). `dnd-kit`·`react-beautiful-dnd` 불필요
- **왜 WebP인가**: 버킷 `allowed_mime_types`에 이미 포함돼 있고, 같은 화질에서 JPEG보다 작다. `toBlob`이 WebP를 못 만드는 환경이면 `image/jpeg` 0.85로 폴백
- **왜 1600px인가**: 상세 갤러리(9.5)가 대표 사진을 크게 쓰고, web은 Next.js Image로 파생 변형을 만들지만 **app은 서명 원본을 그대로 받는다**(ADR-IMG-01). 1600px WebP면 보통 200~400KB — 목록 10장 다중 다운로드에서도 NFR7 안쪽

⚠️ **Next 16 함정**: web 작업 전 `node_modules/next/dist/docs/`를 먼저 읽는다(`web/AGENTS.md`). 훈련데이터의 Next와 다르다. Tailwind는 **v4** — `tailwind.config.js`가 아니라 CSS `@theme`.
