# Story 7.4: 판매자 — 본인 매물 관리·구매완료 (재현 FR6·8)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 판매자(모바일),
I want 앱에서 내 매물을 목록으로 보고 수정·삭제하고 구매완료 처리하고 싶다,
so that 이동 중에도 거래를 마무리한다(FR6·FR8 재현). 본인 매물만 만질 수 있고, sold는 구매자에게 안 보인다(FR11, 7.2에서 이미 적용).

## Acceptance Criteria

1. **(AC1 — 본인 매물 목록 조회, FR7 신뢰성)** 판매자로 로그인한 앱에서 "내 매물 관리" 화면에 진입하면 **본인이 등록한 매물만**(`seller_id == 현재 user.id`) 최신순으로 나열된다. 각 행에 요약(제조사·모델·연식·가격)과 **상태 배지(판매중/판매완료)**, 관리 동작(수정·삭제·구매완료)이 보인다. 조회 실패(네트워크·RLS·DB)는 "없음"과 구분해 한국어 오류로 표시한다(방금 등록·관리한 매물이 사라진 것처럼 오인되지 않게).
   - *근거*: web `(user)/sell/page.tsx`의 "내가 등록한 매물" 섹션 이식 — `seller_id` 명시 필터 + `listings_select_own` RLS 이중. RLS만으로는 "남의 on_sale"이 섞이므로 앱 쿼리에서 `seller_id` 필터 필수.

2. **(AC2 — 본인 매물 수정, FR6)** 본인 매물의 **수정**을 누르면 7.3 등록 폼(`listing_form.dart`의 `validateAndBuildListing`·15필드·검증)을 **재사용**한 수정 화면에 기존 값이 채워진 채 열리고, 제출하면 `listings` 행이 **UPDATE**된다(같은 15필드·검증·한국어 거절). `seller_id`·`status`는 폼에서 바꾸지 않는다(소유권 이전·상태 전환은 수정 범위 밖). **판매완료(sold) 매물은 수정 진입을 막는다**(목록에서 수정 버튼 숨김 + 화면 진입 시 한 번 더 차단 = 이중 방어). 남의 매물 수정은 RLS(`listings_update_own`)가 0행/42501로 차단.
   - *근거*: web `(user)/sell/[id]/edit/page.tsx`(본인+on_sale만 조회해 폼에 채움, sold·타인은 한국어 차단) + `SellForm.tsx` edit 모드(UPDATE). 0002 `listings_update_own`(`auth.uid()=seller_id`).

3. **(AC3 — 본인 매물 삭제, FR6)** 본인 매물의 **삭제**를 누르면 실수 방지 확인 후 `listings` 행이 **DELETE**된다(상태 무관 — 정리 목적). 남의 매물 삭제는 RLS(`listings_delete_own`)가 **에러가 아니라 0행**으로 막으므로, 삭제된 행 수가 0이면 "본인 매물만 삭제할 수 있습니다" 한국어 안내. 성공 시 목록에서 즉시 제거.
   - *근거*: web `ListingActions.tsx` `handleDelete`(`.delete().eq('id',id).select('id')` → 0행이면 거부 안내) 이식. 0002 `listings_delete_own`.

4. **(AC4 — 구매 완료, FR8)** 본인 **판매중(on_sale)** 매물의 **"구매 완료"**를 누르면 실수 방지 확인 후 `status`를 **`sold`로 전환**한다. UPDATE payload는 **`status`만** 보낸다(`seller_id`·다른 필드 위조/부수변경 차단). **전제조건 `.eq('status','on_sale')`** 을 걸어, 이미 sold거나 화면이 낡아 그새 바뀐 매물을 다시 누르거나 강제 재전환을 시도하면 0행이 돼 거부된다(서버측 빗장). 0행/42501이면 "본인 판매중 매물만 구매 완료 처리할 수 있습니다" 한국어 안내. 성공 시 배지가 "판매완료"로 갱신되고, **sold 매물은 구매자 목록/상세에 비노출**(FR11 — 7.2 `_buyerQuery`에서 이미 강제, 본 스토리는 회귀만 확인). status 값(`on_sale`/`sold`)·전이 규칙은 web `LISTING_STATUS`·마이그 0002 CHECK와 **바이트 일치**.
   - *근거*: web `ListingActions.tsx` `handleComplete`(`.update({status:SOLD}).eq('id',id).eq('status',ON_SALE).select('id')`) 이식. 0002 `status check (status in ('on_sale','sold'))` + `listings_update_own`.

5. **(AC5 — 역할 가드)** **판매자(seller)만** "내 매물 관리"·수정 화면에 진입한다. buyer·admin은 차단(admin은 7.1 AuthGate에서 이미 모바일 제외). 홈(`home_screen.dart`)에서 판매자에게만 "내 매물 관리" 진입 동선을 노출하고, 화면 진입 시 역할을 한 번 더 확인한다(이중 방어, 7.3 `sell_screen.dart`의 `sell_role_blocked` 패턴 재사용).
   - *근거*: 7.1 `user_role.dart`·`AuthGate`. web `(user)/sell/layout.tsx requireRole(seller)`. nav-ia §2 판매자 전용(등록·관리).

6. **(AC6 — 검증·테스트)** `flutter analyze`(0 error)·`flutter test`(컨트롤러·레포 순수 로직 단위 테스트)·`flutter build web --dart-define-from-file=.env.json`이 통과한다. 가능하면 판매자 토큰으로 본인 매물 **수정·삭제·sold 전환 성공** + **남의 매물 차단(42501/0행)** + **이미 sold 재전환 거부(0행)**를 live Supabase HTTP(REST) 스모크로 실증(생성/변경한 테스트 행은 정리, 못 하면 표기). 브라우저 픽셀 렌더는 시스템 라이브러리 미비 시 빌드+테스트로 갈음(그 사실 표기).
   - *근거*: 7.3 테스트 전략(실폰·에뮬 없음 → build web + test + live HTTP 스모크). 프롬프트 §검증.

### 단일 출처 / 계약 (drift 금지)

- **status enum·전이·CHECK의 출처** = `supabase/migrations/0002_listings.sql`:
  - `status text not null default 'on_sale' check (status in ('on_sale','sold'))` — **두 값만 허용**. "그 외 값" 전환은 CHECK가 차단.
  - 전이 규칙(본 서비스): **`on_sale → sold` 단방향만**(구매 완료). `sold → on_sale` 되돌리기 없음(web에도 없음 — markSold만 존재). 그래서 markSold는 `.eq('status','on_sale')` 전제조건을 둬 재전환·역전환을 0행으로 막는다.
  - 소유권 RLS: `listings_update_own`/`listings_delete_own`(`auth.uid()=seller_id`) — 타인 매물 수정·삭제·sold 전환을 **에러(42501) 또는 0행**으로 차단. `with check (auth.uid()=seller_id)`가 `seller_id` 재지정(남에게 넘기기)도 막는다.
  - SELECT RLS: `on_sale 공개 ∪ 본인 전부 ∪ admin` OR 결합 → **앱 쿼리에서 `seller_id` 명시 필터 필수**(안 하면 "남의 on_sale"이 본인 목록에 섞임). web sell/page.tsx 주석과 동일 함정.
