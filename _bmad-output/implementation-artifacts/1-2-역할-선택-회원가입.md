## Story

As a 신규 사용자,
I want 이메일·비밀번호로 가입하면서 구매자/판매자 역할을 선택하고 싶다,

## Acceptance Criteria

1. **(스키마)** `0001_profiles` 마이그레이션이 적용되면, `profiles(id → auth.users, role check(buyer/seller/admin), status check(active/suspended))` 테이블이 존재한다. (단일 출처 = `supabase/migrations/0001_profiles.sql`)
2. **(가입 → 계정+프로필 동시 생성)** 가입 화면에서 이메일·비밀번호·역할(구매자/판매자)을 입력해 제출하면, Supabase Auth 계정과 `profiles` 행(`role`=선택값, `status='active'`)이 **함께** 생성된다.
3. **(중복 이메일 거절)** 이미 가입된 이메일로 다시 가입을 시도하면 **한국어 오류 메시지**로 거절된다.
4. **(역할 고정)** 역할은 가입 시 1개로 고정된다(둘 다 하려면 계정 2개). 가입 경로에서 `admin` 역할은 선택할 수 없고, 우회 입력해도 서버(트리거)에서 admin으로 저장되지 않는다.

### RLS 동거 원칙 (왜 0001에 RLS를 함께 넣는가)
- 아키텍처 결정: **각 테이블 RLS는 그 테이블 마이그레이션에 동거**시킨다(`0001`=profiles RLS). 단일 `rls_policies` 묶음으로 몰면 RLS가 늦게 적용돼 선행 기능이 막히므로 **금지**. 관리자 전권 교차 정책만 `0005_admin_policies`로 분리. [Source: architecture.md#Authentication-&-Security, #Complete-Project-Directory-Structure(line 329), epics.md(AR2)]

### 🔥 치명적 함정 #1 — profiles RLS 자기참조 무한재귀
- "관리자는 전체 profiles 조회" 정책을 짤 때 정책 안에서 `profiles`를 직접 서브쿼리하면(`exists(select 1 from profiles where id=auth.uid() and role='admin')`) **RLS가 자기 자신을 재귀 평가**해 `infinite recursion detected in policy` 에러가 난다(Postgres RLS 대표 함정).
- **해결:** `SECURITY DEFINER` 함수 `public.is_admin()`로 분리한다 — 이 함수는 정의자 권한으로 RLS를 우회해 profiles를 읽으므로 재귀가 끊긴다. 정책은 `using (public.is_admin())`만 호출. (아래 참조 SQL 그대로 사용 권장)

### 🔥 치명적 함정 #2 — Supabase 이메일 확인(email confirmation)과 중복 거절(AC3)
- Supabase Auth는 **"Confirm email" 설정이 켜져 있으면** 이미 가입된 이메일로 `signUp`해도 **에러를 던지지 않고**(이메일 열거 공격 방지) 모호한 성공 응답(`user` 객체에 `identities: []`)을 반환할 수 있다 → 단순 try/catch로는 AC3(중복 거절)가 안 잡힌다.
  - **데모 권장:** Supabase 대시보드에서 **이메일 확인을 끄면**(아래 "사용자 직접 처리"), 중복 가입이 명시적 에러(`user_already_exists`)로 오고 가입 즉시 세션이 생겨 1.3 로그인 흐름과 매끄럽다.

### 🔥 치명적 함정 #3 — admin 역할 우회 차단(AC4)
- `profiles.role` CHECK는 `admin`을 **허용**한다(Story 1.5 시드 관리자 때문). 따라서 CHECK만으로는 가입 경로 admin을 막지 못한다.
- **방어선:** 가입 트리거 `handle_new_user`가 `raw_user_meta_data->>'role'`을 읽을 때 **`buyer`/`seller`가 아니면 `buyer`로 강제**한다(또는 예외). 클라이언트가 devtools로 `role:'admin'`을 보내도 DB엔 admin이 안 들어간다. UI는 2지선다만 노출. [Source: epics.md#Story-1.5(일반 가입 경로로는 admin 선택 불가)]

### 마이그레이션 참조 SQL (`supabase/migrations/0001_profiles.sql`)
> 그대로 사용하거나 프로젝트에 맞게 적응. `id`는 `auth.users(id)` FK, `on delete cascade`로 계정 삭제 시 프로필도 제거.

```sql
-- 0001_profiles.sql — FR1~4 기반: 역할·상태 프로필 + 신규 가입 트리거 + profiles RLS
-- (스키마 단일 출처: supabase/migrations/)

create table public.profiles (
  id          uuid primary key references auth.users(id) on delete cascade,
  role        text not null check (role in ('buyer','seller','admin')),
  status      text not null default 'active' check (status in ('active','suspended')),
  created_at  timestamptz not null default now()
);

-- 신규 auth.users 생성 시 profiles 행을 함께 생성. 역할은 가입 메타데이터에서 읽되
-- buyer/seller가 아니면 buyer로 강제(가입 경로 admin 차단 — 치명적 함정 #3).
-- SECURITY DEFINER: 정의자 권한으로 RLS/권한 우회해 안전하게 insert.
create or replace function public.handle_new_user()
returns trigger
language plpgsql
security definer
set search_path = public
as $$
declare
  v_role text := coalesce(new.raw_user_meta_data->>'role', 'buyer');
begin
  if v_role not in ('buyer','seller') then
    v_role := 'buyer';
  end if;
  insert into public.profiles (id, role, status)
  values (new.id, v_role, 'active');
  return new;
end;
$$;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function public.handle_new_user();

-- 관리자 판별(자기참조 RLS 무한재귀 회피 — 치명적 함정 #1). RLS 우회해 profiles를 읽음.
create or replace function public.is_admin()
returns boolean
language sql
security definer
set search_path = public
stable
as $$
  select exists (
    select 1 from public.profiles where id = auth.uid() and role = 'admin'
  );
$$;

-- RLS: 본인 행만 읽기 + 관리자 전체 읽기(두 정책은 OR로 결합)
alter table public.profiles enable row level security;

create policy "profiles_select_self" on public.profiles
  for select using (auth.uid() = id);

create policy "profiles_select_admin" on public.profiles
  for select using (public.is_admin());
```

> **INSERT 정책 불필요:** 프로필 생성은 트리거(SECURITY DEFINER)가 전담하므로 사용자용 INSERT 정책을 만들지 않는다(만들면 우회 경로가 생김). UPDATE/DELETE 정책은 이 스토리 범위 밖(회원 정지/삭제는 Epic 6, status 변경은 거기서 다룸).
