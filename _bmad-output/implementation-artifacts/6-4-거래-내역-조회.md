# Story 6.4: 거래 내역 조회 (FR24)

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 관리자,
I want `/admin` 안에서 **거래 내역(판매완료된 매물)** 을 조회 전용으로 보고 싶다,
so that 어떤 매물이 거래(판매 완료)됐는지 운영 관점에서 확인할 수 있다(FR24).

## Acceptance Criteria

1. **Given** 관리자(role=admin)가 거래 내역 화면(`/admin/transactions`)을 열면, **When** 목록을 조회하면, **Then** `status='sold'`인 매물(거래 완료분)만 보인다. `on_sale`(판매중) 매물은 노출되지 않는다. (FR24 — 거래 내역 = 판매완료 매물)
2. **Given** 거래 내역 화면은 **조회 전용**이므로, **When** 각 행을 보면, **Then** 삭제·수정 등 변이(mutation) 액션 컴포넌트가 **없다**(6-3 `/admin/listings`와 달리 행 액션 버튼이 렌더되지 않는다). 제조사·모델·연식·가격·"판매완료" 배지만 표시된다.
3. **Given** 비관리자/비로그인이 `/admin/transactions`에 접근하면, **When** 진입을 시도하면, **Then** 기존 `(admin)` 가드(proxy 1차 + layout `requireRole(admin)` 2차)로 차단된다(별도 가드 추가 없이 `(admin)` 그룹 상속).
4. **Given** 조회가 에러(네트워크/권한)면, **When** 결과가 에러면, **Then** 원본 에러는 콘솔에만 남기고 사용자에겐 한국어 일반 안내를 보여준다(6-3 page.tsx 규율과 동일). 빈 목록이면 "거래 내역이 없습니다." 안내.
5. **Given** 관리자 홈(`/admin`)에서, **When** 화면을 보면, **Then** "거래 내역"(`/admin/transactions`) 진입 링크가 노출된다(회원 관리·매물 관리 링크와 동일 형식). 자리표시 문구는 채팅(6-5)만 남기도록 갱신한다.
6. **Given** 본 스토리 구현 후, **When** 마이그레이션/RLS 상태를 확인하면, **Then** 신규 마이그레이션은 없다. 관리자 전체 조회 `listings_select_admin`(SELECT, using=`is_admin()`)이 0002에 이미 있어 sold 행을 본다. 기능은 순수 web(Next.js) 화면 추가로 달성된다.

## Dev Notes

### 핵심 요지(가장 먼저 이해할 것)
- 이 스토리는 **6-3(전체 매물 관리)의 조회 전용·sold 한정 버전**이다. 6-3은 "전체 매물(sold 포함) + 삭제"였고, 6-4는 "sold만 + 조회 전용(액션 없음)".
- 차이점 딱 2개: ① `.eq('status', 'sold')` 필터를 건다. ② 행 액션 컴포넌트(`ListingAdminActions`)를 넣지 않는다. 나머지(서버 컴포넌트 구조·에러/빈목록 분기·배지·admin 링크)는 6-3을 그대로 따른다.
- **DB 작업 0건.** 관리자 전체 조회 RLS(`listings_select_admin`, 0002)가 이미 sold 행을 열어준다. 거래 내역은 그 위에 status 필터만 추가로 거는 화면이다.

### 반드시 먼저 읽을 것 (기존 패턴 — drift 금지)
- `web/src/app/(admin)/admin/listings/page.tsx` — ⭐ **이 스토리의 직접 본보기**. 서버 컴포넌트가 `listings` 조회 + 에러/빈목록 한국어 분기 + 상태 배지. **transactions/page.tsx는 여기서 `.eq('status', SOLD)` 필터를 추가하고 `<ListingAdminActions>` 렌더 부분을 제거**한 형태다.
- `web/src/app/(admin)/admin/page.tsx` — 진입 링크 추가 위치(회원 관리·매물 관리 패턴). 거래 내역 링크 1개 추가 + 자리표시 문구 갱신.
- `web/src/app/(admin)/layout.tsx` — `requireRole(USER_ROLE.ADMIN)`. `/admin/transactions`는 `(admin)` 그룹 하위라 **자동으로 같은 가드를 상속**(별도 가드 불필요).
- `web/src/lib/supabase/server.ts` — 서버 조회 클라이언트. service_role 키 사용 금지.
- `web/src/lib/constants.ts` — `LISTING_STATUS`(ON_SALE/SOLD)·`UNITS`(price 단위). 필터·배지·가격 표시에 재사용. **신규 상수 추가 없음**.

### 설계 결정(확정)
- **거래 내역 = sold 필터 조회**: 6-3은 관리자가 전부 보려고 필터를 뺐지만, 6-4는 "거래(판매 완료)된 것"만 보는 게 목적이므로 `.eq('status', 'sold')`를 명시한다. 관리자 세션의 `listings_select_admin`(is_admin())이 sold 행을 열어주므로 RLS 추가 없이 화면 필터만으로 충족된다.
- **조회 전용(범위 컷)**: FR24의 동작은 "거래 내역을 본다"이다. 삭제·수정 같은 변이는 6-3(매물 관리)의 역할이고 거래 내역엔 없다. 따라서 액션 컴포넌트를 두지 않는다(읽기 전용 화면).
- **상태 배지**: 필터로 전부 sold이므로 "판매완료"(회색) 배지만 표시(6-3의 SellPage 회색 스타일 재사용). 별도 분기 불필요하나 6-3 구조 일관성을 위해 동일 스타일을 쓴다.
- **회귀 안전**: 신규 파일 1개(transactions/page.tsx) + 기존 admin/page.tsx에 링크 1줄 추가. 기존 (user)/(auth)/(admin/members)/(admin/listings) 동선·RLS·마이그레이션·constants 무변경.
