# Story 2.4: 구매 완료 처리

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 판매자,
I want 거래가 끝난 매물을 "구매 완료"로 표시하고 싶다,
so that 더 이상 구매자에게 노출되지 않는다.

## Acceptance Criteria

1. **(구매 완료 전환)** 본인 매물(판매중) 관리 목록/상세에서 **"구매 완료"** 버튼을 누르면 `status`가 `sold`로 바뀐다(판매자만, 상대 지정 불필요). 실수 방지를 위해 전환 전 확인 단계를 둔다. 성공 시 목록이 즉시 갱신돼 해당 매물이 "판매완료"로 표시된다. [FR8]
2. **(이미 sold면 거부)** 이미 `sold`인 매물은 "구매 완료" 버튼이 노출되지 않는다(앱측). URL/이중 클릭 등으로 다시 전환을 시도해도 거부된다(앱 한국어 안내 + DB 멱등). [FR8]
3. **(타인 매물 차단 — RLS)** 타인 매물을 `sold`로 전환하려 하면 **RLS(`listings_update_own`)로 차단**된다(0행). 앱은 0행을 권한 거부로 판정해 한국어 안내("본인 매물만 구매 완료 처리할 수 있습니다.")하고, 원본 에러·코드는 콘솔만. status·seller_id 위조는 UPDATE payload에 `{ status: 'sold' }`만 보내 차단(seller_id 미포함). [FR8·NFR2]
4. **(상태값 무결성)** 전환은 `status`를 `sold`로만 바꾼다. `on_sale`/`sold` 외 값은 DB CHECK가 거부한다(이미 2-1에 라이브). 전환 시 다른 필드(가격·옵션 등)는 변경하지 않는다(status 단일 변경). [FR8]
5. **(역할 게이트 유지)** 구매 완료 처리는 seller 역할만 접근 가능하다(`(user)/sell` layout `requireRole(SELLER)` + proxy `/sell` 보호가 그대로 커버). 비로그인→`/login`, 다른 역할→홈. [FR3 기반·2-2/2-3 확정]

> **범위 주의(반드시 준수):** 이 스토리는 **판매자 본인 매물의 구매 완료(`on_sale`→`sold`) 전환(FR8)** 까지다.
> - **구매자 화면/필터/상세/판매완료 비노출 확인은 Epic 3** 소관 — 만들지 않는다(비노출 RLS는 2-1에 이미 라이브, 여기선 status만 바꾼다).
> - **샘플 시드는 2-5.**
> - **마이그레이션·DB 스키마는 2-1에서 완료** — 건드리지 않는다. status 전환은 기존 `listings_update_own` RLS + `status` CHECK로 충분하다(**새 마이그레이션 불필요**). `embedding`은 NULL로 둔다(Epic 4 backfill).
> - **되돌리기(sold→on_sale 재오픈) 버튼은 만들지 않는다.** AC상 RLS가 재오픈 자체를 막지는 않지만(단방향 트리거 없음), 본 스토리의 명시 기능은 "구매 완료"(단방향)뿐이다 — 재오픈 UI는 범위 밖(과잉구현 금지).

### 2-1에서 확정된 DB 계약 (재발명·재설계 금지 — 새 마이그레이션 불필요)
- **status CHECK**: `status text not null default 'on_sale' check (status in ('on_sale','sold'))` → "그 외 값" 전환을 DB가 거부(AC4). [Source: 0002_listings.sql:29]
- **UPDATE 정책 `listings_update_own`**: `using (auth.uid()=seller_id) with check (auth.uid()=seller_id)` → 타인 매물 status 전환 차단(AC3), seller_id 재지정 차단. **status를 sold로 바꾸는 전환 자체는 이 정책으로 충분히 허용·보호된다.** [Source: 0002_listings.sql:111-114]
- **updated_at 트리거 + created_at 불변**: UPDATE 시 `updated_at=now()`, `created_at=old.created_at` 강제. status만 바꿔도 updated_at 갱신. [Source: 0002_listings.sql:65-84]
- **RLS 거부 = 0행**(PostgREST): 타인 매물 UPDATE는 에러가 아니라 0행 → `.select()` 행 수 0을 "차단됨"으로 판정해 한국어 안내(2-3 삭제·수정과 동일). [Source: 2-3 Dev Notes, 0002_listings.sql RLS]
- **sold→on_sale 재오픈**: 단방향 트리거가 없어 RLS상 본인 매물이면 재오픈도 기술적으로 가능하나, 본 스토리는 "구매 완료"(단방향)만 구현한다(재오픈 UI 범위 밖). [Source: 본 스토리 범위 주의]
