# Story 1.3: 로그인 / 로그아웃

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 가입한 사용자,
I want 로그인·로그아웃하고 싶다,
so that 내 세션으로 서비스를 이용하거나 안전하게 종료할 수 있다.

## Acceptance Criteria

1. **(로그인 성공 → 세션 생성 + 랜딩 이동)** 유효한 이메일·비밀번호로 로그인하면 Supabase 세션이 생성되고, 로그인 후 첫 화면(홈 `/`)으로 이동한다. 홈은 로그인 상태(이메일·역할)를 표시한다.
2. **(로그인 실패 → 한국어 오류, 세션 없음)** 잘못된 자격(없는 이메일·틀린 비밀번호)으로 로그인하면 **한국어 오류 메시지**가 표시되고 세션이 생기지 않는다.
3. **(로그아웃 → 세션 파기)** 로그인 상태에서 로그아웃하면 세션이 파기되고, 화면이 비로그인 상태(로그인/회원가입 링크 노출)로 전환된다.

> **범위 주의(반드시 준수):** AC1의 "역할에 맞는 첫 화면"은 **현재 역할별 화면이 아직 없으므로** 공통 홈 `/`으로 이동하고 홈에서 역할을 **표시**하는 선까지다. AC3의 "보호 화면 접근 차단"의 **강제 집행(직접 URL 접근 시 리다이렉트, `/admin` 게이팅)은 Story 1.4(middleware)** 소관이다. 1.3은 **세션 생성/파기 + UI 상태 반영**까지만 구현·검증한다. middleware·역할 분기 라우팅을 여기서 만들지 말 것.

### 재발명 금지 — 기존 자산 재사용
- **역할 상수:** `web/src/lib/constants.ts`의 `USER_ROLE`/`UserRole` import. 새 enum·문자열 리터럴(`'buyer'` 등) 직접 작성 금지. 역할 라벨(구매자/판매자/관리자)이 필요하면 매핑 객체를 만들되 값은 `USER_ROLE` 기준.
- **Supabase 클라이언트:** 브라우저는 `@/lib/supabase/client`의 `createClient()`(이미 존재), 서버 컴포넌트는 `@/lib/supabase/server`의 `createClient()`(async — `await createClient()`). 새 클라이언트 팩토리 만들지 말 것. [Source: 1-1/1-2 File List]
- **에러 매핑 패턴:** 가입 화면 `toKoreanError`의 구조(코드/메시지로 좁혀 한국어 반환, 원본 비노출)를 본떠 로그인용 `toKoreanLoginError`를 만든다. 로그인은 에러 종류가 달라(잘못된 자격) 별도 함수가 맞다 — 다만 폼 마크업·loading·trim·더블서브밋 가드는 그대로 복제. [Source: web/src/app/(auth)/signup/page.tsx]
- **디렉터리:** 로그인 화면은 라우트 그룹 `(auth)` 아래 → 실제 URL은 `/login`(그룹명은 URL에 안 들어감). 가입은 `/signup`로 이미 존재. [Source: architecture.md#Complete-Project-Directory-Structure(line 345 `(auth)/login · signup`)]

### 로그인 구현 패턴 (웹)
- **호출:** `'use client'` 컴포넌트에서 `createClient().auth.signInWithPassword({ email, password })`. `@supabase/ssr`의 `createBrowserClient`라 성공 시 세션 쿠키가 자동 기록된다. [Source: web/src/lib/supabase/client.ts, architecture.md#API-&-Communication-Patterns]
- **성공 후 이동:** `useRouter().push('/')` **직후 `router.refresh()`** — App Router 서버 컴포넌트(홈)가 방금 생긴 세션 쿠키를 다시 읽어 로그인 상태로 렌더하게 한다. refresh 없이 push만 하면 캐시된 비로그인 화면이 보일 수 있다.
- **실패:** `signInWithPassword`는 잘못된 자격에 **에러를 반환**한다(가입의 모호한 응답과 달리 명확). `error.code === 'invalid_credentials'` 또는 메시지 `"Invalid login credentials"` → 한국어로 "이메일 또는 비밀번호가 올바르지 않습니다."(보안상 어느 쪽이 틀렸는지 구분하지 않는다).

### 로그아웃 + 홈 랜딩 패턴
- **로그아웃:** 클라이언트 컴포넌트에서 `createClient().auth.signOut()` → `router.push('/login')` + `router.refresh()`. signOut은 로컬 세션 쿠키와 서버 세션을 모두 무효화한다.
- **홈(서버 컴포넌트) 세션 판별:** `const supabase = await createClient(); const { data: { user } } = await supabase.auth.getUser();`
  - **⚠️ 서버에서는 `getUser()`를 쓴다(`getSession()` 금지).** `getSession`은 쿠키를 그대로 신뢰해 위변조에 취약하다는 Supabase 경고가 있다 — `getUser`는 Auth 서버에 재검증한다. [Source: @supabase/ssr 권장 패턴, node_modules/next/dist/docs(authentication 가이드 — 코드 작성 전 확인)]
  - 로그인 시 `await supabase.from('profiles').select('role').eq('id', user.id).single()`로 역할 조회(본인 행은 `profiles_select_self` RLS로 허용). 역할 한국어 라벨로 표시.
- **세션 새로고침(token refresh) 한계:** 아직 `middleware.ts`가 없어(1.4 예정) 서버 측 자동 토큰 갱신이 없다. 1.3 범위(즉시 로그인/로그아웃 흐름)에선 문제없지만, 만료된 세션의 자동 갱신은 1.4 middleware에서 완성된다. 이 한계를 Completion Notes에 명시. [Source: web/src/lib/supabase/server.ts 주석("미들웨어가 세션 갱신을 담당"), epics.md#Story-1.4]
