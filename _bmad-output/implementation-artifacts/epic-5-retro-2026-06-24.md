## 잘된 점 (반복할 패턴)

1. **"무결성은 DB로 못 박는다"가 Epic 2(소유권)·Epic 5(채팅 위조)에서 일관.** 코드리뷰가 찾은 seller_id 위조(구매자가 아무 판매자·매물 지정 가능)를 앱 검증이 아니라 **BEFORE INSERT 트리거로 seller_id를 매물 실소유자로 강제 덮어쓰기**로 해결. 클라이언트 입력을 아예 신뢰하지 않아 위조 벡터가 원천 소멸. NFR2 "결정론적 안전장치" 철학의 연장.
2. **단일출처(single source of truth) 유지.** 방 생성·재사용은 `lib/chat.ts` `openOrCreateRoom` 한 곳, 메시지 조회·전송은 `lib/messages.ts` 한 곳. 폴링·전송 두 경로가 같은 dedupe·정렬 로직을 공유해 drift가 없음(5-3 코드리뷰 패치도 단일출처라 양 경로에 자동 적용).
3. **FR11(판매완료 비노출)을 채팅에서도 일관 적용.** sold 매물 채팅 헤더가 깨지는 [Decision]을, RLS를 넓히지 않고 **"판매완료/조회 불가" 빈 상태**로 처리(옵션 D). "구매자에게 sold 상세를 보이지 않는다"는 핵심 규칙이 이미 답을 정함 — 새 규칙을 만들지 않고 기존 불변식에 맞춤.
4. **경계·동시성을 코드가 먼저 막음.** UNIQUE 경합(23505)→재조회 흡수, 본인 매물(23514)·없는 매물(23503)→한국어 거부. 5-3은 `gt(>)` 커서가 동일 created_at 메시지를 영구 누락하는 버그를 라이브 검증으로 발견 → `gte(>=) + id dedupe`로 사전 차단.
5. **trigger 함수 RPC 노출까지 마감.** security definer 트리거 함수가 PostgREST `/rpc`로 노출되는 advisor 경고(0028/0029)를 EXECUTE 회수(anon·authenticated)로 해소. "동작하면 끝"이 아니라 보안 권고까지 0으로.
6. **오케스트레이터 자체 E2E가 실제로 결함을 잡음.** preview(운영 빌드)에서 로그인→문의→전송→폴링 수신까지 직접 통과. 로컬 dev HMR이 첫 클릭을 삼키는 환경 이슈를 운영 빌드 preview로 우회해 "실사용 동작"을 확인.

## 어려웠던 점 / 반복된 함정

1. **코드리뷰 [Decision] 2건이 보안·규칙 사안 — 자동 진행을 멈추고 사용자 승인을 받음.** (5-1) seller_id 위조(High)는 공유 DB 스키마 변경이라 하니스가 오케스트레이터 권한만으로의 적용을 차단 → 사용자 본인 승인(옵션 A) 후 진행. (5-2) sold 헤더는 FR11 해석이 걸려 확인 후 옵션 D. 무인 진행이라도 "공유 리소스·핵심 규칙" 변경은 사람이 판단.
2. **데이터 수정 CTE의 스냅샷 함정.** 검증용 `WITH ins AS (INSERT ...) DELETE ... WHERE id IN (SELECT FROM ins)`가 같은 문장 안에서 삽입 행을 못 봐 정리가 안 되고 테스트 행이 남음 → 별도 DELETE로 정리. 멀티-CTE 데이터 변경의 가시성 규칙을 재확인.
3. **Supabase가 anon/authenticated에 EXECUTE를 직접 부여.** `revoke ... from public`만으론 함수 권한이 안 빠져 advisor가 계속 경고 → 두 롤에서 명시 회수 필요. 기본 권한 모델을 가정하면 틀림.
4. **로컬 dev HMR ↔ Playwright 클릭 충돌(Epic 5 내내 재현).** 5-2·5-3·오케스트레이터 E2E 모두 첫 클릭이 Fast Refresh 리빌드에 삼켜짐. 코드 결함이 아니라 dev 환경 특성 — 운영 빌드(preview)엔 없음. 매번 두 번째 시도/직접 호출로 우회.
5. **마이그레이션 ledger와 파일의 미세 분기.** 트리거는 `apply_migration`(0003c)로, 권한 회수는 `execute_sql`로 적용해 원격 ledger가 파일과 어긋날 뻔함 → 회수문을 별도 ledger 항목으로도 기록해 새 환경 재현성 확보. 파일(단일출처)은 항상 완전해야 함.

## 핵심 교훈

1. **클라이언트 입력은 "참고"이지 "신뢰"가 아니다 — 권위 있는 값은 서버/DB가 도출한다.** seller_id를 클라가 보내게 두지 않고 매물에서 강제 도출한 게 위조를 원천 차단. "검증"보다 "도출/덮어쓰기"가 더 강한 방어일 때가 있다.
2. **새 규칙을 만들기 전에 기존 불변식이 이미 답했는지 보라.** sold 헤더 [Decision]은 FR11이 이미 "비노출"로 정해, RLS 확대(새 규칙) 대신 빈 상태 처리(규칙 준수)가 정답. 결정거리처럼 보여도 핵심 규칙이 결론을 강제하면 그건 결정이 아니다.
3. **공유 리소스·보안·핵심 규칙 변경은 무인 모드라도 사람 승인.** 하니스가 오케스트레이터 위임 승인으로는 라이브 DB 변경을 막은 것은 옳은 가드. 일상 판단은 자율, 보안/규칙은 escalate — 이 경계가 자동 개발의 안전선.
4. **폴링은 커서 경계가 함정.** `>` vs `>=`, 동일 타임스탬프 동시 도착, dedupe·안정 정렬 — 준실시간 UX의 버그는 "흔한 동시 전송"에서 나온다. 단일출처에 모아 한 번 고치면 모든 경로가 고쳐진다.
5. **"동작 확인"에 advisor·권고까지 포함하라.** 기능이 되는 것과 보안 권고 0은 다른 문제. 트리거 함수 RPC 노출처럼, 마감은 advisor가 깨끗할 때다.
